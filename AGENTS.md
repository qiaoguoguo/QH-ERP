# 仓库规范

## 项目角色与主代理职责（高优先级）

- 本规则优先级高于普通开发、实现、测试、推进和验收规则；如与旧交接、旧计划或旧文档冲突，以用户最新明确要求和本节规则为准。
- 主代理担任项目负责人，只负责阶段统筹、总体规划、范围控制、工作包划分、并行协调、风险判断、审查汇总、交付决策和向用户汇报。
- 产品设计、UI 设计、前端实现、后端实现、测试执行和业务功能修复必须由对应固定角色完成。主代理不得因工作量小、执行时间长、工具故障、返工困难或进度压力直接接手，也不得以临时排障或顺手修改为由参与业务实现。
- 用户明确要求处理仓库规则、只读文档、配置、Git 操作、交接记录或小型治理修正时，主代理可以直接处理；该例外不得扩展到产品、UI、前端、后端、测试或业务功能实现。
- 项目固定使用产品经理、UI 设计师、前端开发、后端开发、测试 5 个长期角色。正常情况下每个角色只有一个当前有效的长期会话；线程 ID 可以变化，角色职责不得变化，不得因任务、审查或返工增加第 6 类角色或额外审查角色。
- 五个角色会话必须跨阶段复用。任务结束以及线程显示 `completed` 或 `idle` 均不代表失效，不得因此归档、关闭或重建；只有原会话明确失效且恢复失败时，才允许创建同角色替代。
- 规格、代码质量、视觉和测试审查由固定 5 个角色在阶段集中审查中分工完成。承担对应实现的角色不得自审自己的规格或代码，应在现有五角色内交叉审核；不得仅因审核需要创建同角色替代会话。
- 新建任何固定角色会话或失效角色的替代会话时，只允许使用 `gpt-5.5` 模型并显式设置推理强度为 `xhigh`（极高）。如当前协作工具无法同时满足模型、推理强度或五角色长期复用要求，可以使用满足这些条件的 Codex 后台线程；不得降低模型或推理强度。平台无法满足时，必须暂停并向用户说明。

### Codex 后台固定角色会话运行规范

- Codex 后台线程是固定角色的长期会话载体，不是一次性任务代理。本轮任务结束后仍应通过后台消息继续复用原线程。
- 固定角色线程 ID、创建原因、复用状态、失效记录和替代原因统一登记在当前交接文档；根规则只保存稳定的角色、模型、推理强度和复用原则。
- 每次派发至少写明角色、目标、权威工作区绝对路径、目标分支或提交、输入、职责边界、预期成果、验收标准、禁止发散范围，以及是否允许编辑文件、运行测试、操作数据库或执行 Git 写操作。
- 后台线程保存的历史工作目录不得作为当前权威代码依据。每次派发都必须明确权威工作区、分支和基准提交，并要求角色开始工作前核对 `git status --short --branch` 与 `git rev-parse HEAD`。
- 同一角色的增量任务只补充本轮新增输入和差异范围，不重复发送整个阶段的历史上下文；主代理负责汇总角色结论、消除重复或冲突，并写回唯一阶段说明、审查记录或交接。
- 线程异常时，必须先读取原线程状态并尝试向原线程发送恢复任务。只有线程不可读、恢复后仍无法继续或工具明确返回永久错误时，才允许创建同角色替代；替代前应记录原线程 ID、故障现象、恢复尝试和替代原因，替代后登记新线程 ID。
- 后台固定角色的具体操作模板、交叉审查矩阵和接管清单见 `docs/ops/stage-collaboration-delivery-process.md`；该文件用于解释执行方式，若与本文件冲突，以本文件为准。

## 阶段整包协作模式（高优先级）
- 本规则优先级高于普通开发、实现、测试、推进和验收规则；新阶段默认采用“阶段整包、并行工作包、集中审查、交付验证窗口”模式。
- 阶段开始时，5 个固定角色只进行一轮并行讨论。主代理一次性汇总阶段目标、范围边界、接口约束、关键风险和验收矩阵；涉及重大方向或范围变化时再提交用户确认。
- 阶段工作包不设置固定数量上限，也不受“最多 4 个子任务”限制，但拆分不宜过细。工作包必须按完整业务能力、清晰所有权和真实依赖划分；不得把同一能力机械拆成规格、红测、实现、转绿、审查和返工等多个流程任务。如工作包数量明显增多，主代理必须说明原因、合并相近事项并防止流程膨胀。
- 测试与实现属于同一工作包。可以采用测试先行，但不要求为红色基线单独创建任务、提交或审查门禁。
- 前端、后端、UI 和测试应在接口与验收矩阵明确后并行推进；存在真实依赖时才串行等待，不得制造流程性等待。
- 开发过程中只运行受影响测试、定向接口测试、必要的编译或类型检查。工作包完成后可进行一次受影响模块集成检查，但不得反复执行项目全量验证。
- 全部工作包达到功能完整后，优先组织一轮阶段集中审查：产品经理核对业务验收矩阵，UI 设计师检查关键页面，前端和后端分别审查非本人实现的实现质量，测试检查覆盖、异常路径和回归风险。
- 主代理必须合并并去重审查意见，一次性交给对应角色整改。阻断和严重问题必须修复；一般问题默认进入后续清单，除非影响当前验收或用户明确要求，否则不得反复阻断阶段交付。
- 集中整改后只复审修复差异和受影响路径，不重新审查未变化的整个阶段。必须尽量避免反复的“审核-返工”循环；仍有阻断或严重问题时才继续处理。
- 阶段原则上只安排一轮集中审查、一轮集中整改和一次差异复审。只有差异复审仍存在阻断或严重问题时才允许继续返工，不得因一般意见开启多轮审核循环。
- 阶段功能完整、集中审查和定向复审通过后，进入唯一的交付前全量验证窗口。全量验证通过后，才允许更新交接、合入、推送和通知用户验收。
- 全量验证窗口开始后，必须先执行完计划内全部可执行项目并统一记录结果；不得每发现一个缺陷就中断全量验证，立即进入修复、审核和复测。全量结果汇总去重后，再一次性派发整改并只复验缺陷差异和受影响路径；仅当环境或阻断故障使剩余项目客观无法继续时，才允许提前结束并明确未执行项。

### 阶段生命周期与顺序门禁

每个业务阶段必须按以下顺序推进，不得把后序证据倒推为前序门禁已经满足：

1. **接管与基线核验**：读取本文件、当前交接和相关阶段说明；确认权威 `main`、`origin/main`、工作树、运行环境、迁移版本和已知问题。旧工作树、旧交接或历史截图不得替代真实核验。
2. **唯一一轮五角色目标讨论**：五个固定角色并行核对目标、范围、排除项、接口/数据边界、风险和验收矩阵；主代理合并去重。重大方向或范围变化提交用户确认。
3. **冻结唯一阶段说明**：一份权威阶段文件同时承载目标、业务规则、状态机、权限、接口、数据迁移、工作包、验收矩阵、阻断规则和变更记录。不得为了流程形式重复创建含义相同的规格、计划和任务文档。
4. **隔离分支整包开发**：从最新 `origin/main` 建立独立分支/工作树；按完整业务能力分包，前端、后端、UI 和测试在真实依赖允许时并行。实现与受影响测试属于同一工作包，开发期只运行定向验证。
5. **唯一一轮集中审查**：产品核对业务契约，UI 检查本阶段真实桌面页面，前后端交叉审查非本人实现，测试核对覆盖和回归风险。主代理只汇总去重，不参与业务实现或测试执行。
6. **一次集中整改与差异复审**：阻断和严重问题一次性派回原固定角色；只复审修复差异和受影响路径。一般问题默认登记后续清单，不为一般问题重复开启全阶段审查。
7. **唯一交付前全量验证窗口**：集中复审通过后，统一执行全量后端、全量前端、类型检查、生产构建、迁移回归、真实桌面浏览器功能与视觉检验、环境健康和空白检查。视觉检验不截图，移动端不测试、不验收。
8. **交付与交接**：全量验证通过后才更新任务/交接，提交功能分支，按项目约定合入 `main`，推送 `origin/main`，从主分支启动验收服务并核对远端哈希。远端未同步或主分支服务不可访问时不得宣称可验收或可接管。

阶段中发现需求、接口、迁移或验收口径变化时，必须先在唯一阶段说明记录原因、影响文件、数据兼容、验收变化和决策，再继续实现；不得通过聊天结论静默改变阶段范围。

## 最优方案与故障优先规则（永久约束）
- 本项目任何任务优先采用面向长期质量、可维护性与产品目标的最优方案，不以“快速完成”或“临时绕过”替代。
- 决策应兼顾架构一致性、用户体验、数据可信度、复杂度、可验证性和后续维护成本。
- 遇到故障、阻塞、工具异常、环境问题、测试/构建/部署失败或设计偏差时，必须优先定位并修复问题，不得跳过验证改走替代路线。
- 禁止在未完成充分诊断前更改为更弱实现、降低需求标准或用猜测代替真实运行结果。
- 只有在原方案客观不可行且说明清楚原因、影响与权衡后，才能调整；调整方案必须仍满足原始目标与完成条件。
- 如无法满足原目标，应暂停并明确阻塞点，提交决策请求而非自行降级交付。

## 项目方向
- 本项目用于开发一套面向制造业单公司使用的 Web ERP 系统。
- 系统定位不是零售业、门店收银或通用进销存，而是以生产管理为核心的企业内部管理系统。
- 长期方向需要覆盖常规 ERP 能力，包括基础资料、采购、销售、库存、生产、财务、报表等。
- 生产相关能力是系统重点，尤其需要重视物料管理、BOM、库存、生产工单、领料、报工、完工入库、成本归集等业务链路。

## 阶段说明与成果约束
- 阶段执行以用户明确指令或已确认的阶段说明和验收矩阵为依据。一个阶段优先只维护一份权威说明，不得为了流程完整而重复创建规格、计划和任务文档。
- 工作包必须直接服务阶段成果，不得把同一需求机械拆成红测、编码、转绿、规格审查和质量审查等多个子任务。
- 执行过程中持续对照阶段范围和完成条件；不得展示模型能力、扩展无关功能或追加未经确认的设想。
- 如发现目标冲突、范围缺失或确需扩展，应先说明原因、影响和建议调整点；重大变化等待用户确认，小范围实现细节可由主代理在既定边界内决策并记录，但实现仍必须派发给对应子代理。

## 变更控制与质量阻断规则
- 需求、范围、计划、技术方案或验收标准发生变更时，必须记录变更原因、影响范围、涉及文件、验收影响和最终决策依据。
- 已进入执行阶段的任务不得在未记录变更的情况下改变目标、扩大范围、降低标准或替换验收方式。
- 用户已授权主代理独立推进的事项，主代理可以在不扩大项目定位的前提下组织子代理讨论并形成决策；该授权不代表主代理可以直接接手业务实现。涉及产品方向重大改变、一期边界重大扩大或无法满足原目标时，必须提交用户确认。
- 阻断缺陷包括系统无法启动、核心页面无法浏览器访问、关键接口不可用、权限绕过、关键业务数据错误、数据库迁移失败、阶段验收路径不可执行。
- 存在阻断缺陷时，包含该缺陷的业务阶段不得合入或推送主分支，不得通知用户进行阶段成果验收，不得以文字说明替代修复和验证。独立的仓库规则、只读文档或紧急治理变更按“分支开发与阶段验收规则”的例外条款处理。
- 严重缺陷如影响核心流程、数据可信度或权限正确性，必须先完成影响评估和处理结论，再决定是否继续当前阶段。

## 分支开发与阶段验收规则
- 项目采用多分支开发模式，日常功能开发、问题修复和方案验证应在独立任务分支或功能分支中完成。
- 主分支用于用户进行阶段性成果验收，应保持为相对稳定、可部署、可浏览器访问的验收分支。
- 未完成、未验证或不满足任务文档验收标准的功能，不得合入或推送到主分支。
- 当一个阶段性模块完成时，例如账号管理模块，必须先完成任务文档要求的实现、验证和结果汇总，再合入或推送到主分支。
- 阶段开发期间禁止把项目全量测试、全量构建或完整浏览器验收作为每个工作包的重复门禁；只运行受影响范围的定向验证。
- 全量后端测试、全量前端测试、类型检查、生产构建、数据库迁移验证、浏览器验收和本阶段相关页面视觉检验统一放在阶段交付前的全量验证窗口执行。
- 全量验证窗口发现问题时，先用定向测试定位和修复，再验证受影响范围；交付前必须在同一窗口完成最终全量确认。不得以开发期间的局部测试代替最终全量验证。
- 推送到主分支后，必须在本地完成部署或启动服务，并向用户提供可通过浏览器访问的地址、验收范围、已完成内容和必要的验证说明。
- 阶段性验收以用户能够在浏览器中实际查看和操作交付成果为准，不得仅以代码完成、接口完成或文字说明替代。
- 浏览器验收阶段只对本阶段实际新增或修改的相关页面、组件和受影响业务状态进行视觉检验，不扩展检查未变化页面。
- 设计和开发仍必须考虑移动端、窄屏、触屏和响应式适配，避免实现结构阻碍后续不同视口使用；但后续自动化测试、人工测试、集中审查和阶段验收只覆盖桌面端，不把移动端、窄屏或响应式兼容性作为测试项、验收项或阻断条件，除非用户后续明确要求。
- 视觉检验必须基于真实运行页面完成，但默认不采集、不保存截图，不建立阶段视觉截图目录，也不把截图作为验收证据或交付成果；只需在任务或交接记录中写明检查范围、发现问题、处理结果和最终结论。
- 视觉检验应确认桌面端页面布局、信息密度、关键操作可见性、文案溢出、控件重叠和业务状态识别满足当前任务验收标准。
- 影响核心操作、关键数据识别、权限状态判断或明显降低工作台扫描效率的视觉问题，必须先修复并复验受影响的桌面端路径，不得带入主分支阶段验收。
- 如本地部署、启动服务或浏览器访问失败，必须优先定位并修复问题；在故障未解决前，不得声称该阶段成果已可验收。
- 用户明确要求合入的仓库规则、只读文档或紧急治理变更，可以独立合入主分支，无需启动业务阶段全量验证；提交必须不含业务实现，并且不得据此宣称业务阶段已可验收。

## 项目负责人统筹与子代理派发规则
- 主代理负责阶段节奏、工作包划分、跨角色依赖、风险识别、审查意见去重和最终交付判断，不得用流程数量代替质量判断。
- 角色任务必须服务当前阶段目标；未确认的想法、额外探索或范围外内容不得派发。
- 可并行的讨论、实现和集中审查必须并行派发；只有存在真实输入依赖时才串行等待。
- 同一阶段不得让多个角色重复审查同一问题。主代理应先合并意见，再一次性派发整改，避免多轮零散返工。
- 规格审核和代码质量审核必须由不同于实现者的子代理承担；不得让实现开发子代理审核自己的规格、代码或测试结论。
- 除非角色进程明确故障、长时间无响应或工具返回异常，不得中断、关闭或替换正在执行的固定角色。
- 角色故障时先记录现象、影响和当前工作区状态，优先恢复同一角色会话；只有无法恢复时才使用 `gpt-5.5`、`xhigh`（极高）创建同角色替代。
- 用户要求主代理直接处理时，仅适用于仓库规则、只读文档、配置、Git 操作、交接记录或小型治理修正；涉及业务功能、产品方案、UI 设计、前后端代码或测试执行时，仍必须派发给对应固定角色子代理。

## 智能体专用说明

硬性语言规定：所有智能体与用户的对话沟通只能使用中文。除非用户明确要求，或必须引用第三方技术接口、库名、命令、文件路径、协议名等不可翻译的技术标识，否则不得在解释、注释、文档、提交信息、议题、拉取请求、界面文案或任何面向项目的内容中使用非中文表达。
