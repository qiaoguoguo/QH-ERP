# 电气制造业全量演示数据基线与重建验收单

## 文档定位

本文是当前验收环境全量演示数据的唯一范围、生成、重建和验收入口。演示数据服务于 000 至 023 已实现能力的浏览器演示、权限核对、来源追溯和新代理接管，不代表 024 及以后路线已经实现。

演示数据不进入 Flyway，不作为生产初始化数据。Flyway 只负责结构、固定审批定义、打印模板和权限种子；业务事实由独立演示数据运行器通过真实 API/业务服务生成，SQL 只允许用于受控清理、只读校验和必要的序列对齐。

## 环境与安全边界

- 目标数据库：Docker 容器 `qherp-postgres` 中的验收库 `qherp`。
- 目标对象存储：Docker 容器 `qherp-minio` 中的私有 bucket。
- 前端、后端和演示数据运行器当前运行在 Windows 宿主机，不得表述为全部容器隔离。
- 清库只允许作用于 QH ERP 的 PostgreSQL 数据库、MinIO bucket 和明确的宿主服务进程，不得删除其他容器、卷、工作树或用户文件。
- 清库前输出 PostgreSQL 与 MinIO 备份位置；用户已明确授权丢弃旧验收样例，但备份仍用于重建失败时回滚。
- 运行器必须要求显式确认目标数据库名和容器名，默认拒绝未知目标；不得接受生产连接串。

## 演示企业与时间轴

- 企业：桥合电气成套设备有限公司，单公司制造企业。
- 主产品：低压动力配电柜、PLC 控制柜；包含柜体、母排组件、控制线束等半成品。
- 原材料：断路器、接触器、继电器、铜排、端子、控制线缆、绝缘件、铭牌和包装件。
- 数据日期：主链路围绕 2026-07-01 至 2026-07-15；同时保留 2026-06 锁定历史和 2026-08 未来有效样例。
- 编码：使用稳定、可搜索的 `DEMO-ELEC-20260715-*` 幂等键或批次标识；业务编号保持电气行业可读，如 `FG-*`、`RAW-*`、`BOM-*`、`SP*`、`CT*`，不得再使用 `021-BROWSER`、`023ui` 等临时验收前缀。

## 生成架构

采用混合方案：

1. 启动 PostgreSQL 和 MinIO，空库执行 Flyway V1 至 V25。
2. 应用启动，由 `AccountPermissionInitializer` 初始化管理员、系统管理员角色和完整权限。
3. 演示数据运行器使用真实登录和业务 API，按阶段提交主数据与业务事实。
4. 审批、消息、附件、打印、导入导出和文档任务必须经过真实平台服务；不得直接插入完成状态。
5. 所有库存数量与价值必须经过统一过账与估值服务；不得直接写库存余额、公共计价池、项目成本层或价值流水。
6. 运行器输出本次运行 ID、Git 提交、Flyway 版本、生成对象编号/ID、对象存储 key、任务 ID、关键数量和金额摘要。
7. 独立验证器读取 API 与数据库只读结果，检查对象数量、状态分布、来源链、库存一致性、权限脱敏、对象存储和审计。

不采用纯 SQL 业务灌数，因为它会绕过权限、编码、业务期间、乐观锁、审批、审计、MinIO、任务租约和库存计价约束。演示数据运行器不得修改 V1 至 V25 迁移，也不得为了造演示数据创建 Flyway 迁移。V25 是全量真实 API 重建暴露出的价值流水字段长度缺陷修复，不包含任何演示业务事实。

## 账号与权限最低覆盖

演示环境至少建立以下账号类型，密码通过运行参数或环境变量注入，不在仓库提交真实密码：

- 全权限管理员。
- 销售/项目角色。
- 采购角色。
- 仓库角色。
- 质检角色。
- 生产计划与车间角色。
- 成本/财务角色，具有库存价值查看权限。
- 独立审批人，不能审批自己提交的任务。
- 只读用户。
- 无库存价值权限用户，用于验证金额和成本层脱敏。
- 停用用户，用于登录拒绝。

角色使用现有 `sys_permission.code` 组合，不手工拼接路由或 API 权限。菜单、路由、按钮、接口、字段、附件下载、打印、导入导出和来源跳转必须一致受控。

## 业务数据最低标准

| 能力 | 最低数据与状态 |
|---|---|
| 基础资料 | 单位不少于 5 个、仓库不少于 4 个、客户/供应商各不少于 5 个、物料不少于 20 个；覆盖启用/停用、计价/非计价、批次/序列/不追踪、单位换算、结算税务脱敏和编码规则 |
| BOM 与 021 治理 | BOM 不少于 6 个；覆盖草稿、当前有效、未来生效、历史失效、停用、多层明细、替代料、ECO 草稿/审批中/已应用/已取消；工单按计划开工日期命中版本并保留快照 |
| 业务期间 | 至少一个开放期间和一个锁定期间，历史查询可见，锁定期写入被拒绝 |
| 采购 | 采购订单、部分/全部收货、采购退货、在途；至少两次不同单价公共入库形成移动平均 |
| 质量 | 待检、合格、不合格、冻结与解冻；非合格库存不得被销售或生产消费 |
| 库存与追踪 | 公共库存和项目库存；至少 2 仓、2 批次、3 个序列；期初、调整、预留、占用、释放、冻结、来源去向追溯 |
| 生产 | 工单草稿、已发布、生产中、已完成、已取消；BOM 快照、领料、退料、补料、报工、完工入库和业务成本记录 |
| 销售项目与合同 | 项目、主合同、补充合同、草稿/生效/关闭/取消；合同固定审批、附件和打印；销售订单可选关联有效项目合同 |
| 销售 | 订单、预留、部分/全部出库、销售退货和来源继承 |
| 财务往来 | 应收/收款、应付/付款、未结/部分结/已结/取消、往来调整和来源追溯；不得伪造发票、凭证、总账或月结 |
| 退货与反冲 | 销售退货、采购退货、生产退料/补料和往来冲减；原单不改写，反向事实可追溯 |
| 022 平台 | 审批实例不少于 8 条，覆盖待办、通过、驳回、撤回、取消和我发起；附件不少于 8 个；合同审批与 BOM ECO 审批两类固定打印任务均需成功；物料/BOM 导入导出任务不少于 6 个，覆盖待确认、成功、失败、取消或过期 |
| 023 价值 | 公共移动平均池不少于 3 个物料；项目成本层覆盖至少 2 个项目且至少一项目有 2 个来源层；调拨、三类所有权转换、零/正/负差异盘点、期初估值、暂估调整和只读对账 |

盘点样例至少一张超过默认单页行数，必须同时出现未盘、实盘 `0` 和非零实盘，验证稳定分页和差异估值。

## 端到端演示故事

1. 建立单位、换算、编码、仓库、供应商、客户、物料与结算税务资料。
2. 建立销售项目、主合同和补充合同，上传合同附件，提交固定审批，由独立审批人处理并生成消息和固定打印任务。
3. 建立低压配电柜 BOM：历史 V1、当前 V2、未来 V3 和草稿版本，通过 ECO 形成受控变更和替代料关系。
4. 以不同未税单价采购断路器、铜排和线缆，收货后质检，形成公共移动平均、批次和序列追踪。
5. 建立销售订单并形成库存预留；建立生产工单，按计划开工日期选择 BOM，发布后保留用料快照。
6. 执行生产领料、退料、补料、报工和完工入库，形成数量、价值和成本来源。
7. 执行销售出库和销售退货，保持预留、追踪、库存价值和应收来源一致。
8. 执行采购退货、生产反向和往来调整，证明已过账原单不被改写。
9. 执行仓库调拨，企业总数量与总价值不变；执行公共转项目、项目转公共和项目转项目，保留实际成本层来源。
10. 执行零差异、盘盈和盘亏盘点，盘差按固定审批过账；执行期初估值与暂估调整，金额变更不重写历史出库。
11. 在任务中心查看物料/BOM 导入导出、固定打印、失败明细和结果下载；用低权限账号验证附件、成本和业务对象跳转不可越权。
12. 从经营报表和来源抽屉核对采购、销售、库存、生产、成本和往来净额。

## 明确排除

演示数据不得伪造以下尚未交付能力：024 项目专采、请购、询价比价和价格协议；025 销售深化；026 MRP/缺料净算；027 项目生产深化、外协深化、工艺路线、工作中心、APS/MES；028 至 032 发票、费用、凭证、正式项目成本、月结、总账和财务关账；通用 BPM、动态表单、模板设计器、多组织、多账套、多租户、库位/WMS 和 PDA 作业。

## 重建步骤与门禁

正式运行器完成后，标准入口必须是一条带显式确认参数的命令。执行顺序固定为：

1. 核对权威主工作区、目标容器、数据库名和当前服务进程。
2. 备份 PostgreSQL 与 MinIO，并记录备份路径。
3. 停止只属于 QH ERP 的宿主后端与前端进程。
4. 清理并重建 QH ERP PostgreSQL/MinIO 数据，不影响其他 Docker 资源。
5. 从当前候选提交启动后端，完成 V1 至 V25 与权限初始化。
6. 运行演示数据生成器，按阶段输出进度和 manifest；任一阶段失败立即停止。
7. 运行独立数据验证器；库存数量价值、权限、附件或任务不一致均为失败。
8. 运行唯一交付前全量验证窗口。
9. 合入并推送后，从 `main` 重启服务，再执行一次健康、登录、关键列表、来源追溯和桌面视觉检查。

## 验收规则

必须同时满足：

- Flyway 最新成功版本为 V25，权限种子幂等，管理员和演示用户可按预期登录或被拒绝。
- 所有已实现菜单有可演示数据或明确合理空态；详情、候选、版本和 `allowedActions` 不断链。
- 无负库存、负公共池或负项目成本层；公共池、库存余额、项目层和不可变价值流水一致。
- 仓库调拨前后企业总价值不变；项目库存不被公共或其他项目无控制消费。
- 审批、消息、附件、打印、导入导出和任务中心通过真实服务产生；MinIO 对象可读且哈希匹配。
- 锁定期间、库存不足、质量冻结、追踪不匹配、BOM 无效、审批绕过、过期版本和权限拒绝均返回稳定业务错误。
- 经营报表和来源明细与业务事实一致；不出现 024+ 虚构对象。
- 桌面端真实页面完成视觉检验：布局、信息密度、关键操作、状态、金额脱敏、长文案、溢出、重叠和错误反馈均不影响判断。
- 不保存截图、不建立视觉目录、不测试或验收移动端；响应式仍是设计开发约束。

任一数据库迁移失败、核心路径不可操作、权限绕过、附件泄露、库存数量价值不一致、对象存储断链、任务假完成或主分支/远端未同步，均阻断交付与接管。

## 2026-07-15 验收口径变更记录

- 全量真实 API 重建在计价物料生产补料过账时发现 `PRODUCTION_MATERIAL_SUPPLEMENT_OUT` 长度为 34，而 V20 建立的 `inv_value_movement.movement_type` 仅为 `varchar(32)`。数据库因此拒绝价值流水并被接口包装为 500。修复采用不可变前向迁移 V25，把该列扩展为 `varchar(64)`；它是既有 023 价值流水的业务缺陷修复，不是演示数据迁移。独立验证器的 Flyway 最新版本规则随之从 V24 更新为 V25，规则总数仍为 114 条。
- 初始验证器在旧验收库建立了可执行红色基线；随后依据 021 已确认的 BOM 时效设计，新增“同一父项的已发布有效期不得重叠”和“工单计划开工日期必须落在所用 BOM 有效期内”两条一致性规则。SQL 规则由 108 条增至 110 条，连同 4 条环境规则，当前总数固定为 114 条。新增规则用于保护业务日期选版语义，不改变 BOM 可保存多个历史/未来版本的设计。
- 财务恒等式按 015 之后的现行结构修正为“总额 = 已收/已付 + 累计调整额 + 未收/未付”。该修正只纠正验证表达式遗漏 `adjusted_amount` 的错误，不代表既有 `BG-0006` 业务缺陷已经修复。
- 销售合同生效规则由错误的 `status = 'ACTIVE'` 修正为 `status = 'EFFECTIVE'`，规则代码同步改为 `SALES_CONTRACT_EFFECTIVE_MIN_1`。依据是合同枚举、V17 数据库约束、合同激活服务和 022 权威说明；项目使用 `ACTIVE`，合同使用 `EFFECTIVE`。阈值和规则总数不变，生成器不得制造不存在的 `ACTIVE` 合同来迎合旧规则。
