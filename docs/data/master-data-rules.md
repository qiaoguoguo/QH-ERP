# 主数据与业务规则基线

## 目标

统一一期核心主数据、状态、编码、数量、金额、时间和初始化数据口径，为账号权限、物料、BOM、库存、生产以及销售项目与合同提供共同语言。

## 一期优先主数据

### 账号权限

- 用户。
- 角色。
- 权限点。
- 菜单。
- 按钮或操作。

### 基础资料

- 计量单位。
- 仓库。
- 供应商。
- 客户。
- 物料分类。
- 物料。

### 生产主链路

- BOM。
- BOM 版本。
- 生产工单。
- 领料单。
- 报工记录。
- 完工入库单。
- 成本记录。

### 销售项目与合同

- 销售项目。
- 主合同。
- 补充合同。

## 编码规则

一期必须先定义以下编码，具体格式在对应模块任务文档中落定：

- 用户账号：企业内部唯一。
- 角色编码：系统内唯一，建议稳定不随名称变更。
- 权限编码：按模块和动作分段，例如账号、角色、物料、BOM、工单等。
- 物料编码：系统内唯一，不依赖物料名称。
- 仓库编码：系统内唯一。
- BOM 版本：同一产出物料下唯一。
- 销售项目编号：后端生成、系统内唯一且不可编辑。
- 内部合同编号：后端生成、系统内唯一且不可编辑；外部纸质合同号可空、可重复，并可按关键词查询。
- 业务单据编号：按单据类型和生成规则唯一。

编码一旦被业务数据引用，不得通过简单改名破坏历史追溯。

## 状态规则

核心状态必须由后端定义，前端只展示和提交允许操作。

一期优先统一：

- 用户状态：启用、停用。
- 角色状态：启用、停用。
- 物料状态：启用、停用。
- BOM 状态：草稿、启用、停用。
- 销售项目状态：`DRAFT`、`ACTIVE`、`CLOSED`、`CANCELLED`。
- 销售合同状态：`DRAFT`、`EFFECTIVE`、`CLOSED`、`TERMINATED`、`CANCELLED`。
- 工单状态：草稿、已下达、生产中、已完工、已关闭。
- 领料状态：草稿、已出库、已作废。
- 报工状态：已提交、已作废。
- 完工入库状态：草稿、已入库、已作废。
- 成本记录状态：已记录、已作废。

## 销售项目与合同规则

- 销售项目创建后项目编号和客户锁定。`ACTIVE` 项目只允许更新负责人、计划日期、目标收入、目标成本和备注；`CLOSED`、`CANCELLED` 项目不可更新。
- 项目负责人引用现有系统用户，不新建员工主数据；负责人候选只返回启用用户的最小身份信息。
- 一个项目最多存在一个非取消主合同。主合同金额必须大于 0，补充合同调整金额允许正负但不得为 0。
- 补充合同必须引用同项目主合同，只允许在项目为 `ACTIVE` 且主合同为 `EFFECTIVE` 时创建，激活时再次校验相同前置条件。
- 项目、合同和关联销售订单客户必须一致。销售订单的 `projectId` 与 `contractId` 必须同时为空或同时有值，仅草稿订单可关联、切换或解除，确认时再次校验状态、归属和客户。
- 历史销售订单允许项目与合同均为空，不强制回填。020 只有销售订单接入项目合同关联；采购订单、生产工单、库存、成本、发票和收入确认仍无项目关联。
- 项目关闭、项目取消、合同关闭、合同终止和合同取消必须填写 1 至 200 字原因。项目与合同结束条件及允许动作由后端状态机校验。

## 数量、金额和时间

- 数量和金额禁止使用浮点数承载业务计算。
- 数量精度、金额精度和舍入规则必须在模块任务文档中明确。
- 计量单位换算必须有来源，不允许前端临时换算作为业务依据。
- 接口时间格式必须统一，并明确时区口径。
- 创建时间、更新时间和关键操作时间以后端生成或校验为准。

## 数据追溯

库存、生产、成本和财务相关数据必须保留来源：

- 来源单据类型。
- 来源单据编号或标识。
- 操作人。
- 操作时间。
- 业务状态。
- 必要的变更字段、原因或来源摘要。

销售项目、合同的所有写入和状态动作必须写入现有系统审计。销售订单关联、解除或切换项目合同时也必须写入销售项目审计，保留订单号及旧、新关联摘要；审计不扩展现有表，也不要求结构化变更前后值。审计查询必须遵守合同和销售订单查看权限，不能泄露受限摘要。

## 数据库迁移与初始化

- 数据库结构必须由 Flyway 迁移脚本管理。
- 初始化数据也应纳入可重复执行的机制。
- 超级管理员、基础角色、基础权限和基础菜单必须幂等初始化。
- 本地空 PostgreSQL 数据库必须能通过迁移和初始化得到可登录、可验收的基础环境。
- 禁止把手工改库作为阶段成果交付依据。
