### 项目定位

QH ERP 是面向制造业单公司内部使用的 Web ERP 系统，不是零售、门店收银、电商交易、会员营销、通用轻量进销存或多租户平台。

长期方向需要覆盖常规 ERP 能力，包括基础资料、采购、销售、库存、生产、财务、报表等。当前产品重心是制造业生产管理，尤其重视物料、BOM、库存、生产工单、领料、报工、完工入库、成本归集、采购入库、销售出库、财务往来、经营报表和反向业务追溯。

### 当前明确不包含

截至当前节点，系统不应被误解为已经具备以下能力：

- 正式财务关账、会计期间、总账、凭证、税务、银行对账、期间快照、库存月结、成本结转和损益结转。系统当前具备的是业务期间锁定与历史写入保护，不是正式财务结账。
- 复杂 QMS、SPC、抽样方案、不合格处置流程、供应商质量绩效和完整不合格品处理闭环。当前仅具备质量确认与库存质量状态基础。
- MRP、生产计划排程和产能计划。
- 审批流、多组织、多公司、多账套、多币种。
- 可配置 BI 平台。
- 正式发票、红字发票、银行支付和税务申报。

上述能力可以作为未来候选方向讨论，但不得在未形成阶段任务文档、接口契约、测试计划和验收标准前直接实现。

## 固定协作模式

### 主代理职责

主代理是项目负责人，只负责项目统筹、任务拆解、上下文提供、任务派发、等待结果、组织审查、验收判断和向用户汇报。主代理默认不直接承担产品设计、UI 设计、前端开发、后端开发或测试执行工作。

涉及具体产品设计、UI 设计、前端实现、后端实现、测试执行或审查时，应优先派发给固定角色子代理。主代理不得绕过固定角色直接推进大范围实现。

### 固定五角色

项目固定具备五类角色：

- 产品经理。
- UI 设计师。
- 前端开发。
- 后端开发。
- 测试。

旧交接文档中曾记录过具体线程 ID，但线程 ID 可能随工具环境、工作树或会话变化而失效。后续接手时应以角色职责为准，不以旧线程 ID 为准。如果某个角色线程失效，只能创建同角色替代子代理，并记录替代原因，不得增加第六类角色或创建无角色临时子代理。

### `/goal` 派发规则

主代理向子代理派发任务时必须用 `/goal` 组织任务，并写清：

- 使用哪个固定角色。
- 该角色承担讨论、实现、审查、复审还是测试执行职责。
- 任务目标。
- 输入上下文。
- 职责边界。
- 预期成果。
- 验收标准。
- 禁止发散范围。

规格审查、代码质量审查和修复复审都必须由五个既定角色中的合适角色承担，不得额外创建“规格审查子代理”“代码质量审查子代理”等新角色。

### 新阶段推进流程

任何新阶段开始前必须先执行：

1. 组织产品经理、UI 设计师、前端开发、后端开发、测试五个角色分别讨论。
2. 主代理汇总五角色结论。
3. 形成阶段目标、范围边界、关键风险、验收标准和实施计划。
4. 计划未形成前不得进入实现。
5. 计划确认后拆分任务，逐项派发给对应实现角色。
6. 每个实现任务完成后先做规格审查。
7. 规格审查通过后再做代码质量审查。
8. 审查失败必须派发修复任务并复审。
9. 主代理根据真实验证证据判断是否可以进入阶段验收。

## 分支、合入和验收规则

- 日常功能开发、问题修复和方案验证应在独立任务分支或功能分支中完成。
- 主分支用于用户阶段验收，应保持稳定、可部署、可浏览器访问。

生产主链路：

```text
基础资料 -> 物料 -> BOM -> 库存 -> 生产工单 -> 领料 -> 报工 -> 完工入库 -> 成本归集
```

```text
供应商 -> 采购订单 -> 采购入库 -> 库存增加 -> 应付台账 -> 付款记录 -> 来源追溯
```

销售链路：

```text
客户 -> 销售订单 -> 销售出库 -> 库存扣减 -> 应收台账 -> 收款记录 -> 来源追溯
```

经营报表链路：

```text
采购、销售、库存、生产、成本、财务往来 -> 固定经营报表 -> 来源明细 -> 来源业务详情
```

退货退款与业务反冲链路：

```text
销售出库 -> 销售退货 -> 退货入库流水 -> 应收冲减 -> 销售净额报表 -> 来源追溯
采购入库 -> 采购退货 -> 退货出库流水 -> 应付冲减 -> 采购净额报表 -> 来源追溯
生产领料 -> 生产退料/补料 -> 库存反向或补充流水 -> 成本净额 -> 生产与成本报表 -> 来源追溯
```

质量库存状态链路：

```text
采购入库、生产完工、销售退货、生产退料 -> 默认待检库存 -> 质量确认 -> 合格、不合格、冻结库存
合格库存 -> 销售出库、生产领料、生产补料可用
待检、不合格、冻结库存 -> 不参与销售出库、生产领料、生产补料可用量
冻结库存 -> 先解冻后才能参与后续允许动作
```

## 权限和安全边界

当前系统已实现账号、角色、菜单、按钮和接口权限基础。前端菜单和按钮控制只作为体验控制，后端接口鉴权是最终安全边界。

后续开发必须保持以下原则：

- 无权限菜单应隐藏。
- 无权限路由应被拦截或展示明确无权限状态。
- 写操作按钮应按权限隐藏或禁用。
- 后端写接口必须独立鉴权，不能依赖前端隐藏按钮。
- 只读账号应可查看允许页面，但不应显示或执行写操作。
- 超级管理员账号如出现页面无权限，应优先检查菜单权限初始化、当前登录态补水和后端权限点，不要简单改前端绕过。

## 数据和业务口径提醒

系统当前是业务闭环基础，不是完整财务核算系统。必须避免把业务台账、成本归集和经营报表误表达为正式财务结果。

需要持续保持的口径：

- 库存变动必须可追溯来源单据。
- 采购入库过账后影响库存，并可生成应付台账。
- 销售出库过账后影响库存，并可生成应收台账。
- 收付款更新往来余额和状态，但不代表正式银行对账或凭证。
- 成本归集是业务记录与追溯，不代表正式成本结转。
- 经营报表是固定经营汇总和来源追溯，不是可配置 BI、总账或正式财务报表。
- 退货退款与反冲通过反向单据表达净额影响，不应直接改写已过账历史原单。

## 下一阶段开发目标方向

下一阶段不应由本报告直接指定具体任务。具体实施方向必须由新主代理组织产品经理、UI 设计师、前端开发、后端开发和测试五个固定角色讨论后形成。

从当前项目状态看，下一阶段的大方向应从“继续扩展业务模块”重新转向“围绕制造业 ERP 核心闭环补齐更深层业务能力和数据可信度”。候选方向可以包括但不限于：

- 库存可用量、占用、冻结与预留：在质量状态基础上进一步明确账面库存、可用库存、占用库存、冻结库存和在途库存口径。
- 批次、序列号与来源去向追溯：在质量状态和库存流水基础上讨论采购、生产、销售、退货和召回追踪。
- 生产计划和物料需求：在现有工单、BOM、合格可用库存和采购销售基础上讨论计划、需求和缺口识别能力。
- 财务往来深化：在不直接跳到正式总账的前提下，完善业务往来、核销、余额和来源一致性。
- 报表和经营分析深化：基于固定报表继续提升经营可追溯性，但不要误扩展为完整 BI。
- 验收数据和操作体验整理：建立更稳定的演示数据、验收路径和手册维护机制。

方向选择原则：

- 优先服务制造业单公司内部 ERP，而不是泛化为通用平台。
- 优先补齐会影响数据可信度、历史追溯、库存准确性、生产闭环和经营判断的能力。
- 不要因为当前页面治理完成就忽视后续页面规范，新模块必须继承本阶段页面基线。
- 不要在未讨论清楚边界时直接进入正式财务、审批流、多组织、多账套或 BI 等大范围能力。
- 每个候选方向都必须先形成任务文档、接口契约、测试计划、视觉验收计划和子代理实施计划。

建议新主代理向五角色发起的问题不是“马上做哪个页面”，而是：

- 当前系统最影响制造业业务闭环可信度的缺口是什么？
- 哪个候选方向能在保持范围可控的前提下提高系统验收价值？
- 该方向会触碰哪些已有业务口径、接口、数据模型和页面规范？
- 需要哪些验收数据、异常场景、权限差异和视觉证据？
- 是否存在必须先治理的历史风险或环境问题？
