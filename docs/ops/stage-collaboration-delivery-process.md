# 固定五角色阶段协作与交付流程

## 文档定位

本文把根目录 `AGENTS.md` 的强制规则展开为可执行流程，供新主代理、固定角色和交接审查使用。它适用于后续所有业务阶段，也适用于需要完整产品、实现、审查和验收的跨阶段治理任务。

权威优先级依次为：用户最新明确指令、根目录 `AGENTS.md`、当前唯一阶段说明、当前交接、真实仓库与运行结果、本文。本文不得改变更高优先级规则。

## 协作载体

项目长期只维护以下五类角色：产品经理、UI 设计师、前端开发、后端开发、测试。主代理是项目负责人，不是第六个实现角色。

当普通子代理工具不能同时满足五个长期角色、`gpt-5.5` 和 `xhigh` 时，使用 Codex 后台线程承载固定角色。后台线程具有以下含义：

- 一个线程长期对应一个角色，不对应一个工作包。
- 线程本轮完成后继续保留，后续通过后台消息增量派发。
- 线程 ID、当前状态和替代记录只在当前交接登记。
- 新建或替代线程必须显式使用 `gpt-5.5` 与 `xhigh`；平台不支持时暂停，不降级。
- 后台线程自己的工作目录可能停留在历史分支。每次派发都必须写出本轮权威工作区、分支和基准提交，并要求角色先核对 Git 状态。

## 任务派发信息

每次派发至少包含：

```text
角色：固定角色名称。
目标：本轮需要达到的具体结果。
权威输入：工作区绝对路径、分支/提交、阶段说明和相关文件。
职责边界：允许读取、编辑、测试、浏览器、数据库和 Git 的范围。
成果：需要返回或落盘的文件、结论、命令证据和问题清单。
验收标准：什么结果才算完成。
禁止发散：明确不属于本阶段的功能、文件和验证。
```

实现、审查和整改任务还必须写明文件所有权、接口依赖、允许运行的定向测试以及不得修改的相邻模块。增量任务只发送新增事实和差异范围，不重复灌入整阶段历史。

## 角色职责与交叉审查

| 角色 | 计划与开发职责 | 集中审查职责 |
|---|---|---|
| 产品经理 | 冻结业务目标、范围、状态、规则和验收矩阵 | 核对实现是否满足业务契约，不审实现代码质量 |
| UI 设计师 | 冻结页面信息结构、状态表达、桌面交互和响应式开发约束 | 检查本阶段真实桌面页面，不截图、不验收移动端 |
| 前端开发 | 实现前端、接口消费、权限态和前端定向测试 | 审查非本人后端接口契约、错误语义和前端可消费性 |
| 后端开发 | 实现服务、数据、迁移、权限、审计和后端定向测试 | 审查非本人前端请求是否绕过业务规则或泄露数据 |
| 测试 | 建立独立集成、迁移、权限、异常和回归保护 | 核对覆盖、证据、环境和阻断条件；执行交付验证 |

实现者不得批准自己的规格、代码或测试结论。主代理负责去重和裁决，不直接补产品、UI、前后端或测试实现。

## 阶段执行流程

### 1. 接管与基线

- 阅读 `AGENTS.md`、当前交接、文档索引和相关阶段说明。
- 在权威主工作区核对 `main`、`origin/main`、工作树和最新提交。
- 检查 PostgreSQL、MinIO、后端健康、前端 HTTP 和当前迁移版本。
- 明确哪些能力已交付、哪些只是未来路线；发现文档冲突先修正文档或记录阻断。

### 2. 目标讨论与权威说明

- 五角色只进行一轮并行目标讨论。
- 主代理汇总目标、范围、排除项、接口/数据边界、工作包、风险和验收矩阵。
- 一个阶段优先只有一份权威说明；该文件必须能让固定角色不依赖聊天历史工作。
- 重大方向或范围扩大等待用户确认；既定范围内的小型实现细节由主代理记录决策后推进。

### 3. 隔离开发与定向验证

- 从最新 `origin/main` 建立 `codex/` 前缀独立分支或使用现有隔离工作树。
- 工作包按完整能力和文件所有权划分，不机械拆成规格、红测、实现、转绿和提交。
- 前后端、UI 和测试在接口契约冻结后并行；只有真实依赖才等待。
- 开发期运行受影响测试、定向接口测试、必要类型检查和迁移验证；不重复运行项目全量验证。
- 自动化测试与共享验收资源必须失败关闭：Maven 测试未显式配置对象存储时只能指向不可用测试端点和测试 bucket，不能继承 `QHERP_*` 验收环境变量后落到 `qherp-private`。
- 需要真实对象存储的集成测试必须使用独立 MinIO Testcontainer 和独立测试 bucket，并通过 `@DynamicPropertySource` 或等价高优先级配置覆盖失败关闭默认值；不得用 mock 掩盖上传、下载和 bucket 初始化路径。

### 4. 集中审查、整改和差异复审

- 功能完整后，五角色并行执行唯一一轮集中审查。
- 主代理把重复问题合并，按阻断、严重、一般分级，一次性派发给原角色整改。
- 阻断与严重必须闭合；一般问题不影响当前验收时进入后续清单。
- 整改后只复审差异和受影响路径。只有仍有阻断或严重问题才继续返工。

### 5. 唯一交付前验证窗口

验证窗口至少覆盖：

- 后端全量测试和构建结果。
- 前端全量测试、类型检查和生产构建。
- 空库迁移、代表性存量升级和最新迁移版本。
- 权限、异常、并发、审计和历史兼容。
- 本阶段新增或修改页面的真实桌面浏览器功能与视觉检验。
- PostgreSQL、MinIO、后端、前端和残留进程状态。
- 全量测试前后共享验收 bucket 对象数与数据库 `AVAILABLE` 文件对象数保持一致；测试写入不得出现在共享验收 bucket。
- `git diff --check`、工作树范围和未提交文件。

设计开发仍考虑响应式；验证和验收不测试移动端、窄屏或响应式兼容。视觉检查不采集、不保存截图，只在阶段说明或交接记录检查范围、发现、处理和结论。

全量验证窗口开始后，先完成计划内全部可执行项目并统一记录结果，不得发现单个缺陷后立即中断窗口进入修复、审核和复测。窗口结束后由主代理汇总、去重和分级全部缺陷，再一次性派发整改；整改后只复验缺陷差异和受影响路径。仅当环境或阻断故障使剩余项目客观无法执行时，才允许提前结束，并必须列出已执行、未执行和阻断原因。

### 6. 合入、推送和主分支验收

- 更新唯一阶段说明、当前交接和确有事实变化的关联文档。
- 提交功能分支；确认全量验证证据对应最终提交。
- 按项目约定合入 `main`，推送 `origin/main`，确认本地和远端哈希一致。
- 从主分支工作区启动或重启前后端，核对浏览器可访问地址。
- 只有主分支、远端、环境和交接均闭合后，才通知用户验收或允许新代理接手。

## 当前环境边界

当前 `compose.yaml` 只容器化 PostgreSQL 与 MinIO。前端 Vite、后端 Java、Node 测试和 Maven/JDK 验证运行在 Windows 宿主机；后端集成测试按需使用 Testcontainers。除非后续新增完整应用镜像与统一执行器，不得把开发、测试和验收描述为全部在隔离 Docker 容器中执行。

Maven 测试的对象存储默认值由 Surefire 失败关闭配置控制；只有显式声明独立 MinIO Testcontainer 的测试才能覆盖该默认值。共享验收环境的独立验证器要求 MinIO 对象总数等于数据库 `platform_file_object` 中 `AVAILABLE` 数量且不少于 8。任一测试污染共享 bucket、出现孤儿对象或数量不一致，都属于交付前阻断，必须先修复测试隔离，再受控重建验收环境，不能只手工删除污染对象。

## 接管检查清单

- [ ] 当前交接登记的五个后台角色线程均可读取并复用。
- [ ] 权威主工作区、`main`、`origin/main` 和当前提交一致且工作树干净。
- [ ] 当前已完成阶段、下一阶段和未来路线在 README、产品决策、交接和文档索引中一致。
- [ ] 当前阶段唯一说明包含范围、排除项、接口、迁移、权限、工作包、验收和变更记录。
- [ ] PostgreSQL、MinIO、后端和前端运行边界与实际一致。
- [ ] 最近交付的全量验证、桌面视觉检查和非阻断后续项可追溯。
- [ ] 演示账号、演示数据重建方式、服务地址和已知风险已经记录。
- [ ] 未提交、未合入、未推送或仅存在于旧工作树的成果没有被写成已交付事实。
