# 产品决策记录

## 2026-07-10：业务期间与历史数据保护基础

- 决策：启动 016 业务期间与历史数据保护基础阶段，以业务日期为边界维护开放和锁定期间。
- 原因：015 已要求通过反向业务而非直接改写已过账原单表达修正，但历史业务日期的新写入仍可能污染库存、采购、销售、生产、成本、往来和经营报表口径。
- 范围：提供期间维护、锁定、解锁、独立权限、审计和服务层统一写入保护；锁定期间保持查询、追溯和报表查看。
- 跨期规则：开放期间可引用锁定期间历史原单创建反向业务，但反向影响按自身业务日期进入开放期间，不回写历史数据。
- 排除：本阶段不交付正式财务关账、会计期间、总账、凭证、税务、期间快照或成本结转。

## 2026-07-10：质量检验与库存质量状态基础

- 决策：启动 017 质量检验与库存质量状态基础阶段，将库存余额从仓库+物料升级为仓库+物料+质量状态，并把合格库存作为唯一可用库存口径。
- 原因：现有 `availableQuantity` 等于现存量，无法区分待检、不合格和冻结库存，可能导致销售出库、生产领料、生产补料和后续计划净算误用不可用库存。
- 范围：建立待检、合格、不合格和冻结四类库存质量状态；历史库存默认迁移为合格；采购入库、完工入库、销售退货和生产退料默认形成待检库存；轻量质量确认支持待检转合格、不合格或冻结；合格库存可冻结并解冻回合格。
- 可用量规则：`availableQuantity` 只代表合格可用库存。销售出库、生产领料和生产补料只能消耗合格库存；总库存充足但合格库存不足时必须明确提示质量状态不可用或指定质量状态库存不足。
- 退货规则：采购退货必须显式选择质量状态，允许待检、不合格或合格库存退回供应商；冻结库存需先解冻。
- 期间保护：质量确认、冻结、解冻、入库质量状态落账和按质量状态扣减库存均受 016 业务期间保护、权限、原因和审计约束。
- 排除：本阶段不交付完整 QMS、检验标准、抽样方案、SPC、批次、序列号、库位、MRP、审批流、正式财务、复杂返工返修或质量成本归集。

## 已确认决策

### 项目定位

本项目是制造业单公司内部使用的 Web ERP 系统，以生产管理为核心，不做零售、门店收银、电商或多租户平台方向。

### 首个业务模块

首个业务模块为账号与权限基础。

原因：

- 所有后续模块都依赖登录态、用户身份、角色权限、菜单入口和接口鉴权。
- 先完成账号权限可以让后续模块从第一天起按真实权限场景验收。
- 避免物料、BOM、库存和生产模块完成后再反向补权限，造成返工。

### 第二个业务模块

第二个业务模块确认为基础资料与物料管理，采用轻量基础资料 + 物料管理范围。

纳入计量单位、仓库、供应商、客户、物料分类和物料档案。该阶段只做主数据维护，不进入 BOM、库存流水、采购销售单据或财务核算。

原因：

- 物料、BOM、库存和生产工单都依赖稳定主数据。
- 仓库、供应商、客户是常规 ERP 后续采购、销售、库存模块的前置资料。
- 提前完成轻量主数据可以避免后续模块反复补字段和补权限。

### 第三个业务模块

第三个业务模块确认为 BOM 管理，采用标准 BOM 主数据 + 版本管理 + 明细维护范围。

纳入 BOM 列表、详情、新增、编辑草稿、复制版本、启用、停用、明细用料维护、启用版本唯一、物料引用校验、可检测循环引用校验、权限控制、审计记录和浏览器视觉分析。该阶段不进入库存、生产工单、领料、报工、完工入库、成本、采购销售、工艺路线、替代料、选配 BOM、工程变更审批或复杂多级展开 UI。

原因：

- BOM 是物料档案之后、生产工单之前的关键主数据。
- 后续领料、工单用料和成本归集都需要稳定的父子物料结构和版本口径。
- 先完成 BOM 版本和启用规则，可以避免生产模块直接依赖临时用料结构。
- 保存和启用时拒绝可检测循环引用，能够降低后续工单引用错误 BOM 的风险。

### 第四个业务模块

第四个业务模块确认为仓库与库存基础能力，采用库存余额 + 库存变动流水 + 期初/调整单据范围。

纳入库存余额查询、库存变动流水查询、期初库存草稿与过账、库存调整草稿与过账、库存数量不可为负、同一仓库物料期初只允许过账一次、库存变动追溯、权限控制、审计记录和浏览器视觉分析。该阶段不进入生产工单、领料、报工、完工入库、成本核算、采购销售单据、批次、序列号、库位、条码、调拨、盘点任务、库存占用锁定、缺料分析、先进先出、多单位换算、多组织或多租户。

原因：

- 库存是 BOM 之后、生产工单和领料之前的关键基础能力。
- 后续领料、完工入库和成本归集都依赖可信库存余额和可追溯变动记录。
- 先建立期初和调整两类受控库存来源，可以避免后续生产模块直接修改库存余额。
- 一期不允许负库存、已过账单据不直接回改，有利于保证数据可信度和审计追溯。

已确认口径：

- 库存基础阶段只按“仓库 + 物料”维度管理余额，不引入批次、库位、序列号或条码。
- 一期可用数量等于现存数量，锁定数量固定为 0 或仅作为响应预留，不启用库存占用。
- 期初和调整均先保存草稿，再通过过账写入余额和流水。
- 已过账单据不可编辑；如需修正，通过新的库存调整单处理。
- 数量入账使用物料基本单位，不做单位换算。

### 第五个业务模块

第五个业务模块确认为生产执行基础能力，采用生产工单 + 领料 + 报工 + 完工入库范围。

纳入生产工单列表、详情、新增、编辑草稿、发布、完成、取消；工单发布时生成 BOM 用料快照；基于工单创建、编辑和过账生产领料；基于工单创建、编辑和过账生产报工；基于工单创建、编辑和过账完工入库；领料扣减库存、完工入库增加库存；生产来源库存流水追溯；权限控制、审计记录和浏览器视觉分析。该阶段不进入高级排产、工艺路线、工序流转、批次、序列号、库位、条码、倒冲、退料、补料、返工、委外、采购销售联动、审批流、正式成本核算、多单位换算、多组织或多租户。

原因：

- 生产执行是库存之后、成本归集之前的一期核心链路。
- 工单需要稳定引用已启用 BOM，并把用料结构冻结为快照，避免后续 BOM 修改影响历史生产任务。
- 领料和完工入库必须复用库存过账与流水追溯能力，不能直接旁路修改库存余额。
- 报工为完工入库和后续成本归集提供生产结果口径。
- 一期先打通可信生产闭环，比直接进入排产、工艺路线或复杂成本更符合当前基础阶段。

已确认口径：

- 工单编号由后端自动生成。
- 工单发布时生成 BOM 用料快照，已发布工单不再受 BOM 后续修改影响。
- 一期不启用库存占用锁定，领料过账时实时扣减库存。
- 一期不允许超领、超报、超入。
- 报工采用草稿和过账模式，报工过账只更新生产结果，不直接修改库存。
- 完工入库累计数量不得超过累计合格报工数量。
- 领料、报工和完工入库先作为工单上下文能力，不建立独立生产单据中心。
- 窄屏侧栏占用空间不单独作为阻断；若导致核心操作不可见、表单不可提交或权限状态无法识别，则作为阻断视觉缺陷。

### 第六个业务模块

第六个业务模块确认为成本归集基础记录，采用成本业务记录 + 来源追溯 + 手工补充记录范围。

纳入成本记录列表、详情、手工记录、生产领料自动材料来源记录、生产报工自动人工数量口径记录、完工入库产出数量追溯、生产工单详情成本归集分区、权限控制、审计记录和浏览器视觉分析。该阶段不进入正式财务核算、总账、凭证、会计期间、结账、标准成本、移动加权、先进先出、成本差异分析、在产品成本、完工成本分摊、制造费用自动分配、采购销售联动、多币种、多组织、多公司、多账套、多租户、审批流或 BI 报表。

原因：

- 成本归集是生产执行之后、财务和报表能力之前的一期桥梁。
- 领料、报工、完工入库已经形成可信生产来源，下一步需要沉淀为可追溯的成本业务记录。
- 当前库存和生产执行尚无稳定单价来源，不应直接承诺正式材料成本金额。
- 先建立业务记录、来源单据、数量和金额口径，可以为后续正式核算规则提供数据基础。
- 把成本记录独立建模，避免把后续财务能力混入生产执行服务。

已确认口径：

- 一期成本归集只做业务记录和追溯，不承诺正式财务核算结果。
- 自动材料来源记录来自已过账生产领料明细，只记录数量、单位、物料、工单、产品和来源单据，不自动计算材料金额。
- 自动人工来源记录来自已过账生产报工，只记录报工数量口径，不自动计算人工金额。
- 完工入库作为产出数量和归集对象追溯，不作为成本类型计入金额。
- 手工记录可用于人工金额、制造费用、其他成本或必要材料金额补录。
- 手工记录必须关联有效生产工单，产品从工单带出；本阶段不支持仅按产品录入成本记录。
- 手工记录必须填写成本类型、业务日期、金额或数量口径和说明。
- 成本记录不得反写库存余额、生产工单状态或生产单据状态。
- 页面文案必须明确“业务记录与追溯”，避免被理解为正式财务成本核算结果。

### 一期验收方式

一期采用模块逐步验收，并在生产主链路完成后追加端到端闭环验收。

模块逐步验收用于控制交付节奏；端到端闭环验收用于验证物料、BOM、库存、工单、领料、报工、完工入库和成本记录之间的数据贯通。

### 一期生产主链路端到端闭环验收与稳定化

一期六个业务里程碑完成后，追加一期生产主链路端到端闭环验收与稳定化阶段。

该阶段确认从账号权限、基础资料与物料、BOM、库存、生产执行到成本归集的完整链路真实可用，并集中验证权限一致、数据可追溯、页面可操作、异常受控和阻断缺陷已修复。

该阶段不是第七个业务模块，不新增采购、销售、正式财务、报表、审批、多组织、多租户、工艺路线、批次、库位、条码或正式成本核算能力。发现相关诉求时只登记为后续候选，不纳入本阶段验收范围。

### 一期后第一个业务模块

一期后第一个业务模块确认为采购管理基础模块，采用采购订单 + 采购入库 + 库存增加 + 来源追溯范围。

纳入采购订单列表、详情、新建草稿、编辑草稿、确认、取消、关闭、明细维护；基于已确认或部分入库采购订单创建采购入库草稿、编辑草稿、过账；采购入库过账后增加库存余额、生成 `PURCHASE_RECEIPT` 库存流水；采购订单、采购入库和库存流水三方追溯；权限控制、审计记录和浏览器视觉分析。该阶段不进入应付、发票、付款、税务、审批、询价、比价、供应商评分、MRP、采购退货、质检入库、暂估入库、多币种、多组织、正式财务凭证或报表分析。

原因：

- 一期已经完成从库存到生产执行再到成本归集的内部闭环，但原材料库存来源仍主要依赖期初和库存调整。
- 采购管理补齐制造业原材料进入企业的真实业务入口。
- 采购入库可以复用库存余额和库存流水追溯机制，技术边界清晰。
- 采购基础稳定后，后续应付、发票、付款、销售和正式财务能力才有可靠业务来源。

已确认口径：

- 采购作为独立业务域，不并入库存单据或生产域。
- 采购订单状态包含草稿、已确认、部分入库、已入库、已关闭和已取消。
- 采购入库状态包含草稿和已过账。
- 采购入库过账必须生成 `PURCHASE_RECEIPT` 库存流水，并通过来源字段追溯到采购入库明细。
- 采购单价只作为采购业务记录字段，不生成应付、不计算税额、不形成正式成本核算结果。
- 本阶段统一使用物料基本单位，不做采购包装单位或单位换算。
- 已过账采购入库不可编辑，不支持反过账；必要修正通过库存调整并保留说明。

### 一期后第二个业务模块

一期后第二个业务模块确认为销售管理基础模块，采用销售订单 + 销售出库 + 库存扣减 + 来源追溯范围。

纳入销售订单列表、详情、新建草稿、编辑草稿、确认、取消、关闭、明细维护；基于已确认或部分出库销售订单创建销售出库草稿、编辑草稿、过账；销售出库过账后扣减库存余额、生成销售来源库存流水；销售订单、销售出库和库存流水三方追溯；权限控制、审计记录和浏览器视觉分析。该阶段不进入应收、发票、收款、税务、正式收入确认、销售报价、合同、信用额度、价格体系、折扣政策、审批流、销售退货、换货、发运物流、对账单、MRP、销售预测、产销排程、报表看板、多币种、多组织、多公司、多账套、正式财务凭证或多租户。

原因：

- 采购管理基础已经补齐原材料进入企业的采购入库链路，下一步需要补齐成品或半成品离开企业的销售出库链路。
- 销售出库可以复用库存余额和库存流水追溯机制，业务边界清晰。
- 销售订单与销售出库完成后，系统具备采购入库、生产执行、库存扣减和销售来源追溯的更完整制造业业务闭环。
- 销售基础稳定后，后续应收、发票、收款、经营看板和正式财务能力才有可靠业务来源。

已确认口径：

- 销售作为独立业务域，不并入库存单据或生产域。
- 销售订单状态包含草稿、已确认、部分出库、全部出库、已关闭和已取消。
- 销售出库状态包含草稿和已过账。
- 销售出库过账必须生成销售来源库存流水，并通过来源字段追溯到销售出库明细。
- 销售出库相关契约命名统一使用 `SALES_SHIPMENT`、`sales:shipment:*` 和 `/sales/shipments`，不使用 outbound 命名。
- 本阶段销售订单明细只允许选择启用的 `FINISHED_GOOD` 与 `SEMI_FINISHED` 物料。
- 启用的 `RAW_MATERIAL` 和 `AUXILIARY` 在本阶段视为不可销售物料，创建、更新或确认销售订单时必须拒绝。
- 本阶段不新增物料“可销售”字段，不扩大物料主数据范围。
- 销售订单创建、更新、确认时客户必须存在且启用。
- 销售订单确认后客户若被停用，不阻止基于该已确认或部分出库订单创建、更新或过账销售出库；销售出库仍必须校验订单状态、来源行、物料、单位、仓库和库存。
- 销售单价只作为销售业务记录字段，不生成应收、不计算税额、不确认收入、不形成正式财务凭证。
- 本阶段统一使用物料基本单位，不做销售包装单位或单位换算。
- 已过账销售出库不可编辑，不支持反过账；必要修正通过库存调整并保留说明。
- 销售出库库存不足必须拒绝，不允许负库存。

### 一期后第三个业务模块

一期后第三个业务模块确认为财务往来基础模块，采用应收台账 + 收款记录 + 应付台账 + 付款记录 + 来源追溯范围。

纳入基于已过账销售出库生成应收草稿、应收确认、取消、关闭、收款草稿、收款过账、应收余额更新、销售来源追溯；基于已过账采购入库生成应付草稿、应付确认、取消、关闭、付款草稿、付款过账、应付余额更新、采购来源追溯；权限控制、审计记录和浏览器视觉分析。该阶段不进入总账、凭证、会计科目、会计期间、结账、税务申报、完整发票生命周期、银行对账、报销、固定资产、工资、多币种汇兑损益、多组织、多公司、多账套、多租户、正式收入确认、正式成本结转、销售退货、采购退货、退款、退票或 BI 报表。

原因：

- 采购管理和销售管理基础已经形成可信采购入库、销售出库和库存追溯来源，下一步需要把采购、销售业务结果沉淀为往来台账。
- 应收和应付是正式财务核算之前的关键业务桥梁，先做业务台账与收付款登记，可以避免直接引入凭证、科目和结账造成范围过大。
- 收付款登记必须保证余额、状态和来源一致，为后续正式财务、发票、对账和报表能力提供可信业务数据基础。
- 将财务往来独立建模，可以避免把收付款逻辑混入采购、销售或库存服务。

已确认口径：

- 财务往来基础只做业务台账、收付款登记、余额核销和来源追溯，不承诺正式财务核算结果。
- 应收来源固定为已过账销售出库，金额按销售出库明细数量和来源销售订单单价形成业务金额快照。
- 应付来源固定为已过账采购入库，金额按采购入库明细数量和来源采购订单单价形成业务金额快照。
- 同一销售出库明细只能生成一次应收来源；同一采购入库明细只能生成一次应付来源。
- 应收状态包含草稿、待收款、部分收款、已收清、已关闭和已取消。
- 收款状态包含草稿、已过账和已取消；已过账收款不可编辑、不可取消。
- 应付状态包含草稿、待付款、部分付款、已付清、已关闭和已取消。
- 付款状态包含草稿、已过账和已取消；已过账付款不可编辑、不可取消。
- 收付款过账必须在事务内更新核销明细、累计金额、未结余额和状态；失败不得产生部分写入。
- 收款金额不得超过未收金额，付款金额不得超过未付金额。
- 页面文案必须明确“财务往来、应收台账、收款记录、应付台账、付款记录、来源追溯”，不得表达为总账、凭证、税务、完整发票或 BI 报表能力。

### 成本归集一期深度

一期成本归集做到业务记录，不承诺正式财务核算结果。

业务记录应至少能说明：

- 成本来源。
- 关联生产工单和产品。
- 成本类型。
- 成本金额或数量口径。
- 记录人和记录时间。
- 后续可追溯的来源单据。

正式核算结果需要稳定的物料、库存、工单、领料、报工、入库和财务规则支撑，后续阶段再展开。

### 一期后第四个业务模块

一期后第四个业务模块确认为经营报表基础模块，采用固定口径经营报表 + 来源追溯 + 经营异常清单范围。

纳入经营概览、销售经营汇总、采购经营汇总、库存收发存汇总、生产执行汇总、成本归集汇总、应收应付与收付款汇总、经营异常清单、报表权限控制、来源追溯、浏览器验收和视觉分析。该阶段不进入可配置 BI、拖拽报表、自定义 SQL、总账、凭证、会计科目、会计期间、结账、正式财务报表、税务申报、银行对账、多组织、多公司、多账套、多租户、销售退货、采购退货、退款、反冲、期间关账或经营大屏。

原因：

- 当前采购、销售、库存、生产、成本和财务往来基础模块已经形成可信业务数据，需要固定经营视图帮助管理者核对业务状态。
- 报表基础可以反向验证跨模块数据一致性，暴露来源追溯、权限和业务状态口径问题。
- 采用固定口径报表可以避免过早建设复杂 BI 平台，也避免把经营金额误交付为正式财务报表。
- 报表基础稳定后，后续退货退款、业务反冲和期间关账才有明确的数据影响范围。

已确认口径：

- 报表一期只做固定口径经营报表，不做可配置报表或拖拽式 BI。
- 报表只读，不生成、修改或删除任何业务单据。
- 销售金额以已过账销售出库和来源销售订单单价形成经营金额，不等同正式收入确认。
- 采购金额以已过账采购入库和来源采购订单单价形成经营金额，不等同正式成本入账。
- 库存收发存第一版按库存流水业务日期和流水顺序推算，不建立期间快照。
- 成本归集报表只展示业务成本记录，必须明确不属于正式财务核算。
- 应收应付与收付款报表只展示业务往来台账和余额，不属于总账余额表。
- 所有报表结果必须能追溯来源；来源详情跳转继续受来源模块权限控制。

### 一期后第五个业务模块

一期后第五个业务模块确认为退货退款与业务反冲基础模块，采用销售退货 + 采购退货 + 生产退料/补料 + 往来冲减 + 经营报表净额同步范围。

纳入基于已过账销售出库的销售退货、退货入库库存流水、应收冲减和销售净额报表；基于已过账采购入库的采购退货、退货出库库存流水、应付冲减和采购净额报表；基于已过账生产领料和工单物料的生产退料、补料、库存反向或补充流水、成本业务净额；往来冲减和退款/付款冲减业务记录；来源追溯、权限控制、审计记录、浏览器验收和视觉分析。该阶段不进入审批流、正式红字发票、税务处理、银行退款支付、总账凭证、会计期间、结账、多组织、多币种、高级质量体系、复杂售后维修、换货维修流程或正式财务核算。

原因：

- 采购、销售、库存、生产、成本、往来和经营报表正向链路已完成，真实 ERP 使用中必须支持错误修正和反向业务闭环。
- 经营报表基础已经提供跨模块经营数据视图，若没有退货、退款、反冲和净额口径，报表数据无法解释真实业务变化。
- 已过账原单不可直接修改是当前系统数据可信度的核心原则，反向业务必须以新增记录方式表达。
- 先补齐退货退款与业务反冲，比直接进入期间关账、质量体系或计划排产更能保护已有业务数据可信度。

已确认口径：

- 已过账原单是历史业务事实，退货、退款、退料、补料和反冲不得直接改写或删除原单。
- 销售退货基于已过账销售出库行，过账后生成销售退货入库库存流水，并冲减应收业务余额。
- 采购退货基于已过账采购入库行，过账后生成采购退货出库库存流水，并冲减应付业务余额。
- 生产退料基于已过账生产领料行，过账后生成退料入库流水，并影响工单材料消耗和成本业务净额。
- 生产补料基于有效生产工单和工单物料行，过账后生成补料出库流水，并影响工单材料消耗和成本业务净额。
- 往来冲减只作为业务余额调整和来源追溯，不作为正式银行退款、税务、总账或凭证处理。
- 草稿反向单不影响库存、往来、成本和报表；已过账反向单才进入经营数据。
- 经营报表必须能区分原发生、反向发生和净额，并能追溯反向来源。
- 来源权限不足时继续脱敏，不能通过反向业务或报表绕过原模块权限。

## 待后续模块确认

- 审批流程是否进入账号权限模块一期。
- 数据范围是否按仓库、部门或业务线限制。
- 后续正式财务核算、凭证、科目、结账等能力另行立项确认，不属于财务往来基础模块。
- 生产计划与物料需求、批次追溯、库位管理、完整 QMS、SPC 和正式财务能力另行按后续阶段确认，不纳入 017。

## 决策变更规则

任何产品决策变更必须记录：

- 变更原因。
- 影响模块。
- 影响任务文档和验收标准。
- 是否影响当前阶段成果。
- 主代理决策结论或用户确认结果。
