# 角色权限草案

## 目标

建立一期权限讨论基础，避免后续账号、菜单、按钮和数据访问规则混乱。

## 建议基础角色

- 系统管理员：管理账号、角色、基础配置。
- 生产人员：查看和处理生产工单、报工等生产任务。
- 仓库人员：处理入库、出库、领料和库存查询。
- 采购人员：处理供应商、采购需求和采购单据。
- 销售人员：处理客户、销售需求和订单相关信息。
- 财务人员：查看应收、应付、库存金额和成本归集信息。
- 管理者：查看关键报表和跨模块数据。

## 权限维度

- 菜单权限。
- 按钮权限。
- 字段可见和可编辑权限。
- 数据范围权限。
- 单据状态下的操作权限。

## 020 销售项目与合同权限

销售项目位于现有销售菜单下，菜单权限为 `sales:project`。项目与合同查看、维护和状态动作独立授权，查看项目不自动获得合同或关联销售订单的查看权限。

### 销售项目权限

- `sales:project:view`：查看销售项目。
- `sales:project:create`：创建销售项目。
- `sales:project:update`：更新允许状态下的销售项目字段。
- `sales:project:activate`：激活销售项目。
- `sales:project:close`：关闭销售项目。
- `sales:project:cancel`：取消草稿销售项目。

负责人候选能力允许具备 `sales:project:view`、`sales:project:create` 或 `sales:project:update` 任一权限的用户调用，只返回启用用户的 `userId`、`username`、`displayName`，不要求系统用户列表查看权限。

### 销售合同权限

- `sales:contract:view`：查看项目合同及合同摘要。
- `sales:contract:create`：创建主合同或满足前置条件的补充合同。
- `sales:contract:update`：更新合同草稿。
- `sales:contract:activate`：激活合同，使状态进入 `EFFECTIVE`。
- `sales:contract:close`：正常关闭已生效合同。
- `sales:contract:terminate`：提前终止已生效合同。
- `sales:contract:cancel`：取消合同草稿。

### 销售订单关联权限

- `sales:order:create`：创建销售订单时使用最小项目合同候选并保存可选关联。
- `sales:order:update`：更新草稿销售订单时关联、切换或解除项目合同。
- `sales:order:confirm`：确认销售订单时再次校验已有项目合同关联。
- `sales:order:view`：在具备项目查看权限的前提下，查看项目下关联销售订单数量、摘要和列表。

销售订单项目合同候选能力由创建或更新销售订单权限控制，不要求完整项目或合同查看权限，也不新增独立的项目关联权限。

### 受限信息保护

- 无 `sales:contract:view` 时，项目列表和详情中的合同字段返回 `null` 或空集合，合同数量返回 `null`，并标记合同摘要受限；不得泄露合同编号、金额、状态或数量。
- 无 `sales:order:view` 时，项目列表和详情中的销售订单数量与摘要返回 `null` 或空，标记订单摘要受限，前端不得显示为 0，也不得泄露订单号、金额、状态或行明细。
- 有订单查看权限但无合同查看权限时，可查看订单字段，但订单摘要或列表中的合同标识和编号必须隐藏。
- 项目操作记录同样按合同和订单查看权限过滤或脱敏，不能通过审计摘要绕过上述限制。

## 一期落地边界

- 必须落地菜单访问、页面查看、按钮或操作、后端接口权限。
- 字段可见、字段可编辑和数据范围权限先建立口径，不作为账号权限首个模块完整配置界面。
- 020 已对项目中的合同与销售订单摘要实施固定的服务端受限返回规则；这不等同于已实现可配置的通用字段级权限体系。
- 权限不能只依赖前端隐藏，后端接口必须强制校验。
- 权限变更以后端实时校验为准，前端通过刷新当前用户信息更新菜单和按钮。

## 一期暂不深入

- 岗位和角色分离：一期先按角色授权，不单独建立岗位授权体系。
- 审批权限：账号权限首个模块不实现完整审批流，仅保留后续扩展口径。
- 数据范围：按仓库、部门或业务线限制数据范围先保留模型口径，不实现复杂配置界面。
