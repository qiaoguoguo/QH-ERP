# 电气制造业全量演示数据重建实施计划

> 本计划执行 `docs/data/electrical-manufacturing-demo-data.md`，不改变 000 至 023 业务边界。固定五角色通过已登记的 Codex 后台线程执行，主代理只负责统筹、文档、Git 和交付判断。

## 全局约束

- 权威分支：`codex/acceptance-demo-handoff`，基准提交 `78d8030665463c90607d17f43be0cd9ce4b9f709`。
- 演示数据不进入 `apps/api/src/main/resources/db/migration`，不得修改 V1 至 V25；重建过程中发现的业务结构缺陷必须通过独立前向迁移修复，迁移中不得包含演示事实。
- 核心业务事实通过真实 API/业务服务创建；不得直接写库存余额、计价池、项目成本层、价值流水、审批终态、文档任务终态或附件元数据。
- 清理只针对 `qherp-postgres`、`qherp-minio` 和 QH ERP 宿主服务；先备份，显式确认目标后执行。
- 实现与测试使用固定后端、测试角色；前端角色只做 API/路由契约核对和后续浏览器路径，不修改产品页面，除非真实缺陷要求。
- 开发期只运行演示工具定向验证；唯一交付前窗口才运行项目全量验证。
- 浏览器与视觉只检查桌面端，不截图、不验收移动端；响应式仍是开发约束。

## 工作包一：独立验证器失败基线

**所有者：** 测试角色。

**目标：** 在演示数据生成器之前建立可重复验收器，并在当前非全量验收库上观察预期失败，证明验证规则有效。

**文件：**

- 新建 `tools/demo-data/validate-demo-data.ps1`
- 新建 `tools/demo-data/sql/validate-demo-data.sql`
- 可按需新建 `tools/demo-data/README.md` 的验证器参数章节；不得编辑生成器文件。

**要求：**

- 参数显式接收 API 地址、数据库容器/数据库名、管理员或专用只读凭据；不得提交真实密码。
- 检查 Flyway V25、账号角色、主数据/BOM/ECO、采购销售生产财务、退货反冲、业务期间、质量、预留、批次序列、020-023 对象数量和状态分布。
- 检查库存无负数、公共池与公共余额、项目层与项目余额、价值流水来源、调拨总价值、MinIO 元数据/对象、审批/消息/任务/审计。
- 失败输出必须列出具体规则、实际值和期望值；成功时输出机器可读摘要。
- 先在当前验收库运行，至少因空业务表和不足数量失败；记录命令和失败规则，不把环境连接错误当成有效红色基线。

## 工作包二：演示数据运行器与安全重建入口

**所有者：** 后端角色。

**前置：** 工作包一已有可运行、因数据缺口正确失败的验证器。

**文件：**

- 新建 `tools/demo-data/rebuild-acceptance.ps1`
- 新建 `tools/demo-data/generate-demo-data.ps1`，或在 `tools/demo-data/` 下采用等价、独立且不暴露生产入口的实现结构
- 新建必要的 `tools/demo-data/assets/` 文本/CSV 演示附件或导入文件
- 新建运行器自己的定向测试；不得修改 V1 至 V25

**要求：**

- 重建入口先校验仓库、目标容器、数据库名和显式确认参数，再备份 PostgreSQL/MinIO、停止 QH ERP 宿主服务、重建目标数据、启动 API、执行生成、运行验证器。
- 生成器通过真实登录和 API/业务服务建立 `docs/data/electrical-manufacturing-demo-data.md` 中的账号、主数据、BOM、业务期间、采购、质量、库存、生产、销售、财务、反冲、022 平台和 023 价值链。
- 使用稳定业务编码、运行 ID 和幂等键，不依赖固定数据库 ID；失败时标明阶段和已生成对象。
- 附件必须写入 MinIO；打印、导入导出和文档任务必须由真实 worker 完成并等待到可验证终态。
- 输出 manifest 至被 Git 忽略的构建/临时目录，包含 Git 提交、Flyway 版本、对象编码/ID、文件 key、任务 ID 和关键余额金额。
- 禁止一大事务包住全部生成；按依赖阶段提交并在失败后安全退出。
- 开发期运行生成器定向测试和验证器，不运行全项目测试。

## 工作包三：契约核对与集中审查

**所有者：** 固定五角色并行，主代理汇总。

- 产品经理核对 000 至 023 覆盖、真实电气制造叙事和 024+ 排除项。
- UI 设计师核对编码、状态、数据密度和桌面页面可见性；此时只读代码/数据，不截图。
- 前端开发核对运行器调用的 API、候选、分页、版本、幂等、`allowedActions`、审批/消息深链和成本脱敏契约。
- 后端开发不自审自身代码；只响应其他角色问题并修复归属后端的缺陷。
- 测试核对验证器规则、红绿证据、异常、权限、对象存储和数量价值一致性。

主代理合并问题后只派发一轮集中整改；阻断与严重必须闭合，一般项记入后续清单。差异复审只看整改差异。

## 工作包四：清库重建与唯一交付前验证

**所有者：** 测试角色执行；UI 设计师执行桌面视觉检验；主代理只统筹和记录。

1. 在明确的临时目录完成旧验收库 PostgreSQL/MinIO 备份。
2. 执行安全重建入口，确认空库 V1 至 V25、权限初始化、全量演示数据和独立验证器通过。
3. 执行后端全量测试。
4. 执行前端全量测试、类型检查和生产构建。
5. 核对迁移、权限、审计、附件、任务、库存数量价值和来源追溯。
6. 从候选分支启动前后端，测试角色执行桌面核心路径，UI 角色检验全量演示页面的信息表达；不截图、不测试移动端。
7. 核对 `git diff --check`、工作树范围、Docker 健康、后端健康、前端 HTTP 和残留测试进程。

任何迁移失败、权限绕过、库存不一致、对象存储断链、任务假完成或核心页面不可用都阻断合入。

## 变更记录

- 2026-07-15：临时空库的真实 API 生成在计价物料生产补料过账时暴露 `inv_value_movement.movement_type varchar(32)` 无法容纳 34 字符枚举的问题。该问题阻断生产补料价值流水，不能通过调整演示数据绕过，因此在本计划范围内追加独立 V25 前向迁移和回归测试。V25 只扩展字段长度，不写入演示事实；验证器最新版本规则同步到 V25，总规则数保持 114。
- 在演示数据基线中写入最终运行 ID、账号用途、精确数量、备份位置、生成命令和验证结果；密码只写环境变量名或本地交付方式。
- 提交分支，确认最终验证对应提交；按纯快进或项目允许方式合入 `main` 并推送 `origin/main`。
- 从主分支重启服务，再核对健康、登录、关键列表和远端哈希。
- 只有工作树干净、本地 `main` 与 `origin/main` 一致、交接无陈旧口径、演示库验证通过时，才声明可由新代理接手。
