# 账号与权限基础模块设计

## 目标

建立 QH ERP 首个业务基础模块：企业内部用户可登录系统，管理员可维护用户、角色和权限，后端按最新授权强制保护接口，前端按权限展示菜单和按钮。

## 范围

本设计覆盖登录、退出、当前用户、初始超级管理员、用户管理、角色管理、角色权限配置、用户角色分配、菜单权限、按钮权限、后端接口鉴权、停用账号拦截和基础审计。

不覆盖外部账号、多租户、复杂审批、单点登录、第三方登录、完整字段级权限配置界面和复杂数据范围配置界面。

## 方案取舍

采用“服务端 Session + HttpOnly Cookie + 后端实时鉴权”的方案。内部 ERP 更需要可控退出、停用账号即时失效和服务端可信权限判断。前端只负责体验层的菜单、按钮和路由表现，不能作为安全边界。

权限变更以后端实时校验为准；前端菜单和按钮在刷新当前用户信息或重新登录后更新。这样既避免权限绕过，又降低前端状态同步复杂度。

## 后端设计

后端建立 `sys_user`、`sys_role`、`sys_permission`、`sys_user_role`、`sys_role_permission` 和 `sys_audit_log`。用户、角色、权限使用软状态停用，不物理删除。用户名、角色编码、权限编码唯一。用户、角色和权限变更写入审计日志。

接口分为认证、用户、角色、权限树和审计查询。所有管理接口必须登录并具备对应权限编码。停用账号、停用角色和无权限请求必须由后端拒绝。

## 前端设计

前端新增账号权限模块页面：登录页、后台布局、用户管理、角色管理、角色权限配置、用户角色分配和无权限页。Pinia 保存当前用户、菜单和权限编码；路由守卫判断未登录和无页面权限；按钮按权限编码显示或禁用。

页面统一使用 ERP 后台模式：查询区、工具栏、表格、分页、标准表单、权限树、确认弹窗、状态标签、加载、空状态和错误反馈。

## 数据流

用户登录后，后端创建服务端会话并通过 HttpOnly Cookie 维持登录态。前端调用当前用户接口获取用户信息、角色、菜单树和权限编码。页面渲染依据后端返回结果；关键操作调用后端接口，后端重新校验账号状态、角色状态和接口权限。

## 错误处理

未登录返回认证错误；无权限返回权限错误；账号停用返回账号停用错误；参数错误返回字段明细；唯一约束冲突返回业务冲突。前端将错误转成清晰提示，不能静默失败或空白页。

## 测试策略

后端使用 JUnit、Spring Boot Test 和 Testcontainers 覆盖迁移、初始化数据、认证、权限、停用、唯一约束和审计。前端使用 Vitest 覆盖登录态、路由守卫、权限显示、表单校验和页面状态。阶段验收必须启动本地数据库、后端和前端，通过浏览器完成核心路径。

## 自检结论

本设计无未完成条目；范围聚焦账号与权限基础模块；接口、数据、权限、前端和测试边界与项目开工准备基线一致。
