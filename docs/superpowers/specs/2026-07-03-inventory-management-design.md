# 仓库与库存基础设计规格

## 目标

仓库与库存基础阶段建立库存余额、库存变动流水、期初库存和库存调整能力。系统必须保证库存变化可追溯、权限可控、数量可信，并为后续生产工单、领料、完工入库和成本归集提供稳定基础。

## 阶段边界

本阶段只覆盖库存基础能力：

- 库存余额查询。
- 库存变动流水查询。
- 期初库存单据草稿、编辑和过账。
- 库存调整单据草稿、编辑和过账。
- 权限控制、审计记录、异常反馈和浏览器视觉分析。

本阶段不覆盖：

- 生产工单、领料、报工、完工入库。
- 成本核算、采购销售单据、财务凭证。
- 批次、库位、序列号、条码、调拨、盘点任务。
- 库存占用锁定、缺料分析、先进先出、多单位换算。
- 多组织、多公司、多租户。

## 已确认业务口径

- 库存余额维度为“仓库 + 物料”。
- 一期不允许负库存。
- 期初同一仓库物料只允许过账一次。
- 已过账单据不可编辑；修正通过新调整单完成。
- 调整单使用调整数量录入，并展示变动前和变动后数量。
- 一期可用数量等于现存数量，锁定数量固定为 0 或仅响应预留。
- 只允许按物料基本单位入账，不做单位换算。
- 所有影响库存余额的动作必须生成库存变动流水和审计日志。

## 核心对象

### 库存余额

库存余额记录某个物料在某个仓库中的当前数量。

关键字段：

- 仓库。
- 物料。
- 基本单位。
- 现存数量。
- 锁定数量。
- 可用数量。
- 更新时间。

### 库存变动流水

库存变动流水记录每一次库存余额变化，是库存追溯的核心凭据。

关键字段：

- 变动编号。
- 变动类型。
- 变动方向。
- 仓库。
- 物料。
- 单位。
- 变动前数量。
- 变动数量。
- 变动后数量。
- 来源类型。
- 来源单据和来源明细。
- 业务日期。
- 原因、备注、操作人和发生时间。

### 库存单据

库存单据承载期初和调整两类库存来源。

状态：

- 草稿：可编辑、可过账。
- 已过账：已写入余额和流水，不可编辑。

类型：

- 期初库存。
- 库存调整。

## 后端设计

### 数据表

建议新增：

- `inv_stock_balance`：库存余额。
- `inv_stock_movement`：库存变动流水。
- `inv_inventory_document`：库存单据主表。
- `inv_inventory_document_line`：库存单据明细。

关键约束：

- 库存余额对 `warehouse_id + material_id` 建唯一约束。
- 库存变动 `movement_no` 全局唯一。
- 库存变动对 `source_type + source_line_id` 建唯一约束，防止重复入账。
- 单据明细对 `document_id + line_no` 建唯一约束。

### 事务规则

过账必须在单个数据库事务内完成：

1. 校验单据存在且为草稿。
2. 校验仓库、物料、单位有效。
3. 校验同一仓库物料期初未过账。
4. 锁定库存余额行；余额不存在时先创建 0 余额再锁定。
5. 计算变动前、变动数量、变动后数量。
6. 调减时校验变动后数量不小于 0。
7. 写入库存变动流水。
8. 更新库存余额。
9. 更新单据状态为已过账。
10. 写入审计日志。

### 接口范围

- `GET /api/admin/inventory/balances`
- `GET /api/admin/inventory/movements`
- `GET /api/admin/inventory/documents`
- `GET /api/admin/inventory/documents/{id}`
- `POST /api/admin/inventory/documents`
- `PUT /api/admin/inventory/documents/{id}`
- `PUT /api/admin/inventory/documents/{id}/post`

### 权限

- `inventory`
- `inventory:balance:view`
- `inventory:movement:view`
- `inventory:document:view`
- `inventory:document:create`
- `inventory:document:update`
- `inventory:document:post`

## 前端设计

### 路由

- `/inventory/balances`：库存余额。
- `/inventory/movements`：库存变动流水。
- `/inventory/documents`：库存单据。
- `/inventory/documents/create`：新建库存单据。
- `/inventory/documents/:id`：库存单据详情。
- `/inventory/documents/:id/edit`：编辑库存单据草稿。

### 页面

库存余额页：

- 查询区：物料关键词、仓库、物料类型、只看有库存。
- 表格：物料编码、物料名称、规格、仓库、单位、现存数量、可用数量、更新时间、操作。
- 操作：查看变动流水。

库存变动页：

- 查询区：物料、仓库、变动类型、方向、业务日期范围。
- 表格：发生时间、物料、仓库、变动类型、方向、变动前、变动数量、变动后、来源单据、操作人。
- 本页只读。

库存单据页：

- 查询区：单据类型、状态、业务日期、关键词。
- 表格：单据编号、类型、状态、业务日期、明细数、创建人、过账人、更新时间、操作。
- 草稿可编辑和过账，已过账只读。

库存单据表单：

- 主表：单据类型、业务日期、原因、备注。
- 明细：仓库、物料、单位、调整方向、数量、备注。
- 期初单不显示调整方向。
- 调整单显示调整方向。

## UI 与视觉验收

- 页面保持后台 ERP 信息密度，使用现有列表、查询区、表格、弹窗或抽屉模式。
- 数量列右对齐，变动前、变动数量、变动后成组展示。
- 正负方向不能只靠颜色识别，必须有文字或符号。
- 期初和调整入口必须明确区分，避免误操作。
- 过账操作必须二次确认，说明会影响库存余额。
- 只读用户不显示创建、编辑、过账按钮。
- 窄屏下表格允许横向滚动，表单按钮不遮挡内容。

## 测试与验收

自动化测试必须覆盖：

- 数据库迁移和权限初始化。
- 库存余额查询、库存变动查询和库存单据查询。
- 期初创建、更新、过账和重复期初拒绝。
- 调增、调减和库存不足拒绝。
- 重复过账拒绝。
- 并发过账同一仓库物料不丢失更新。
- 停用仓库、停用物料和非基本单位拒绝。
- 未登录、无权限和 CSRF 拒绝。
- 审计日志写入。
- 前端路由、权限、列表、表单、动态明细、错误反馈和窄屏展示。

浏览器验收必须覆盖：

- 管理员完成期初、余额查询、流水追溯、调增、调减和异常验证。
- 仓库人员按权限完成查询、创建或过账。
- 只读用户只能查看，写按钮不可见，直接写接口被拒绝。
- 视觉分析截图与结论保存到 `docs/testing/inventory-management-visual-audit/`。

## 风险与处理

- 余额和流水不一致：通过单事务过账和来源明细唯一约束控制。
- 并发过账错误：过账时锁定仓库物料余额行。
- 负库存：调减时强制校验变动后数量不小于 0。
- 重复期初：过账时检查同一仓库物料是否已有期初流水。
- 数量精度：使用 `NUMERIC(18,6)` 和 `BigDecimal`。
- 权限绕过：前端按钮只做体验，后端鉴权为最终边界。

## 子代理讨论结论

- 产品经理建议：阶段目标是建立可信库存余额和变动追溯底座，严格排除生产、采购销售和成本实现。
- UI 设计师建议：围绕库存余额、流水、期初和调整构建工作型页面，重点保证数量识别、误操作提示和窄屏可用。
- 前端开发建议：新增 `inventory` 模块、库存 API、余额/流水/单据页面和动态明细编辑器，复用现有权限和页面模式。
- 后端开发建议：采用余额表、流水表、库存单据主从表，过账时通过事务、锁和唯一约束保证一致性。
- 测试建议：重点覆盖权限、数量精度、库存不可为负、重复期初、并发调整、审计和浏览器视觉分析。
