# 采购管理基础模块设计规格

## 目标

采购管理基础模块建立制造业原材料进入企业的业务入口，让系统从“期初和库存调整产生库存”的验收型来源，推进到“供应商采购入库产生库存”的真实业务来源。

本阶段主链路为：

```text
供应商 -> 采购订单 -> 采购入库 -> 库存增加 -> 生产领料 -> 报工 -> 完工入库 -> 成本归集
```

本阶段只建立采购订单、采购入库、库存增加和来源追溯能力，不承诺应付、发票、付款、税务、审批、MRP、质检入库、采购退货或正式财务结果。

## 阶段边界

本阶段覆盖：

- 采购订单列表、详情、新建草稿、编辑草稿、确认、取消和关闭。
- 采购订单明细维护，包括物料、数量、单位、采购单价、预计到货日期、备注、已入库数量和未入库数量。
- 基于已确认或部分入库采购订单创建采购入库草稿。
- 采购入库草稿编辑、详情查看和过账。
- 采购入库过账后增加库存余额，生成 `PURCHASE_RECEIPT` 库存流水。
- 采购订单、采购入库和库存流水三方追溯。
- 采购管理菜单、采购订单权限、采购入库权限、后端接口鉴权和审计记录。
- 管理员、采购员、仓库角色、只读用户和无权限用户的权限差异验收。
- 自动化测试、本地部署、浏览器功能验收和视觉分析。

本阶段不覆盖：

- 应付账款、发票、付款、税务、暂估入库、总账联动或正式财务凭证。
- 采购审批流、询价、比价、供应商评分、采购价格策略和价格历史分析。
- MRP 自动采购建议、采购需求计划、报表看板或经营分析。
- 采购退货、到货检验、质检入库、待检库存、让步接收。
- 多币种、多组织、多公司、多账套、多租户。
- 批次、序列号、库位、条码、采购包装单位和多单位换算。

采购单价只作为采购业务记录字段和后续扩展基础，不生成应付、不计算税额、不形成正式成本核算结果。

## 已确认业务口径

- 采购作为独立业务域，不并入库存单据或生产域。
- 采购订单是采购业务承诺，采购入库是库存实际入账动作。
- 采购订单草稿、确认、取消和关闭本身不得改变库存。
- 采购入库草稿不得改变库存，不得生成库存流水。
- 采购入库过账必须在单个事务内完成库存余额增加、库存流水生成、订单累计入库更新、入库单状态更新和审计记录。
- 采购入库过账后库存默认可用；质检和待检能力不在本阶段。
- 已过账采购入库不可编辑，不支持反过账；错误修正通过后续库存调整并保留说明。
- 本阶段统一使用物料基本单位，不做采购包装单位或单位换算。
- 入库数量必须大于 0，且不得超过采购订单行未入库数量。
- 停用供应商不允许新建或确认采购订单。
- 停用物料不允许新增采购订单明细。
- 停用仓库不允许创建或过账采购入库。
- 已确认历史订单的后续入库主要由订单状态控制，不因供应商后续停用破坏在途链路；但物料、单位、仓库在入库过账时仍需保持有效。
- 库存流水来源类型必须能清晰识别为采购入库。
- 前端菜单和按钮权限只作为体验控制，后端接口鉴权是最终安全边界。

## 核心对象

### 采购订单

采购订单承载供应商、订单日期、订单状态、采购物料、采购数量、采购单价、预计到货日期和累计入库进度。

状态：

- `DRAFT`：草稿，可编辑、可确认、可取消，不允许入库。
- `CONFIRMED`：已确认，可创建入库，可关闭，不允许编辑关键字段。
- `PARTIALLY_RECEIVED`：部分入库，可继续入库，可关闭未入库余量。
- `RECEIVED`：全部入库，不再允许继续入库，可关闭归档。
- `CLOSED`：已关闭，不允许入库。
- `CANCELLED`：已取消，不允许入库。

### 采购订单明细

采购订单明细记录具体采购物料和数量口径。

关键字段：

- 行号。
- 物料。
- 单位，使用物料基本单位。
- 采购数量。
- 已入库数量。
- 未入库数量。
- 采购单价。
- 预计到货日期。
- 备注。

### 采购入库

采购入库必须基于采购订单创建，过账后增加库存余额并生成库存流水。

状态：

- `DRAFT`：草稿，可编辑、可过账。
- `POSTED`：已过账，已写库存余额和库存流水，不可编辑，不可重复过账。

采购入库单明细必须引用采购订单明细，物料、单位和来源行必须一致。

## 后端设计

### 领域边界

建议新增包：

- `com.qherp.api.system.procurement`

采购域负责采购订单、采购入库、采购状态流转和采购追溯。库存域负责库存余额、库存流水和过账一致性。采购域不得直接绕过库存追溯规则修改库存余额。

### 数据表

建议新增迁移：

- `apps/api/src/main/resources/db/migration/V8__procurement_management_schema.sql`

建议新增表：

- `proc_purchase_order`：采购订单主表。
- `proc_purchase_order_line`：采购订单明细。
- `proc_purchase_receipt`：采购入库主表。
- `proc_purchase_receipt_line`：采购入库明细。

建议修改：

- `inv_stock_movement` 的变动类型约束增加 `PURCHASE_RECEIPT`。

关键约束：

- 采购订单号全局唯一。
- 采购入库单号全局唯一。
- 明细行号在单据内唯一。
- 采购数量和入库数量必须大于 0。
- 已入库数量不得小于 0，且不得超过采购数量。
- 采购单价使用 `NUMERIC(18,6)`，允许为 0，不允许负数。
- 库存流水继续使用 `source_type + source_line_id` 唯一约束防止重复入账。

### 库存过账复用

推荐抽取库存域内部过账端口：

- `InventoryPostingService`

该端口统一处理：

- 锁定或创建库存余额。
- 校验库存方向和数量。
- 写入库存流水。
- 更新库存余额。
- 返回过账前后库存数量。
- 使用 `source_type + source_line_id` 防止重复来源入账。

采购入库过账调用该端口，写入：

- `movement_type = PURCHASE_RECEIPT`
- `direction = IN`
- `source_type = PURCHASE_RECEIPT`
- `source_id = proc_purchase_receipt.id`
- `source_line_id = proc_purchase_receipt_line.id`

### 事务规则

确认采购订单必须在单个事务内完成：

1. 锁定采购订单。
2. 校验订单为 `DRAFT`。
3. 校验供应商启用。
4. 校验明细非空。
5. 校验物料、单位、数量、采购单价和预计到货日期。
6. 更新订单状态为 `CONFIRMED`。
7. 写入审计日志。

采购入库过账必须在单个事务内完成：

1. 锁定采购入库单。
2. 校验入库单为 `DRAFT`。
3. 锁定采购订单和订单明细。
4. 校验订单状态为 `CONFIRMED` 或 `PARTIALLY_RECEIVED`。
5. 校验仓库、物料、单位和剩余未入库数量。
6. 调用库存过账端口写入 `PURCHASE_RECEIPT` 库存流水并更新余额。
7. 回写入库明细过账前后库存。
8. 更新订单行已入库数量。
9. 按累计进度推进订单状态为 `PARTIALLY_RECEIVED` 或 `RECEIVED`。
10. 更新入库单状态为 `POSTED`。
11. 写入审计日志。

异常失败时不得产生部分库存余额、部分库存流水或错误累计入库数量。

### 接口范围

- 采购订单：列表、详情、创建草稿、更新草稿、确认、取消、关闭。
- 采购入库：列表、详情、基于订单创建草稿、更新草稿、过账。
- 追溯：采购订单详情包含入库记录摘要；采购入库详情包含来源订单；库存流水可追溯到采购入库来源。

## 前端设计

### 路由

- `/procurement`：重定向到 `/procurement/orders`。
- `/procurement/orders`：采购订单列表。
- `/procurement/orders/create`：新建采购订单。
- `/procurement/orders/:id`：采购订单详情。
- `/procurement/orders/:id/edit`：编辑采购订单草稿。
- `/procurement/orders/:id/receipts/create`：基于采购订单创建采购入库。
- `/procurement/receipts`：采购入库列表。
- `/procurement/receipts/:id`：采购入库详情。
- `/procurement/receipts/:id/edit`：编辑采购入库草稿。

### 页面

采购订单列表：

- 筛选项包括关键词、供应商、状态、预计到货日期范围。
- 表格展示订单号、供应商、订单日期、预计到货日期、状态、行数、采购数量摘要、已入库进度和操作。
- 操作列按状态和权限展示详情、编辑、确认、取消、关闭和创建入库。

采购订单表单：

- 使用整页表单，不使用普通小弹窗。
- 基础信息包括供应商、订单日期、预计到货日期和备注。
- 明细表包括物料、数量、单位、采购单价、预计到货日期和备注。
- 数量和单价以前端字符串提交，避免浮点误差。

采购订单详情：

- 顶部摘要展示状态、供应商、订单行数、累计入库、未入库和关键操作。
- 分区展示基础信息、采购明细、入库记录、库存流水摘要和审计摘要。
- 确认、取消、关闭使用二次确认。

采购入库列表：

- 筛选项包括关键词、供应商、状态、业务日期和来源订单。
- 表格展示入库单号、来源订单、供应商、入库仓库、业务日期、状态、行数和操作。

采购入库表单：

- 上方展示来源订单摘要。
- 主表字段包括业务日期、入库仓库和备注。
- 明细表默认带出订单未入库数量，录入本次入库数量。
- 明确提示不可超过未入库数量。

采购入库详情：

- 展示入库状态、来源订单、仓库、业务日期、明细、过账前后库存和库存流水来源。
- 已过账状态下所有编辑入口不可用。

### 视觉设计

- 延续现有后台 ERP 的高密度表格、整页表单和详情分区。
- 状态使用克制标签展示。
- 数量、单价和金额类字段右对齐。
- 操作列使用稳定宽度。
- 长单号、物料名称和供应商名称在列表中省略，详情页可查看完整值。
- 窄屏下筛选项和表单字段纵向排列，明细表允许横向滚动。
- 采购入库与库存单据入库必须通过页面标题和来源文案区分，避免业务语义混淆。

## 权限设计

建议权限点：

- `procurement`：采购管理菜单。
- `procurement:order:view`：查看采购订单。
- `procurement:order:create`：创建采购订单。
- `procurement:order:update`：更新采购订单草稿。
- `procurement:order:confirm`：确认采购订单。
- `procurement:order:cancel`：取消采购订单。
- `procurement:order:close`：关闭采购订单。
- `procurement:receipt:view`：查看采购入库。
- `procurement:receipt:create`：创建采购入库。
- `procurement:receipt:update`：更新采购入库草稿。
- `procurement:receipt:post`：过账采购入库。

角色口径：

- 管理员：采购订单和采购入库全量权限。
- 采购员：订单查看、创建、更新、确认、取消、关闭，入库查看。
- 仓库角色：订单查看，入库查看、创建、更新、过账，库存流水查看。
- 只读用户：订单、入库和追溯查看。
- 无权限用户：无采购菜单和采购接口权限。

## 错误处理

必须提供明确业务错误：

- 采购订单不存在。
- 采购入库单不存在。
- 订单状态不允许当前操作。
- 入库单状态不允许当前操作。
- 供应商不存在或已停用。
- 物料不存在、已停用或来源类型不允许采购。
- 单位不存在、已停用或不是物料基本单位。
- 仓库不存在或已停用。
- 订单明细为空。
- 采购数量、入库数量或采购单价不合法。
- 入库数量超过未入库数量。
- 来源订单行与入库物料不一致。
- 已过账采购入库不可编辑。
- 重复过账采购入库。
- 来源明细已生成库存流水。

前端展示原则：

- 接口业务错误展示为中文可理解文案。
- 表单校验错误贴近字段展示。
- 顶部保留错误摘要，便于密集表单扫描。
- 过账失败后保留当前输入并恢复按钮状态。
- 权限不允许的操作不可见或禁用，后端仍必须拒绝越权请求。

## 测试策略

后端测试：

- 数据库迁移、库存流水枚举约束和权限种子。
- 采购订单创建、更新、确认、取消、关闭。
- 草稿外不可编辑，取消、关闭、已入库后不可创建入库。
- 基于已确认订单创建采购入库。
- 采购入库过账后库存余额增加，库存流水为 `PURCHASE_RECEIPT`。
- 订单行已入库数量和订单状态正确推进。
- 重复过账不重复增加库存或流水。
- 入库数量为 0、负数、超精度、超过未入库数量均失败且不改变库存。
- 停用供应商、停用物料、停用仓库被拒绝。
- 来源订单行与入库物料不一致被拒绝。
- 接口鉴权、CSRF、审计记录和错误码。

## 五角色讨论结论

- 产品经理：采购阶段应正式启动，范围锁定采购订单、采购入库、库存增加和来源追溯；保留部分入库和已入库状态。
- UI 设计师：采购管理作为独立菜单，页面采用整页列表、整页表单、详情分区、状态标签和真实浏览器视觉分析。
- 前端开发：新增 `procurement` 前端模块和 `procurementApi`，按现有库存、生产模块模式实现路由、权限和测试。
- 后端开发：新增独立采购域，并优先抽取库存域统一过账端口，采购入库不得直接散写库存余额。
- 测试：核心质量门为库存增加、`PURCHASE_RECEIPT` 流水、三方追溯、权限一致、异常无部分写入和视觉分析归档。

## 主代理决策结论

- 下一阶段名称定为“采购管理基础模块”。
- 采用“独立采购域 + 库存域统一过账端口”方案。
- 采购订单状态保留 `PARTIALLY_RECEIVED` 和 `RECEIVED`，采购入库状态保留 `DRAFT` 和 `POSTED`。
- 采购入库过账必须生成 `PURCHASE_RECEIPT` 库存流水，并通过来源字段追溯到采购入库明细。
- 采购单价只作为业务记录字段，不进入应付、付款、税务、凭证或正式成本核算。
- 采购阶段验收必须包含自动化测试、本地部署、浏览器功能验收、权限验收、异常验收、视觉分析和质量审查。

## 自检记录

- 本规格没有纳入应付、发票、付款、税务、审批、MRP、采购退货、质检入库、正式财务凭证或报表分析。
- 设计范围与 `docs/handoffs/2026-07-04-project-handoff.md` 的下一阶段建议一致。
- 采购入库库存联动明确不允许绕过库存流水和来源唯一约束。
- 权限、接口、测试和视觉分析目录使用同一阶段名称和同一范围边界。
