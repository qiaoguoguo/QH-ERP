# 业务期间与历史数据保护基础设计规格

## 目标

本阶段为 QH ERP 建立轻量业务期间与历史数据保护能力。系统需要支持按业务日期维护期间、锁定期间、限制锁定期间内会影响经营口径的写入动作，并保证已完成的库存、采购、销售、生产、成本、财务往来、退货反冲和经营报表数据不会被后续跨期操作污染。

本阶段不是正式财务关账，不做会计期间、总账、凭证、税务、银行、期间快照、审批流、多组织、多公司、多账套或完整 BI。本阶段只保护业务期间口径和历史业务事实，为后续质量检验、计划需求、正式财务等能力建立更可信的数据边界。

## 背景

当前项目已经完成账号权限、基础资料、物料、BOM、库存、生产执行、成本归集、采购、销售、财务往来、经营报表、退货退款与业务反冲，以及页面体验治理。

015 阶段已经确认“已过账原单不可直接改写，修正必须通过反向业务记录表达”。但系统仍缺少业务期间层面的保护：用户仍可能用历史业务日期新增、确认或过账影响经营数据的单据，导致已经验收过的历史期间报表、库存、往来、成本和反向净额口径发生变化。

本阶段的核心产品判断是：先保护历史期间口径，再继续扩展质量状态、计划需求等更深业务能力。

## 设计原则

- 业务期间优先：按业务日期判断数据是否落入锁定期间。
- 历史不回写：锁定期间内的经营口径不可被新增写入动作改变。
- 查询不受限：锁定期间仍可查询、追溯和报表核对。
- 跨期修正：引用历史原单的修正必须发生在开放期间，并按新业务日期入账。
- 后端可信：前端禁用只做体验，后端统一期间校验才是最终边界。
- 权限独立：期间查看、维护、锁定和解锁使用独立权限点。
- 审计留痕：期间锁定、解锁和关键失败写入都应可审计。
- 范围克制：不把业务期间设计成正式财务结账或期间快照系统。

## 范围

### 本次包含

- 业务期间维护：
  - 期间列表。
  - 新增、编辑开放期间。
  - 按自然月生成期间。
  - 期间日期范围不得重叠。
  - 期间状态为开放、已锁定。
- 期间锁定：
  - 锁定开放期间。
  - 解锁已锁定期间。
  - 锁定和解锁必须填写原因。
  - 锁定和解锁写入审计日志。
- 历史写入保护：
  - 所有影响经营口径的写操作在后端统一校验业务日期。
  - 业务日期落入已锁定期间时拒绝写入。
  - 无匹配期间时按开放处理，但在页面和接口响应中可提示期间未配置风险。
- 跨模块覆盖：
  - 库存期初、库存调整和库存过账。
  - 采购订单确认、采购入库过账。
  - 销售订单确认、销售出库过账。
  - 生产工单发布、生产领料过账、生产报工过账、完工入库过账。
  - 成本手工记录和自动成本影响。
  - 应收应付确认、收付款过账。
  - 销售退货、采购退货、生产退料、生产补料、往来冲减过账。
  - 经营报表查询和来源追溯只读展示。
- 前端页面：
  - 期间列表。
  - 期间创建和编辑表单。
  - 锁定和解锁确认弹窗。
  - 业务单据页面的已锁期间提示和后端错误提示。
  - 报表和追溯页面的期间状态展示。
- 权限和验收：
  - 独立期间权限。
  - 后端接口鉴权。
  - 前端菜单、按钮和路由控制。
  - 自动化测试、浏览器验收和视觉分析。

### 本次不包含

- 正式财务关账、会计期间、总账、凭证、税务、银行对账。
- 期间快照、库存月结、成本结转、损益结转或正式报表封存。
- 审批流、消息通知、电子签核或多人复核。
- 多组织、多公司、多账套、多币种、多租户。
- 质量检验、库存质量状态、MRP、排产、产能计划。
- 自动生成采购订单、生产工单或正式财务单据。
- 历史数据批量修复工具。

## 业务规则

### 期间粒度

第一版以自然月为默认期间粒度，期间编码建议使用 `YYYY-MM`，例如 `2026-07`。系统允许管理员手动维护不重叠日期范围，但不允许两个期间的日期范围交叉。

### 期间状态

- 开放：允许符合业务规则的新增、编辑、确认、过账和反向业务。
- 已锁定：禁止会影响经营口径的写入动作；允许查询、追溯和报表查看。

### 锁定规则

锁定期间后，业务日期落入该期间的写入动作必须被拒绝。拒绝发生在后端服务层，不能只依赖前端按钮禁用。

需要拒绝的动作包括：

- 以锁定期间业务日期创建会影响经营口径的业务单据。
- 修改草稿单据的业务日期进入锁定期间。
- 确认、发布、过账、取消或冲减会影响锁定期间经营数据的记录。
- 创建业务日期落入锁定期间的销售退货、采购退货、生产退料、生产补料或往来冲减。

### 查询规则

已锁定期间的数据仍可查询。用户可以查看列表、详情、库存流水、业务来源、反向影响、经营报表和追溯抽屉。锁定状态不能导致历史数据不可浏览。

### 跨期修正规则

开放期间允许创建反向业务记录来引用锁定期间的历史原单，但反向业务自身的业务日期必须落入开放期间。反向业务不得改写历史原单，也不得把反向影响计入锁定期间。

示例：

```text
2026-06 销售出库已过账，2026-06 已锁定。
2026-07 发现需要退货。
系统允许在 2026-07 创建销售退货，来源引用 2026-06 销售出库。
销售退货入库、应收冲减和销售净额影响计入 2026-07，不回写 2026-06。
```

### 解锁规则

具备期间解锁权限的管理员可以解锁期间，必须填写原因。解锁不触发自动重算、不生成快照、不修改业务数据。解锁后新增业务写入仍必须符合原业务规则。

解锁是高风险操作，前端必须使用统一确认弹窗，后端必须记录审计日志。

## 后端设计

### 领域位置

新增业务期间能力建议放在 `com.qherp.api.system.period`。该模块不承载具体业务单据规则，只提供期间模型、期间权限、期间状态查询和统一保护服务。

### 数据模型

新增 Flyway 迁移，建议表：

- `biz_business_period`
- `biz_business_period_audit`

`biz_business_period` 建议字段：

| 字段 | 说明 |
|---|---|
| `id` | 主键 |
| `period_code` | 期间编码，例如 `2026-07` |
| `period_name` | 期间名称 |
| `start_date` | 开始日期 |
| `end_date` | 结束日期 |
| `status` | `OPEN` 或 `LOCKED` |
| `locked_by` | 锁定人 |
| `locked_at` | 锁定时间 |
| `lock_reason` | 锁定原因 |
| `unlocked_by` | 最近解锁人 |
| `unlocked_at` | 最近解锁时间 |
| `unlock_reason` | 最近解锁原因 |
| `created_at` | 创建时间 |
| `updated_at` | 更新时间 |

`biz_business_period_audit` 记录期间创建、编辑、锁定、解锁和失败写入拦截事件。失败写入审计可记录业务模块、动作、业务日期、用户和错误码，不记录敏感请求体。

### 统一校验服务

新增统一期间保护服务，建议命名为 `BusinessPeriodGuard`。核心方法：

```text
assertWritable(LocalDate businessDate, BusinessPeriodOperation operation, String sourceType, Long sourceId)
```

参数说明：

- `businessDate`：业务日期。
- `operation`：业务动作，例如创建、确认、过账、取消、反向过账、往来冲减。
- `sourceType`：业务来源类型，用于审计和错误提示。
- `sourceId`：来源记录主键，可为空。

当业务日期落入已锁定期间时，抛出稳定业务异常，例如 `BUSINESS_PERIOD_LOCKED`。异常信息应包含期间编码、期间名称和业务日期，不暴露内部 SQL 或堆栈。

### 接口设计

建议接口前缀：

```text
/api/system/business-periods
```

核心接口：

- `GET /api/system/business-periods`
- `POST /api/system/business-periods`
- `PUT /api/system/business-periods/{id}`
- `POST /api/system/business-periods/generate-monthly`
- `POST /api/system/business-periods/{id}/lock`
- `POST /api/system/business-periods/{id}/unlock`
- `GET /api/system/business-periods/resolve?businessDate=YYYY-MM-DD`

锁定和解锁请求体必须包含原因。编辑期间时，如果期间已锁定，不允许修改日期范围。编辑开放期间时，不允许形成重叠日期范围。

### 接入写路径

本阶段必须梳理并接入所有影响经营口径的服务层写路径。优先以过账、确认、发布和冲减动作作为强制校验点；对草稿保存和草稿编辑中的业务日期，也应防止保存到已锁期间。

需要接入的代表服务包括：

- 库存管理服务。
- 采购管理服务。
- 销售管理服务。
- 生产执行服务。
- 成本归集服务。
- 财务往来服务。
- 退货退款与业务反冲服务。

经营报表服务只读，不需要拒绝锁定期间查询，但需要能展示期间锁定状态或按业务日期筛选。

### 权限设计

新增权限点：

- `system:business-period:view`
- `system:business-period:create`
- `system:business-period:update`
- `system:business-period:lock`
- `system:business-period:unlock`

锁定和解锁不得复用普通系统管理权限。后端接口必须独立鉴权，前端菜单和按钮隐藏只是体验控制。

### 错误码

建议新增错误码：

- `BUSINESS_PERIOD_LOCKED`
- `BUSINESS_PERIOD_OVERLAPPED`
- `BUSINESS_PERIOD_DATE_RANGE_INVALID`
- `BUSINESS_PERIOD_REASON_REQUIRED`
- `BUSINESS_PERIOD_NOT_FOUND`
- `BUSINESS_PERIOD_STATUS_INVALID`

## 前端设计

### 页面入口

在系统管理或基础设置下新增“业务期间”菜单。页面遵守现有页面治理规范，不做营销式展示或低密度页面。

### 业务期间列表

列表展示期间编码、名称、开始日期、结束日期、状态、锁定人、锁定时间、锁定原因和操作。默认每页 10 条，支持 `10/20/50/100` 每页条数。筛选条件包括期间编码、日期范围和状态。

行操作：

- 开放期间：编辑、锁定。
- 已锁定期间：查看、解锁。

高风险操作锁定和解锁必须使用统一确认弹窗。

### 表单和交互

新增和编辑表单使用日期选择器，禁止手输日期文本框。日期清空后按空字符串处理。提交失败时错误提示应出现在表单可见区域。

按月生成期间提供起止月份输入，生成前展示将生成的期间数量和日期范围。已存在或重叠期间必须被后端拒绝，并给出明确提示。

### 业务单据提示

当业务单据的业务日期落入已锁期间时：

- 详情页展示只读原因。
- 编辑页应提示该业务日期所在期间已锁定。
- 过账、确认、取消、冲减等按钮按状态和权限禁用或隐藏。
- 如果后端拒绝写入，页面展示统一业务错误提示。

前端不能假设所有锁定状态都已提前加载；所有写入失败都必须正确处理后端错误。

### 报表和追溯

经营报表筛选业务日期时允许选择锁定期间。报表页面可展示“期间已锁定”的轻量状态说明，但不得阻断查询或来源追溯。

从报表或追溯进入历史业务详情时，必须保留返回上下文。

### 推荐前端文件

- `apps/web/src/shared/api/businessPeriodApi.ts`
- `apps/web/src/shared/api/businessPeriodApi.spec.ts`
- `apps/web/src/modules/system/businessPeriods/BusinessPeriodListView.vue`
- `apps/web/src/modules/system/businessPeriods/BusinessPeriodListView.spec.ts`
- `apps/web/src/modules/system/businessPeriods/BusinessPeriodFormDialog.vue`
- `apps/web/src/modules/system/businessPeriods/BusinessPeriodStatusTag.vue`

如现有系统管理模块已有页面工具函数，应复用既有模式，不另起一套页面结构。

## 数据流

期间锁定：

```text
管理员 -> 业务期间列表 -> 锁定确认 -> 后端鉴权 -> 更新期间状态 -> 写审计日志 -> 前端刷新列表
```

业务写入保护：

```text
用户提交业务动作 -> 业务服务取得业务日期 -> BusinessPeriodGuard 校验 -> 开放期间继续原业务逻辑 -> 已锁期间返回 BUSINESS_PERIOD_LOCKED
```

跨期反向业务：

```text
历史原单 -> 开放期间创建反向单 -> 反向单业务日期校验通过 -> 过账写库存/往来/成本/报表影响 -> 追溯保留历史来源
```

报表查询：

```text
用户选择日期范围 -> 报表服务按业务日期查询 -> 返回汇总和来源 -> 页面展示期间状态与来源追溯
```

## 验收标准

- 期间日期范围不可重叠。
- 锁定和解锁必须有权限、原因和审计日志。
- 已锁期间禁止所有影响经营口径的写入动作。
- 锁定期间仍可查询、追溯和报表查看。
- 开放期间可创建反向业务引用锁定期间历史原单，但反向影响进入开放期间。
- 后端接口校验不可被前端绕过。
- 页面符合 `docs/ui/page-standards.md`。
- 自动化测试、类型检查、构建、浏览器验收和视觉分析通过后才可进入阶段交付。

## 风险与处理

- 风险：业务期间被误解为正式财务关账。处理：页面、接口和文档统一使用“业务期间”“历史数据保护”，避免“结账”“凭证”“总账”等正式财务语义。
- 风险：遗漏写接口导致历史期间仍可污染。处理：以服务层统一 `BusinessPeriodGuard` 为准，测试按模块逐一覆盖。
- 风险：跨期反向业务口径不清。处理：明确反向业务按自身业务日期进入开放期间，不回写历史期间。
- 风险：期间解锁被滥用。处理：独立权限、必填原因、审计日志和高风险确认弹窗。
- 风险：前端只禁用按钮但后端未校验。处理：后端校验作为验收门槛，前端只做体验提示。

## 自检结论

- 本设计无未完成内容。
- 范围聚焦业务期间锁定和历史写入保护。
- 设计不包含正式财务关账、期间快照或审批流。
- 后端、前端、权限、测试和浏览器验收边界清晰。
- 本阶段可以继续拆分为实施计划和最多 4 个工作包。
