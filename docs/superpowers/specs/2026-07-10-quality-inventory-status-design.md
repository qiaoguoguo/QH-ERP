# 质量检验与库存质量状态基础设计规格

## 目标

017 阶段为 QH ERP 建立库存质量状态基础能力。系统需要把库存从“仓库 + 物料”的单一余额口径升级为“仓库 + 物料 + 质量状态”的分层余额口径，并让 `availableQuantity` 明确等于合格可用库存数量。

本阶段的核心交付不是完整质量管理体系，而是让质量状态成为库存可用性的硬约束：待检、不合格、冻结库存不得被销售出库、生产领料、生产补料或后续计划净算当作可用库存。采购入库、生产完工入库、销售退货和生产退料等入库来源需要进入受控质量状态，并通过轻量质量确认完成状态流转。

## 背景

当前项目已经完成基础资料、物料、BOM、库存基础、生产执行、成本归集、采购、销售、财务往来、经营报表、退货退款与业务反冲、业务期间与历史数据保护，以及页面体验治理。

现有库存基础能力已经支持库存余额、库存流水、期初库存、库存调整，以及采购、销售、生产和反向业务的库存过账。当前库存余额表以 `warehouse_id + material_id` 为唯一维度，`availableQuantity` 一期等于现存量扣减锁定量。这个口径无法区分待检、合格、不合格和冻结库存，导致后续销售出库、生产领料、MRP、成本和报表可能把不可用库存误认为可用库存。

016 阶段已经建立业务期间与历史数据保护。017 必须继续遵守该保护：所有质量状态生成、检验确认、冻结、解冻和相关库存写入都必须按业务日期受 `BusinessPeriodGuard` 约束。

## 设计原则

- 质量状态进入库存底座：库存余额和库存流水必须持久化质量状态，不能只用附属台账推导。
- 合格才可用：销售出库、生产领料、生产补料和后续计划净算只认合格库存。
- 入库先受控：采购入库、生产完工入库、销售退货和生产退料默认进入待检库存。
- 轻量确认：本阶段只做质量确认和状态转换，不做完整 QMS。
- 后端可信：可用量、不可用原因、状态转换和库存扣减以后端计算为准，前端不得自行推导最终可用量。
- 事务一致：检验确认必须在同一事务内完成待检扣减、目标状态增加、流水、单据状态和审计。
- 历史平滑：历史库存默认迁移为合格库存，不破坏既有演示数据和业务验收口径。
- 来源可追溯：质量状态库存必须能追溯来源单据、状态转换记录和后续消耗结果。
- 范围克制：不扩展批次、库位、序列号、MRP、审批流、正式财务或复杂质量体系。

## 范围

### 本次包含

- 库存质量状态：
  - `PENDING_INSPECTION`：待检。
  - `QUALIFIED`：合格。
  - `REJECTED`：不合格。
  - `FROZEN`：冻结。
- 库存余额口径：
  - `inv_stock_balance` 按仓库、物料、质量状态分层。
  - 历史余额迁移为合格库存。
  - 库存余额接口展示总库存、合格可用库存和各质量状态数量。
- 库存流水口径：
  - `inv_stock_movement` 持久化质量状态。
  - 入库、出库和质量状态转换都形成可追溯流水。
- 入库来源接入：
  - 采购入库。
  - 生产完工入库。
  - 销售退货。
  - 生产退料。
  - 上述来源默认进入待检库存。
- 消耗路径接入：
  - 销售出库只消耗合格库存。
  - 生产领料只消耗合格库存。
  - 生产补料只消耗合格库存。
  - 采购退货允许从待检、不合格或合格库存退货；冻结库存需先解冻。
- 轻量质量确认：
  - 待检转合格。
  - 待检转不合格。
  - 待检转冻结。
  - 合格转冻结。
  - 冻结解冻回合格。
  - 转换必须填写原因并留存审计。
- 权限与期间保护：
  - 查看质量记录。
  - 提交质量确认。
  - 冻结和解冻。
  - 后端接口独立鉴权。
  - 所有影响库存状态和数量的动作接入 016 业务期间保护。
- 前端页面与体验：
  - 质量检验列表。
  - 质量检验详情或处理抽屉。
  - 库存余额质量状态分布。
  - 消耗路径候选库存禁用原因。
  - 权限态、异常态、空态和窄屏视觉审计。

### 本次不包含

- 完整 QMS、检验标准库、抽样方案、缺陷等级、供应商质量评分。
- SPC、质量趋势分析、质量成本归集或质量经营报表。
- 批次、序列号、库位、条码、拣配。
- MRP、计划净算实现、排产、产能计划。
- 审批流、工作流设计器、消息通知。
- 正式财务、凭证、总账、税务、银行对账、成本重算。
- 复杂返工返修、不合格评审、报废处置闭环。
- 多组织、多公司、多账套、多币种、多租户。

## 业务规则

### 质量状态口径

合格库存是唯一普通可用库存。待检库存表示尚未质量确认，不得销售出库或生产领用。不合格库存表示已确认不可用，不得参与普通出库。冻结库存表示被人工或规则冻结，不得参与普通出库、采购退货或计划可用量；如需退货或消耗，必须先解冻到合格或重新进行受控状态转换。

### 历史库存

迁移前已有库存余额和历史库存流水没有质量维度。迁移时全部历史库存余额和历史流水补为 `QUALIFIED`。这样可以保持当前已验收生产、销售、采购、库存和报表路径不被破坏。

迁移后，同一仓库和物料下各质量状态数量合计必须等于迁移前现存数量。

### 入库默认状态

以下入库来源默认进入待检库存：

- 采购入库。
- 生产完工入库。
- 销售退货。
- 生产退料。

017 不提供物料免检配置。免检、按供应商或物料规则自动合格属于后续质量管理深化范围。

### 轻量质量确认

系统自动或手工生成质量确认记录。质量确认记录必须包含来源类型、来源单据、来源行、仓库、物料、单位、待检数量、业务日期、状态、处理人、处理时间、原因和结果数量。

待检数量可以拆分为合格、不合格和冻结数量。提交时三类结果数量合计必须等于本次处理的待检数量，且不得超过当前待检余额。

提交质量确认后：

```text
待检库存减少 -> 合格、不合格或冻结库存增加 -> 写入状态转换流水 -> 更新确认记录 -> 写审计日志
```

任一环节失败必须整体回滚。

### 冻结和解冻

冻结用于临时禁止库存被普通业务消耗。017 允许合格转冻结，也允许冻结解冻回合格。冻结和解冻必须有独立权限、原因、业务日期和审计记录。

不合格转合格、冻结转不合格、合格转不合格等更复杂状态处置不在 017 范围内。

### 消耗控制

普通消耗路径只能扣减合格库存：

- 销售出库。
- 生产领料。
- 生产补料。

当总库存充足但合格库存不足时，必须返回质量状态不可用或指定质量状态库存不足的稳定错误，不能用普通库存不足掩盖业务原因。

采购退货是供应商退回场景，允许显式选择待检、不合格或合格库存退回供应商。冻结库存不得直接退货，需先解冻。

### 可用量

库存余额接口的 `availableQuantity` 调整为合格可用库存数量。页面必须同时展示总现存和质量状态分布，避免用户把总库存误认为可发、可领或可退库存。

### 期间保护

质量确认、冻结、解冻、入库质量状态落账、出库按质量状态扣减等动作都属于影响经营口径和库存事实的写入。业务日期落入已锁定期间时必须被后端拒绝。锁定期间历史质量记录、库存余额、库存流水和来源追溯仍可查询。

## 后端设计

### 领域位置

建议新增 `com.qherp.api.system.quality` 领域承载质量确认单、状态流转和质量权限。库存余额、库存流水和可用量口径仍由 `com.qherp.api.system.inventory` 作为唯一库存底座负责。

质量领域可以调用库存状态转换服务，但不得绕过库存服务直接改余额。

### 数据模型

新增 Flyway 迁移，建议包含：

- `inv_stock_balance` 增加 `quality_status`，唯一键调整为 `warehouse_id, material_id, quality_status`。
- `inv_stock_movement` 增加 `quality_status`。
- 新增 `qua_quality_inspection`。
- 新增 `qua_quality_inspection_line` 或结果表。
- 可选新增 `inv_quality_status_transfer`，用于记录非来源入库的质量状态转换单；如果质量确认表已经完整记录转换，可不单独建转换单表。

`qua_quality_inspection` 建议字段：

| 字段 | 说明 |
|---|---|
| `id` | 主键 |
| `inspection_no` | 质量确认单号 |
| `source_type` | 来源类型 |
| `source_id` | 来源单据 |
| `source_line_id` | 来源行 |
| `warehouse_id` | 仓库 |
| `material_id` | 物料 |
| `unit_id` | 单位 |
| `business_date` | 业务日期 |
| `inspection_quantity` | 待处理数量 |
| `qualified_quantity` | 合格数量 |
| `rejected_quantity` | 不合格数量 |
| `frozen_quantity` | 冻结数量 |
| `status` | `PENDING`、`COMPLETED` |
| `reason` | 处理原因 |
| `remark` | 备注 |
| `created_by`、`created_at` | 创建信息 |
| `completed_by`、`completed_at` | 完成信息 |
| `version` | 乐观版本 |

### 库存过账服务

`InventoryPostingService.PostingRequest` 增加质量状态字段。所有入库和出库必须传入质量状态。过账服务按仓库、物料和质量状态锁定余额。

对普通出库来源，库存服务集中校验质量状态必须为 `QUALIFIED`。需要特别处理的采购退货由调用方显式指定质量状态并使用允许的业务来源规则。

质量状态转换应由库存服务提供同事务转换方法，语义类似：

```text
transferQualityStatus(
  warehouseId,
  materialId,
  unitId,
  fromStatus,
  toStatus,
  quantity,
  sourceType,
  sourceId,
  sourceLineId,
  businessDate,
  reason,
  operatorName
)
```

该方法必须先锁定来源状态余额，校验来源数量足够，再写出库方向的转换流水和入库方向的转换流水，最后更新两个质量状态余额。

### 业务入口接入

入库侧：

- 采购入库过账：写入待检库存，并生成质量确认记录。
- 生产完工入库过账：写入待检库存，并生成质量确认记录。
- 销售退货过账：写入待检库存，并生成质量确认记录。
- 生产退料过账：写入待检库存，并生成质量确认记录。

消耗侧：

- 销售出库过账：只扣减合格库存。
- 生产领料过账：只扣减合格库存。
- 生产补料过账：只扣减合格库存。
- 采购退货过账：显式选择待检、不合格或合格库存；冻结库存拒绝。

### 权限设计

新增权限点：

| 权限编码 | 类型 | 说明 |
|---|---|---|
| `quality` | 菜单 | 质量管理 |
| `quality:inspection:view` | 操作 | 查看质量确认 |
| `quality:inspection:process` | 操作 | 提交质量确认 |
| `quality:status:freeze` | 操作 | 冻结合格库存 |
| `quality:status:unfreeze` | 操作 | 解冻冻结库存 |

前端菜单和按钮按权限控制，后端接口必须独立鉴权。

### 错误码

建议新增稳定错误码：

| 错误码 | 场景 |
|---|---|
| `INVENTORY_QUALITY_STATUS_REQUIRED` | 库存过账缺少质量状态 |
| `INVENTORY_NON_QUALIFIED_NOT_AVAILABLE` | 非合格库存不能用于普通消耗 |
| `INVENTORY_QUALITY_STATUS_BALANCE_NOT_ENOUGH` | 指定质量状态库存不足 |
| `QUALITY_INSPECTION_NOT_FOUND` | 质量确认记录不存在 |
| `QUALITY_INSPECTION_SOURCE_INVALID` | 来源不支持、状态不合法或已处理 |
| `QUALITY_INSPECTION_QUANTITY_MISMATCH` | 结果数量合计不等于处理数量 |
| `QUALITY_INSPECTION_STATUS_INVALID` | 当前质量确认状态不允许操作 |
| `QUALITY_STATUS_TRANSITION_INVALID` | 状态转换路径不允许 |

`BUSINESS_PERIOD_LOCKED` 沿用 016 语义，不新增绕过口径。

## 接口设计

建议接口前缀：

```text
/api/admin/quality/inspections
/api/admin/inventory/quality-transfers
```

核心接口：

- `GET /api/admin/quality/inspections`
- `GET /api/admin/quality/inspections/{id}`
- `POST /api/admin/quality/inspections/{id}/process`
- `POST /api/admin/inventory/quality-transfers/freeze`
- `POST /api/admin/inventory/quality-transfers/unfreeze`

库存余额和库存流水接口扩展：

- `GET /api/admin/inventory/balances` 支持 `qualityStatus` 筛选，响应包含各状态数量和合格可用量。
- `GET /api/admin/inventory/movements` 支持 `qualityStatus` 筛选，响应包含流水质量状态。

采购退货相关候选来源需要返回质量状态、指定质量状态可退数量和禁用原因。销售出库、生产领料、生产补料候选来源默认只返回合格可用库存；如展示非合格库存，必须返回不可用原因。

### 页面入口

新增“质量管理 > 质量确认”菜单。页面遵守 `docs/ui/page-standards.md`，采用高密度列表、筛选区、表格、详情页或抽屉，不做质量大屏。

### 质量确认列表

列表展示：

- 质量确认单号。
- 来源类型和来源单号。
- 仓库、物料、规格、单位。

处理页面或抽屉顶部固定展示来源上下文：物料、来源单号、仓库、待检数量和业务日期。输入区包含合格数量、不合格数量、冻结数量、原因和备注。

### 库存余额页

库存余额页展示：

- 总现存数量。
- 合格可用数量。
- 待检数量。
- 不合格数量。
- 冻结数量。

状态标签统一使用小尺寸胶囊标签。待检为琥珀色，合格为绿色，不合格为红色，冻结为灰蓝或深灰。不得只依靠颜色表达状态，必须显示中文文案。

### 候选库存禁用原因

销售出库、生产领料、采购退货和生产补料的候选库存区域需要展示质量状态和禁用原因。

禁用原因示例：

- `待检库存不可领料`
- `不合格库存不可销售出库`
- `冻结库存不可参与可用量`
- `冻结库存需先解冻后退货`

空态必须区分“无库存”和“有库存但无合格可用库存”。

## 数据流

采购入库到质量确认：

```text
采购入库过账 -> 待检库存增加 -> 生成质量确认记录 -> 质量确认提交 -> 待检减少 -> 合格/不合格/冻结增加
```

完工入库到可销售库存：

```text
生产报工合格 -> 完工入库过账 -> 待检库存增加 -> 质量确认合格 -> 合格库存增加 -> 销售出库可消耗
```

销售退货复检：

```text
销售出库 -> 销售退货过账 -> 待检库存增加 -> 质量确认 -> 合格库存可再次销售或不合格库存隔离
```

普通消耗控制：

```text
用户提交出库或领料 -> 后端锁定合格库存余额 -> 合格库存足够则扣减并写流水 -> 合格库存不足则拒绝
```

期间保护：

```text
用户提交质量确认或冻结 -> BusinessPeriodGuard 校验业务日期 -> 开放期间继续 -> 已锁期间返回 BUSINESS_PERIOD_LOCKED
```

## 测试设计

### 后端测试

后端测试必须覆盖：

- 迁移后历史库存全部为合格，余额汇总与迁移前一致。
- 库存余额按质量状态分层查询。
- 采购入库、完工入库、销售退货、生产退料进入待检库存。
- 待检质量确认可拆分为合格、不合格和冻结。
- 质量确认数量不平被拒绝。
- 质量确认重复提交被拒绝或幂等保护，不重复转移库存。
- 销售出库、生产领料、生产补料不能消耗待检、不合格或冻结库存。
- 采购退货可以扣减待检、不合格或合格库存，不能扣减冻结库存。
- 合格转冻结、冻结解冻回合格。
- 质量确认和冻结受 016 业务期间保护。
- 无权限用户不能处理质量确认、冻结或解冻。
- 并发扣减和并发质量确认不会导致负库存或重复流水。

## 验收标准

- 历史库存默认迁移为合格库存。
- 库存余额按仓库、物料和质量状态分层，汇总数量一致。
- `availableQuantity` 只代表合格可用库存。
- 采购入库、完工入库、销售退货和生产退料默认形成待检库存。
- 质量确认可以把待检库存转为合格、不合格或冻结，且转换前后总库存不变。
- 销售出库、生产领料和生产补料不能消耗非合格库存。
- 采购退货按显式质量状态退货，冻结库存需先解冻。
- 质量状态写入受权限、原因、审计和业务期间保护约束。
- 后端接口校验不可被前端绕过。
- 页面符合 `docs/ui/page-standards.md`。
- 自动化测试、类型检查、构建、浏览器验收和视觉分析通过后才可进入阶段交付。

## 风险与处理

- 风险：余额维度调整影响现有库存、采购、销售、生产和报表查询。处理：先做迁移和库存服务集中改造，再接入业务入口，测试覆盖余额汇总和来源追溯。
- 风险：前端把总库存误显示为可用库存。处理：接口明确返回合格可用量和状态分布，前端不得自行计算最终可用量。
- 风险：状态转换出现部分写入。处理：质量确认和状态转换必须单事务完成，并用来源唯一约束或状态校验防重复提交。
- 风险：入库默认待检改变用户对库存即时可用的预期。处理：页面明确展示待检状态和可用量口径，历史库存仍默认合格。
- 风险：采购退货规则与普通出库规则混淆。处理：采购退货显式选择质量状态，冻结库存先解冻，不走普通合格可用量默认规则。
- 风险：017 被扩大成完整质量系统。处理：明确排除检验标准、抽样、SPC、供应商评分、返工返修和质量成本。

## 工作包建议

1. 库存质量状态底座：数据库迁移、状态枚举、余额唯一键调整、流水质量状态、历史库存默认合格、库存查询与可用量口径。
2. 质量确认与状态转换：质量确认模型、待检转合格/不合格/冻结、冻结/解冻、权限、原因、审计、期间保护和事务一致性。
3. 业务入口接入：采购入库、完工入库、销售退货、生产退料进入待检；销售出库、生产领料、生产补料和采购退货按质量状态消耗。
4. 前端页面与验收：质量确认页面、库存余额质量分布、候选禁用原因、权限态、异常态、视觉审计和定向回归。
