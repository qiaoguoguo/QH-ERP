# 批次、序列号与来源去向追溯基础设计规格

## 目标

019 阶段为 QH ERP 建立批次、序列号与来源去向追溯基础。系统需要在现有采购、质量、库存、销售、生产、退货反冲和冻结解冻链路中持续记录物料追踪身份，并让库存余额、库存流水、可用量、占用预留和追溯查询共享同一套批次/序列口径。

本阶段不是完整 WMS、MES、法规召回平台、扫码设备集成或批次成本核算。交付重点是让核心 ERP 单据链路可以回答：某批或某件物料从哪里来、现在在哪里、处于什么质量和可用状态、用到了哪些生产工单或销售出库，以及反向业务如何影响它。

## 现状

017 已经建立待检、合格、不合格、冻结等质量状态；018 已经建立现货净可用、预留、占用、采购在途参考和仓库级现货可承诺量。当前缺口是所有库存事实仍只到仓库、物料、质量状态维度，没有批次或序列号身份。

如果 019 只在单据行上追加批次号文本，而不改造库存余额、库存流水、预留占用和追溯接口，系统无法保证同一批次的数量守恒，也无法证明销售、生产、退货和冻结使用的是同一批或同一件。因此本阶段必须把追踪维度纳入库存底账和统一过账服务，而不是做页面字段补丁。

## 方案比较

### 推荐方案：核心单据追踪基础

物料增加追踪方式；库存余额、流水、预留占用和通用分配明细增加批次/序列维度；入库生成或继承批次/序列，出库和领料选择可用批次/序列；销售订单和生产工单仍按物料级预留，执行时精确到批次/序列；追溯接口返回来源、去向和当前状态。

优点是能建立可信追溯底账，同时避免提前锁批次带来的复杂 ATP 和计划分配问题。缺点是需要改造多个核心过账点，但改造范围仍可被一个阶段控制。

### 备选方案：全量精确预留追踪

销售订单确认和生产工单发布即精确到批次或序列号。优点是承诺关系最精确；缺点是会强迫业务在计划阶段提前指定实物批次，容易与采购在途、生产变更和仓库执行冲突，且会显著放大预留释放、改批、替代料和缺料处理复杂度。本阶段不采用。

### 备选方案：轻量字段追踪

只在入库、出库、领料和退货行记录批次号或序列号文本，不改库存底账。优点是实现快；缺点是无法保证数量守恒、不可用库存拦截和来源去向可信度，后续需要返工核心模型。本阶段不采用。

## 核心决策

- 追踪方式为 `NONE`、`BATCH`、`SERIAL`，默认 `NONE`。
- 旧库存和旧流水不自动补真实批次或序列号；已有正库存、流水或活动预留的物料禁止直接切换为批次或序列号追踪。
- 批号唯一范围为同一物料内唯一；序列号唯一范围为同一物料内唯一。
- 序列号代表单件，数量固定为 1，不支持小数、部分冻结、部分领用或部分退货。
- 批次物料一条业务行可以拆分到多个批次；批次数量合计必须等于业务行数量。
- 销售订单和生产工单预留保持物料、仓库、质量状态维度；执行出库、领料、补料和采购退货时精确选择批次/序列并形成精确占用或消费。
- 销售退货必须继承原销售出库批次/序列；生产退料必须继承原生产领料批次/序列。
- 质量确认、冻结、解冻只能改变同一追踪身份的质量或可用状态，不得重建或替换批次/序列。
- 追溯接口返回链路摘要并按来源模块权限脱敏，不能绕过采购、销售、生产、质量详情权限。

## 数据模型

### 物料追踪方式

`mst_material` 增加 `tracking_method`：

- `NONE`：不追踪，沿用现有库存口径。
- `BATCH`：批次管理，库存以批次数量为追踪身份。
- `SERIAL`：序列号管理，库存以单件序列号为追踪身份。

物料列表、详情、创建和更新接口都返回该字段。存量物料迁移为 `NONE`。如果物料已有库存余额、库存流水或活动预留，占用或预留未清空前不允许改为 `BATCH` 或 `SERIAL`。

### 批次主档

新增 `inv_batch`：

- `id`、`material_id`、`batch_no`。
- `source_type`、`source_id`、`source_line_id`。
- `business_date`、`remark`。
- `created_by`、`created_at`、`updated_by`、`updated_at`、`version`。

约束：

- `material_id + batch_no` 唯一。
- 批次只能归属一个物料。
- 批次来源可以是采购入库、生产完工、销售退货、生产退料或库存初始化。

### 序列号主档

新增 `inv_serial`：

- `id`、`material_id`、`serial_no`、`batch_id` 可空。
- `source_type`、`source_id`、`source_line_id`。
- 当前状态缓存：仓库、质量状态、库存状态、最后流水。
- `created_by`、`created_at`、`updated_by`、`updated_at`、`version`。

约束：

- `material_id + serial_no` 唯一。
- 序列号数量固定为 1。
- 同一序列号同一时刻只能处于一个当前库存状态或已出库状态。

### 库存余额

`inv_stock_balance` 增加 `batch_id`、`serial_id`。

余额唯一口径：

- 非追踪：仓库 + 物料 + 质量状态，批次和序列为空。
- 批次：仓库 + 物料 + 质量状态 + 批次，序列为空。
- 序列：仓库 + 物料 + 质量状态 + 序列，批次可空。

需要通过部分唯一索引处理空值维度，避免非追踪、批次和序列余额互相冲突。`locked_quantity` 继续作为 018 占用预留缓存，但批次或序列精确占用必须能扣减对应追踪维度的可用量。

### 库存流水

`inv_stock_movement` 增加 `batch_id`、`serial_id`。现有 `source_type + source_line_id` 唯一约束需要改造，因为一条业务行可能拆分多个批次或多个序列，后续唯一性应包含追踪维度或使用独立分配明细约束。

流水必须记录：

- 来源单据和来源行。
- 仓库、物料、单位、质量状态。
- 批次或序列号。
- 方向、数量、前后结存、业务日期、原因和操作人。

### 库存追踪分配明细

新增 `inv_stock_tracking_allocation`，作为业务单据行与批次/序列选择之间的通用表：

- `document_type`、`document_id`、`document_line_id`。
- `allocation_type`：入库、出库、质量转移、冻结、解冻、退货继承等。
- `warehouse_id`、`material_id`、`unit_id`、`quality_status`。
- `batch_id`、`serial_id`、`quantity`。
- `source_type`、`source_id`、`source_line_id`。
- `movement_id`、`reservation_id` 可空。
- 审计字段和版本。

该表用于草稿保存、过账校验、详情回显和追溯聚合，避免每个业务模块重复设计追踪行结构。

### 占用预留

`inv_stock_reservation` 增加 `batch_id`、`serial_id` 和可选 `parent_reservation_id`。物料级预留不填批次/序列；执行单据精确占用或消费时可以关联父预留，并写入精确追踪维度。

## 业务流转

### 入库类动作

采购入库、生产完工入库、销售退货和生产退料属于入库类：

- 不追踪物料不要求分配明细。
- 批次物料必须录入或生成批次；一行可拆多批，分配数量合计等于业务行数量。
- 序列号物料必须录入序列号；序列数量等于业务行数量，且业务数量必须为整数。
- 入库后按当前质量规则进入待检、合格或指定质量状态；采购入库和完工入库默认继续承接 017 的待检链路。
- 退货和退料优先继承原出库或领料追踪身份，不允许任意改写来源身份。

### 出库类动作

销售出库、生产领料、采购退货、生产补料属于出库类：

- 不追踪物料沿用 018 可用量校验。
- 批次物料必须从合格、未冻结、当前可用的批次候选中选择。
- 序列号物料必须从合格、未冻结、当前可用的序列候选中选择。
- 执行时按仓库、物料、质量状态、批次/序列锁定余额行；序列号还要锁定 `inv_serial`。
- 过账后扣减对应追踪余额并写库存流水。

### 质量确认

质量确认处理待检库存时必须保留追踪身份：

- 批次物料允许同一批次部分合格、部分不合格或冻结，余额按质量状态拆分。
- 序列号物料每个序列只能处于一个质量状态；多个序列可以分别确认不同结果。
- 质量状态转移生成库存流水和追踪分配明细，追溯查询能看到状态变化。

### 冻结与解冻

冻结追踪物料必须选择批次或序列：

- 只能冻结合格净可用库存。
- 不允许冻结已被物料级或追踪级占用预留覆盖的不可用数量。
- 解冻后回到同一批次或序列的合格状态。
- 冻结和解冻记录原因、业务日期、操作人和审计。

### 预留和占用

销售订单和生产工单确认或发布时继续生成物料级预留。销售出库、生产领料、生产补料、采购退货保存或过账时选择具体批次或序列，并形成精确占用或直接消费。执行完成后父级预留按数量消费或释放。

这样可以保留 018 的承诺能力，同时避免计划阶段强制指定实物批次。

## 接口设计

### 物料接口

物料列表、详情、创建和更新增加：

- `trackingMethod`
- `trackingMethodName`

查询支持 `trackingMethod` 筛选。已有业务库存或活动预留时，更新追踪方式返回受控错误。

### 批次和序列接口

- `GET /api/admin/inventory/batches`
- `GET /api/admin/inventory/batches/{id}`
- `GET /api/admin/inventory/serials`
- `GET /api/admin/inventory/serials/{id}`

查询条件：

- `keyword`
- `warehouseId`
- `materialId`
- `qualityStatus`
- `onlyAvailable`
- `batchNo`
- `serialNo`
- `sourceType`
- `sourceId`
- `page`
- `pageSize`

候选接口可以复用列表接口并增加 `onlyAvailable=true`，也可以在具体业务模块中返回当前来源可选量。后端是候选上限的唯一来源。

### 库存余额和流水

`GET /api/admin/inventory/balances` 扩展：

- 查询条件增加 `trackingMethod`、`batchNo`、`serialNo`。
- 响应增加 `trackingMethod`、`batchId`、`batchNo`、`serialId`、`serialNo`、`traceableQuantity`。

`GET /api/admin/inventory/movements` 扩展：

- 查询条件增加 `batchId`、`batchNo`、`serialId`、`serialNo`。
- 响应返回追踪字段和来源跳转摘要。

### 追溯接口

- `GET /api/admin/inventory/traces/batches/{id}`
- `GET /api/admin/inventory/traces/serials/{id}`

响应包含：

- 主档信息。
- 当前结存按仓库和质量状态汇总。
- 活动预留和占用摘要。
- 来源入库或初始化记录。
- 质量状态变化记录。
- 冻结和解冻记录。
- 销售出库、生产领料、采购退货、生产补料、销售退货、生产退料等去向。
- 关联库存流水。
- 权限受限来源摘要。

### 业务单据接口

采购入库、生产完工、销售出库、生产领料、销售退货、采购退货、生产退料、生产补料的行载荷统一包含：

- `trackingAllocations`
- 每条分配包含 `batchId`、`batchNo`、`serialId`、`serialNo`、`quantity`、`qualityStatus`、`sourceAllocationId` 可空。

详情响应必须回显追踪分配；只读单据显示追踪身份但不可编辑。

## 权限、错误码和审计

新增权限建议：

- `inventory:trace:view`
- `inventory:batch:view`
- `inventory:serial:view`

批次和序列采集沿用对应业务单据创建、更新、过账权限，不开放独立手工出入库入口。

新增错误码建议：

- `INVENTORY_TRACKING_METHOD_IMMUTABLE`
- `INVENTORY_BATCH_REQUIRED`
- `INVENTORY_SERIAL_REQUIRED`
- `INVENTORY_SERIAL_DUPLICATED`
- `INVENTORY_TRACKING_QUANTITY_MISMATCH`
- `INVENTORY_TRACKING_STOCK_NOT_ENOUGH`
- `INVENTORY_TRACKING_NOT_AVAILABLE`
- `INVENTORY_TRACKING_SOURCE_MISMATCH`

审计动作覆盖批次创建、序列创建、追踪分配保存、追踪过账、冻结、解冻和追溯查看。

## 前端设计

物料档案在库存控制区域展示追踪方式。已有库存或业务记录时追踪方式禁用，并显示原因。

入库类单据的明细行只展示追踪状态、已分配数量和操作入口；点击进入批次/序列分配编辑器。批次用表格拆分数量，序列号支持逐条录入、批量粘贴和去重提示。

出库类单据使用批次/序列选择抽屉：左侧候选库存池，右侧已选清单。批次按数量拆分，序列按勾选条目，已选数量反推业务数量或校验业务数量。

库存余额默认仍以仓库和物料聚合视图为主，追踪字段以摘要和展开明细呈现，避免宽表失控。库存流水增加批次/序列筛选，从余额跳转流水时携带仓库、物料、质量状态和追踪参数。

追溯抽屉复用 018 占用预留抽屉和退货反冲追溯面板模式，分为基础信息、当前库存、来源去向、库存流水和受限来源提示。

移动端优先保障查询、查看和追溯，不要求复杂批量录入达到桌面效率。

## 测试和验收

后端测试覆盖迁移兼容、物料追踪方式、入库生成、出库拆批、序列数量校验、质量确认保留追踪维度、冻结占用拦截、订单预留转执行占用、退货沿用原追踪、重复过账、并发抢同一序列和追溯权限脱敏。

前端测试覆盖 API 类型、追踪方式字段、批次拆分合计、序列号去重和数量一致、不可用候选禁用、详情回显、追溯抽屉加载/错误/空态/权限受限状态。

浏览器验收覆盖物料追踪方式、采购入库批次/序列录入、完工入库、销售出库选择、生产领料选择、退货继承来源、质量确认、冻结解冻、库存余额、库存流水、追溯抽屉、无权限、只读、期间锁定和移动端。

## 交付边界

019 完成后，系统可以在核心 ERP 单据链路中追踪批次和序列号来源去向，并确保追踪身份参与库存余额、库存流水、可用量和质量状态计算。系统仍不提供完整仓储执行、扫码、标签、工序级追溯、法规召回、批次成本或 MRP。
