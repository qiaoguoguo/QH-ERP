# 销售项目与合同管理基础设计规格

## 设计目标

020 阶段建立 QH ERP 的销售项目与合同管理基础。设计目标不是把后续项目成本、库存计价和正式财务提前实现，而是先沉淀一个稳定、可权限控制、可审计、可被后续阶段引用的项目主维度。

完成后，系统应能表达一个销售项目、一个主合同、多个补充合同和多个关联销售订单之间的基础关系。后续采购、库存、生产、费用、发票、收付款、项目成本和项目利润分析都以该项目标识作为统一入口。

020 已于 2026-07-12 完成实现、全量验证和浏览器视觉验收；本规格保留推荐模型和设计边界，阶段范围与验收继续以 `docs/tasks/020-sales-project-contract-foundation.md` 为权威来源。

## 方案比较

### 推荐方案：项目一级对象，合同为项目子对象

销售项目作为一级对象，合同作为项目下的商业文件。一个项目最多一个非取消主合同，可以有多个补充合同；补充合同引用主合同，且仅允许在项目为 `ACTIVE`、主合同为 `EFFECTIVE` 时创建，生效时再次校验。销售订单在草稿阶段可选关联项目和合同，形成后续履约与收入归属骨架。

优点是完全贴合“销售项目是一级成本、收入和利润对象”的产品决策。项目不依附于某一份合同，后续项目专采、公共库存领料、工单、费用、发票和成本都能围绕同一 `projectId` 归集。合同负责商业金额和变更归属，项目负责管理对象和利润对象。

代价是 020 必须同时管理项目状态、合同状态和销售订单关联校验，但复杂度仍可控制在销售域内。

### 备选方案：合同一级对象，项目由合同组派生

合同作为一级对象，项目只是合同组或合同集合的展示结果。该方案适合合同驱动的销售系统，但不适合当前项目型制造 ERP。项目专采、生产工单和项目成本需要一个不随合同变更而变化的主对象，若以合同为一级对象，后续补充合同和追加订单会不断扰动成本归集边界。

本阶段不采用。

### 备选方案：项目合同合并为单一对象

项目表直接承载合同编号、金额、日期和状态，补充合同用备注或金额调整字段表达。该方案最轻，但无法可靠区分主合同、补充合同和销售订单归属，也无法保留合同生效后的商业历史。后续发票、回款和合同变更分析会返工。

本阶段不采用。

## 推荐模型

### 销售项目

销售项目是一级管理对象。核心属性包括项目编号、名称、客户、负责人、计划周期、状态、目标收入、目标成本和备注。

项目编号后端生成且不可编辑。客户是项目、合同和关联销售订单一致性的核心约束，项目创建后客户锁定。负责人引用现有系统用户，不引入员工主数据。目标收入和目标成本仅是管理目标，不代表实际收入、项目实际成本或财务口径。

项目进入 `ACTIVE` 后才允许被销售订单引用。`ACTIVE` 项目允许更新负责人、计划日期、目标收入、目标成本和备注，必须写审计摘要；项目编号和客户锁定。关闭和取消都是终态，但语义不同：关闭表示项目正常结束，取消表示草稿项目未形成有效业务事实。

### 合同

合同是项目下的商业事实。合同分为主合同和补充合同。主合同定义项目最初商业基础；补充合同表达后续商业增减。补充合同创建必须满足项目 `ACTIVE` 且主合同 `EFFECTIVE`，激活补充合同时再次校验。

不建立泛化合同版本或独立合同变更业务表。合同生效前允许编辑草稿；生效后关键商业字段不可直接覆盖。商业增减通过补充合同表达，状态动作和草稿变更通过系统审计保留。

主合同金额必须大于 0。补充合同调整金额允许正数或负数，但不得为 0。合同金额只作为商业金额，不拆税、不产生收入确认、不生成发票或应收。外部纸质合同号可空、可重复，不建唯一约束，可按关键词查询。

### 销售订单关联

销售订单是 020 唯一接入项目合同的既有业务单据。关联规则采用成对字段：`projectId` 与 `contractId` 必须同时为空或同时有值。

仅草稿销售订单可关联或解除项目合同。保存和确认时都要校验项目、合同、客户和状态，避免用户在草稿期间选中有效对象后，对象状态变化导致确认时形成错误归属。项目合同选择使用专用最小候选能力，权限由销售订单创建或更新权限控制，不要求完整项目或合同查看权限。

历史销售订单允许两个字段为空，不做批量回填。采购订单和生产工单不在 020 增加无业务入口的项目字段，分别留到 024 和 027。销售订单列表中 `projectLinked` 与 `projectId` 或 `contractId` 同时出现时返回校验错误，避免筛选语义含糊。

## 关系设计

```text
销售项目
  -> 一个非取消主合同
  -> 多个补充合同
  -> 多个销售订单

主合同
  -> 被补充合同引用
  -> 被草稿销售订单关联

补充合同
  -> 引用主合同
  -> 可被草稿销售订单关联
```

关系约束：

- 合同必须属于一个项目。
- 主合同和补充合同客户必须与项目客户一致。
- 补充合同必须引用同项目下的主合同，且只能在项目 `ACTIVE`、主合同 `EFFECTIVE` 时创建和激活。
- 一个项目最多一个非取消主合同。
- 销售订单关联的项目和合同必须同时存在，且合同属于项目。
- 销售订单客户必须与项目客户、合同客户一致。

## 状态流

### 项目状态流

```text
DRAFT
  -> ACTIVE
      -> CLOSED
  -> CANCELLED
```

`DRAFT` 是录入阶段。只有存在已生效主合同时，项目才能激活。`ACTIVE` 是后续业务可引用状态，可更新负责人、计划日期、目标收入、目标成本和备注。`CLOSED` 要求合同已到达终态，且关联销售订单状态固定为 `SHIPPED`、`CLOSED` 或 `CANCELLED`。`CANCELLED` 仅适用于未发生有效合同和销售订单关联的草稿项目。

项目状态不是期间控制，不使用 `BusinessPeriodGuard`。项目关闭和取消必须填写 1 至 200 字原因。项目状态动作必须记录操作人、时间、原因和关键上下文摘要。

### 合同状态流

```text
DRAFT
  -> EFFECTIVE
      -> CLOSED
      -> TERMINATED
  -> CANCELLED
```

合同草稿可以编辑和取消。合同“生效”动作在接口、权限和审计命名上统一使用 `activate`，状态仍为 `EFFECTIVE`。合同生效后形成商业事实，关键商业字段锁定。正常履约结束进入 `CLOSED`，提前结束进入 `TERMINATED`，两者均为终态。合同关闭、终止和取消都必须填写 1 至 200 字原因。取消只适用于草稿合同。

合同关闭或终止不自动关闭项目。项目关闭需要综合检查所有合同和关联销售订单。

## 页面架构

### 导航

在销售菜单下新增“销售项目”。它是销售域入口，不是财务或生产入口。菜单权限独立控制，避免用户通过项目页面绕过销售订单详情权限。

前端固定路由为 `/sales/projects`、`/sales/projects/create`、`/sales/projects/:id`、`/sales/projects/:id/edit`。`/sales` 根路径进入当前用户第一个可见销售子页面。

### 项目列表

列表用于快速定位项目，建议展示项目编号、名称、客户、负责人、状态、目标收入、目标成本、计划周期和合同摘要。筛选重点是关键词、客户、负责人和状态。金额展示必须明确为目标或商业金额，不能表达为实际收入、实际成本或毛利。有合同查看权限时可展示补充合同数量；无合同查看权限时合同字段为空或 `null`，`contractSummaryRestricted=true`，补充合同数量为 `null`。

### 项目表单

新建和编辑项目聚焦主档字段。项目编号不可编辑，状态通过动作流转，不在普通编辑表单中随意切换。项目创建后客户锁定。负责人选择使用专用最小用户候选能力，权限由项目创建或更新权限控制，不要求系统用户列表查看权限。

### 项目详情

详情页是 020 的核心工作台，建议分为基础信息、合同、关联销售订单和操作记录四个区域。

合同维护放在抽屉中，避免详情页堆叠过多表单。合同摘要和详情字段必须区分，摘要包含状态动作所需版本，详情包含完整字段。无合同查看权限时，项目详情中的合同字段为空或 `null`，并返回合同摘要受限状态，不泄露编号、金额、状态或数量。

关联销售订单摘要按权限展示：有 `sales:order:view` 才显示明细摘要和跳转；无权限时订单数量和摘要为空或 `null`，`salesOrderSummaryRestricted=true`，不得显示为 0，也不得返回敏感明细。

操作记录用于解释状态变化、合同终止、取消和销售订单关联变更，不能替代系统审计，但应能让业务用户理解关键历史。

### 销售订单表单

销售订单草稿表单增加项目和合同选择。选择项目和合同时使用专用最小候选能力，候选能力由销售订单创建或更新权限控制。选择项目后只允许选择该项目下已生效合同。项目和合同必须同选同清。确认动作再次校验项目和合同状态。

非草稿订单只读展示项目合同归属，不允许编辑。

### 窄屏策略

390px 视口下保留列表和详情可读性。表格允许横向滚动，但主状态、主要操作、错误信息和抽屉提交按钮必须可访问。项目和合同终态操作使用原因确认弹窗，弹窗包含 1 至 200 字文本域、错误区域、确认和取消按钮，窄屏下按钮必须可达。合同和操作记录在窄屏下优先使用分区、折叠或纵向信息块。

## 数据流

### 项目激活

用户创建项目草稿，维护主合同草稿并激活为 `EFFECTIVE`。用户点击激活项目时，后端检查项目存在已生效主合同，并确认项目未取消、主合同客户与项目客户一致。通过后项目进入 `ACTIVE`，写审计和操作记录。

### 销售订单关联

用户在销售订单草稿中选择项目和合同。前端根据项目筛选合同，后端保存时仍完整校验。销售订单确认时再次校验项目为 `ACTIVE`、合同为 `EFFECTIVE`、合同属于项目、客户一致。确认成功后，该关联成为后续销售履约归属骨架。

销售订单关联或解除写目标为 `SALES_PROJECT` 的审计记录，摘要包含订单号及旧、新关联，使项目操作记录可以查询到销售订单归属变化。

### 合同生效后变更

合同生效后，不允许覆盖关键商业字段。若商业金额或履约范围发生变化，用户新增补充合同。补充合同保留正负调整金额，未来阶段可用于合同累计商业金额、开票和回款分析。

补充合同只能在项目 `ACTIVE` 且主合同 `EFFECTIVE` 时创建，激活补充合同时再次校验项目和主合同状态。合同日期允许超出项目计划周期，020 不做强制校验。

### 项目关闭

用户发起项目关闭并填写 1 至 200 字原因。后端检查项目下合同均为 `CLOSED`、`TERMINATED` 或 `CANCELLED`，关联销售订单均为 `SHIPPED`、`CLOSED` 或 `CANCELLED`。检查通过后项目进入 `CLOSED`，否则返回明确阻断原因。

## 权限与审计

### 权限原则

项目和合同拥有独立查看、维护和状态动作权限。无合同查看权限时，项目列表和详情的合同字段返回空或 `null`，并返回合同摘要受限标记。销售订单关联仍受销售订单维护权限控制。项目详情内的销售订单摘要必须叠加 `sales:order:view`，不能因为用户有项目查看权限就泄露订单明细；无销售订单查看权限时订单数量和摘要返回空或 `null`，并返回订单摘要受限标记。

建议权限按对象和动作拆分，至少覆盖项目查看、项目维护、项目状态动作、合同维护、合同状态动作。具体权限编码由接口契约落定。

销售订单项目合同候选能力由销售订单创建或更新权限控制，不要求完整项目或合同查看权限。项目负责人候选能力由销售项目创建或更新权限控制，不要求系统用户列表查看权限。

### 审计原则

020 不使用业务期间保护，但必须保留系统审计。审计覆盖：

- 项目创建和编辑。
- 项目激活、关闭、取消。
- 合同创建和草稿编辑。
- 合同生效、关闭、终止、取消。
- 销售订单关联和解除项目合同。

审计基于现有 `sys_audit_log`，不扩展审计表。审计只保证动作、目标、操作人、时间、原因或关键变更字段摘要写入 `targetSummary`，不要求结构化变更前后值。终止、取消、关闭等状态动作的原因应进入操作记录，方便业务用户在页面理解历史。

## 错误处理

错误提示需要面向业务用户，而不是只暴露技术字段。主要错误类型包括：

- 项目状态不允许当前操作。
- 项目缺少已生效主合同，不能激活。
- 项目存在未终态合同或销售订单，不能关闭。
- 项目已有有效业务，不能取消。
- 项目关闭或取消原因缺失或长度不合法。
- 项目客户、合同客户和销售订单客户不一致。
- 同一项目已存在非取消主合同。
- 补充合同缺少主合同、引用了错误主合同，或项目和主合同状态不满足创建或激活条件。
- 合同状态不允许当前操作。
- 生效合同关键商业字段不可直接修改。
- 合同关闭、终止或取消原因缺失或长度不合法。
- 主合同金额必须大于 0。
- 补充合同调整金额不得为 0。
- 项目和合同必须同时选择或同时为空。
- `projectLinked` 与项目或合同筛选组合含糊。
- 无权限查看或维护项目、合同、销售订单摘要。

错误响应应能让前端定位到表单级、字段级或动作级提示。涉及关闭阻断时，应尽量返回阻断对象类型和数量，例如未终态合同数量或未终态订单数量。

## 历史兼容

020 不改写 000 至 019 的历史事实。历史销售订单、销售出库、应收、库存流水、报表和追溯允许没有项目归属。

历史销售订单如果仍是草稿，可以按新规则补选项目和合同；非草稿历史订单保持原状态和原链路，不提供强制补录入口。

采购订单、生产工单、库存余额和成本记录在 020 不增加项目字段。这样避免出现字段已经存在但没有业务入口、校验规则和验收路径的半成品能力。

## 测试设计

### 后端测试

覆盖项目创建、编辑、激活、关闭、取消，`ACTIVE` 项目允许字段更新，合同主补关系、主合同唯一性、补充合同创建和激活状态校验、金额规则、合同状态动作、生效后关键字段锁定、销售订单关联校验、`projectLinked` 互斥筛选、历史订单兼容、权限受限摘要和审计。

重点验证事务一致性：销售订单确认时项目或合同状态已变化时必须整体失败，不得留下部分状态或错误关联。

### 前端测试

覆盖项目列表筛选、项目表单、详情分区、合同抽屉、原因确认弹窗、状态动作按钮显隐、销售订单项目合同联动选择、最小候选权限、错误提示、无权限态、只读态、空态和接口类型。

非草稿销售订单的项目合同字段必须只读，不能通过页面操作解除。

### 浏览器验收

覆盖管理员主路径、只读路径、无权限路径、异常路径、历史订单兼容路径和 390px 窄屏。关键证据包括固定路由、项目列表、项目表单、项目详情、合同抽屉、原因确认弹窗、销售订单关联、操作记录、合同受限态、无权限关联订单摘要和错误阻断提示。

### 视觉分析

视觉分析应关注页面布局、信息密度、关键操作可见性、状态识别、金额口径文案、表格横向滚动、抽屉提交区域、原因确认弹窗、错误提示、权限态、空态和窄屏适配。目录建议为 `docs/testing/020-sales-project-contract-visual-audit/`。

## 交付边界

020 完成后，系统可以创建销售项目，维护主合同和补充合同，让草稿销售订单关联有效项目合同，并在项目详情中查询合同、关联订单摘要和操作历史。

020 完成后，系统仍不提供项目专属库存、库存计价、项目专采、项目生产成本、发票、收入确认、凭证、总账、月结、复杂审批、工艺路线或复杂质量能力。这些能力必须在后续阶段按路线分别设计和验收。
