# 035、036 阶段合并设计

## 决策结论

原 `035 上线准备与非功能验证` 与原 `036 最终生产交付验证` 合并为唯一最终阶段：

- 阶段编号：`035`
- 阶段名称：`上线准备与最终生产交付`
- 原 `036`：正式废止，不保留占位阶段，不再作为路线、依赖或后续能力引用
- 阶段定位：在 000—034 已交付业务能力基础上完成生产上线准备、冻结上线候选、最终全链路验收和完整生产交付

本次调整只改变后续阶段治理与路线，不修改 000—034 的历史交付事实、业务范围、迁移、接口、数据口径或验收结论。

## 合并原因

原 035 与原 036 没有新的业务模块依赖，目标都服务于同一个生产交付结果。继续拆成两个独立阶段会增加阶段说明、目标讨论、审查、交接和验收重复，但把所有活动混成一个无边界窗口又会失去上线候选冻结点。因此采用“一个阶段、两个顺序窗口、一个最终完成结论”的结构。

## 范围边界

035 不新增业务功能，不扩展 000—034 已冻结的业务规则、状态机、接口语义、数据口径和产品范围。

允许范围：

- 权限矩阵核对、最小权限和职责分离验证。
- 管理员、数据库、对象存储和运行环境凭据轮换及敏感配置治理。
- 登录、会话、CSRF、接口权限、敏感字段和依赖安全检查与必要加固。
- 性能、容量、并发、批量任务、月结、报表和关键查询的非功能验证与必要优化。
- PostgreSQL、MinIO、应用配置和发布物的备份恢复演练。
- V34 至 V36 及当前最终迁移版本的真实副本迁移演练、失败处理和恢复验证。
- API、Web、代理、数据库、对象存储的可重复部署、健康检查、日志、监控、告警、回滚和运维基线。
- 冻结上线候选后执行一次完整正向、反向、权限和月结财务全链路桌面验收。
- 最终交付包、部署说明、运维说明、恢复说明、验收记录和生产发布结论。

禁止范围：

- 固定资产、工资薪酬、多币种、正式收入确认、法定财务报表、完整税务申报、完整红字发票、APS、MES、完整 QMS、WMS 等新业务能力。
- 为通过验收降低业务规则、安全要求、权限边界、数据一致性或验收标准。
- 使用阶段性测试、接口测试或代码审查替代真实生产候选的最终浏览器验收。
- 在上线候选冻结后静默修改业务契约、迁移含义或财务口径。

## 阶段内部结构

### 窗口一：上线准备与候选冻结

目标是证明系统能够安全、稳定、可恢复、可重复地部署运行，并形成唯一上线候选。

至少完成：

1. 冻结生产拓扑、环境参数、密钥来源、数据源、对象存储和发布物版本。
2. 完成权限矩阵、安全专项、凭据轮换和敏感信息检查。
3. 按正式数据规模与关键并发场景完成性能和容量验证。
4. 完成 PostgreSQL 与 MinIO 联合备份、恢复、迁移、失败回退和恢复后完整性验证。
5. 完成部署、重启、健康、日志、监控、告警、任务恢复和运维演练。
6. 处理全部阻断和严重问题，只复验修复差异及受影响路径。
7. 冻结唯一候选提交、生产包、配置清单和数据迁移结果。

窗口一未通过时不得进入窗口二。

### 窗口二：最终生产交付验证

目标是在窗口一冻结的同一候选、同一生产式环境和同一连贯数据集上证明业务与交付闭环。

至少完成：

1. 正向主链：项目/合同、销售订单、缺料、采购/生产/外协、库存、发货、发票/收付款/费用、项目成本、业务月结、凭证/总账、财务结账和经营分析。
2. 反向主链：退货、退款或冲减、凭证冲销、业务月结重开、财务反结账和来源变化后的重新处理。
3. 权限主链：超级管理员、业务角色、财务角色、受限角色及双人审批职责分离。
4. 本阶段相关页面的真实桌面功能与页面规范核对；不截图、不测试移动端。
5. 完整交付包、部署与恢复材料、已知问题、验收范围和生产发布决策核对。

窗口二原则上不安排开发。发现阻断或严重缺陷时，必须解除候选冻结，由对应固定角色修复并完成受影响验证；涉及业务契约、数据模型或财务口径变化时，不得作为普通 035 整改处理，应建立新的业务阶段或经用户确认调整范围，然后重新执行窗口一候选冻结和窗口二最终验证。

## 完成条件

只有同时满足以下条件，035 才能标记完成：

- 窗口一全部上线准备与非功能门禁通过。
- 唯一上线候选的提交、发布物、配置、迁移版本和数据源可追溯。
- 窗口二正向、反向、权限、月结财务和桌面验收全部通过。
- 阻断和严重问题为 0；一般问题有明确记录且不影响生产使用。
- 备份可恢复、迁移可复现、部署可重复、服务可监控、故障可处置。
- 最终交付包完整，生产发布结论、访问方式和责任边界明确。

035 完成即代表当前第一版路线完成生产交付；不再存在 036 后续阶段。
