# 根工作台设计规格

## 目标

将根路由 `/` 从历史占位页改造成制造业 ERP 的综合工作台，帮助当前登录用户按以下顺序完成每日扫描：

```text
快捷进入常用业务 -> 处理个人工作流 -> 判断经营与关账状态 -> 处理业务异常
```

工作台只聚合、摘要和导航现有业务能力，不新增业务状态机，不替代审批中心、消息中心、任务中心、经营报表或各业务模块。

## 设计决策

2026-07-23 共形成三套可比视觉方向，用户最终选择当前会话中第三个展示方案。选定方向的核心结构为：

1. 快捷入口。
2. 我的工作流。
3. 本月经营概况。
4. 关键状态。
5. 业务关注。

实现时保留该结构与扫描顺序，但按 `docs/ui/page-standards.md` 和当前 QH ERP 设计令牌收敛概念图中的多彩装饰：

- 黑白中性色作为主体。
- 品牌绿色只表达正常、完成和运行进度。
- 红色、橙色只表达严重或警告。
- 蓝色只用于明确的可点击链接。
- 使用现有 Element Plus 图标，不制作自定义图形资产。
- 只保留外层业务分区，不使用卡片套卡片，不给每条记录单独加厚重卡片。
- 不使用营销式首屏、插画、渐变大背景、巨大指标、无业务价值图表或装饰性大留白。

## 方案比较与技术选择

### 方案一：前端编排现有只读接口

根工作台按权限并行读取审批、文档任务、经营概览、经营财务状态和经营异常接口，在前端形成只读摘要。

优点：

- 不新增业务接口和跨域聚合模型。
- 复用已经验收的权限与数据口径。
- 单个分区失败不影响其他分区。
- 工作台保持轻量，只承担摘要和导航。

代价：

- 首次进入会产生多次并行只读请求。
- 文档任务的处理中和失败数量需要按状态分别查询后汇总。

### 方案二：新增后端聚合接口

后端新增统一工作台端点，一次返回全部摘要。

优点是请求数量少；缺点是把审批、文档任务、报表、月结和权限边界耦合到新的跨域接口。本阶段没有新增业务需求，现有接口已能提供所需数据，因此不采用。

### 方案三：根路由直接跳转经营概览

实现成本低，但缺少个人工作流、快捷入口和异常处置，不满足工作台目标，因此不采用。

最终采用方案一，不新增后端接口。

## 页面范围

新增或修改页面：

- `/`：根工作台，替换历史占位组件。

直接受影响文件预计包括：

- 根工作台视图及其前端测试。
- 路由配置及其测试。
- 工作台专用的数据编排或展示辅助模块及其测试。
- 本阶段唯一任务说明、实施计划、设计质量记录和交接记录。

不修改：

- 审批、消息、文档任务、经营报表、业务月结、会计期间、财务关闭的业务规则。
- 现有接口返回结构和权限点。
- 其他业务页面的布局与交互。
- 顶部栏、侧栏和全局导航的信息架构。

## 页面结构

### 标题区

- 标题：`工作台`。
- 说明：`从待处理到异常处置，按业务节奏推进今日工作。`
- 右侧操作：`刷新`。
- 刷新按钮旁展示最近一次完成加载的时间，例如 `更新于 10:28`。
- 刷新只重新加载当前账号有权查看的分区。

### 常用入口

使用单行或自动换行的紧凑按钮组，按钮全部是可识别的系统操作按钮，不使用普通超链接样式。

入口按权限展示：

| 名称 | 路由 | 权限 |
| --- | --- | --- |
| 销售项目 | `/sales/projects/create` | `sales:project:create` |
| 采购请购 | `/procurement/requisitions/create` | `procurement:requisition:create` |
| 生产工单 | `/production/work-orders/create` | `production:work-order:create` |
| 库存单据 | `/inventory/documents/create` | `inventory:document:create` |
| 质量检验 | `/quality/inspections` | `quality:inspection:view` |

规则：

- 名称使用业务对象名称，不使用概念图中的虚假“新增质量检验”语义。
- 无权限入口不渲染。
- 当前账号没有任何入口权限时，整个分区不占空白高度。
- 每个入口使用现有 Element Plus 图标库中最接近业务语义的图标。

### 我的工作流

左侧主分区按三个连续分组展示，不使用普通大表格，也不把每行做成独立卡片。

#### 等待我处理

- 数据源：`documentPlatformApi.approvalTasks.list`。
- 查询：`scope=TODO`、`page=1`、`pageSize=10`。
- 权限：`platform:todo:view`。
- 分组计数使用接口 `total`，前端按时间排序后最多展示最近 3 条。
- 行内容：业务对象名称或单号、当前步骤、申请人、分配时间。
- 行操作：`处理`，进入 `/platform/approvals`。
- 分组操作：`查看全部`，进入 `/platform/approvals`。

#### 系统处理中

- 数据源：`documentPlatformApi.documentTasks.list`。
- 权限：`platform:document-task:view`。
- 分别按 `QUEUED`、`RUNNING`、`READY_TO_COMMIT` 查询，每次使用 `page=1`、`pageSize=10`，然后合并。
- 分组计数为各状态结果 `total` 之和。
- 列表按创建时间倒序合并，最多展示 3 条。
- 行内容：任务类型中文名称、任务号或业务对象、中文状态、进度、创建时间。
- 行操作：`查看`，进入 `/platform/document-tasks?taskId={id}&returnTo=%2F`。
- 分组操作：`查看全部`，进入 `/platform/document-tasks`。

#### 需要重新处理

- 数据源：`documentPlatformApi.documentTasks.list`。
- 权限：`platform:document-task:view`。
- 分别按 `VALIDATION_FAILED`、`FAILED` 查询，每次使用 `page=1`、`pageSize=10`，然后合并。
- 分组计数为各状态结果 `total` 之和。
- 列表按创建时间倒序合并，最多展示 2 条。
- 行内容：任务类型中文名称、失败原因摘要、中文状态、完成或创建时间。
- 行操作：`处理`，进入对应任务中心筛选。
- 分组操作：`查看全部`，进入任务中心。

工作流规则：

- 状态与任务类型统一复用现有中文字典；未知值显示中文兜底描述，不直接显示英文枚举。
- 进度条只在有合法进度值时显示，范围限制在 0 到 100。
- 单个分组无数据时显示紧凑空态，不用整页大空态。
- 当前账号没有审批和文档任务权限时，工作流区域显示明确的权限空态，不发起无权限接口请求。

消息中心不混入本分组，避免把消息、审批和系统任务变成含义不清的总计；消息继续由现有消息中心负责。

### 本月经营概况

- 数据源：`businessReportingApi.overview.get()`。
- 权限：`report:overview:view`。
- 展示接口返回期间，不在前端伪造期间。
- 固定展示四个指标：
  - 销售出库经营金额。
  - 采购入库经营金额。
  - 生产完工数量。
  - 成本归集金额。
- 金额使用人民币格式，数量使用千分位和业务单位说明。
- 指标采用同一外层表面内的 2×2 网格，用分隔线组织，不使用四张厚重卡片。
- 区块底部提供 `查看经营概览` 按钮，进入 `/reports/overview`。
- 页面明确沿用经营口径，不把金额解释为正式财务入账。

### 关键状态

- 数据源：`businessReportingApi.operatingFinanceOverview.get({ analysisMode: 'LIVE' })`。
- 权限：`report:operating-finance:view`。
- 展示：
  - 当前期间。
  - 业务月结状态。
  - 会计期间状态。
  - 财务关闭状态。
  - 经营/会计定稿状态。
- 所有状态使用现有中文映射与语义标签。
- 管理入口分别按对应页面权限展示：
  - 业务月结：`/period-close/runs`，`system:business-period-close:view`。
  - 会计期间：`/gl/accounting-periods`，`gl:period:view`。
  - 财务关闭：`/gl/financial-close`，`financial-close:period:view`。
- 只有汇总查看权限、没有管理权限时仍可查看汇总状态，但不显示越权按钮。

### 业务关注

- 数据源：`businessReportingApi.exceptions.list`。
- 查询：`page=1`、`pageSize=10`，前端最多展示 4 条。
- 权限：`report:exception:view`。
- 展示最近或接口默认排序的最多 4 条可行动异常：
  - 异常类型中文名称。
  - 严重程度中文标签。
  - 异常对象；来源受限时显示 `来源受限`。
  - 异常说明。
  - 业务日期。
- 每行操作使用 `进入清单`，进入 `/reports/exceptions`，不伪装成当前页面无法提供的行级详情。
- 底部操作：`查看全部业务关注`。
- 无权限时整个分区不渲染；接口返回来源受限时不得泄露对象标识或追溯键。

## 数据加载与错误隔离

- 页面挂载和点击刷新时启动一次统一加载。
- 只调用当前账号拥有对应权限的接口。
- 授权分区使用并行请求，整体采用等价于 `Promise.allSettled` 的隔离策略。
- 各分区独立维护加载、成功、空、失败和无权限状态。
- 任一接口失败时，不清空其他成功分区，不把错误误显示为数值 `0`。
- 分区失败显示短错误说明和 `重新加载` 操作。
- 刷新期间保留已有成功数据，避免页面闪空；对应区域显示轻量加载状态。
- 最近更新时间只在本轮至少一个授权分区完成后更新。
- 页面卸载后不写入已销毁视图状态。

## 权限与安全

- 根路由增加 `requiresAuth: true`，避免未登录用户进入工作台后再依赖页面内请求失败判断身份。
- 工作台本身不新增权限点。
- 页面展示权限与现有 API 权限同时生效，后端仍是最终安全边界。
- 无权限时不得请求对应接口、显示敏感数量或用 `0` 冒充真实业务结果。
- 快捷入口、管理按钮和查看按钮全部按现有权限过滤。
- 工作台不执行任何写操作，不提交审批，不确认任务，不改变单据和期间状态。

## 视觉与交互规范

### 设计令牌

- 背景、表面、文字、边框、圆角和阴影全部使用 `style.css` 中现有 `--qherp-*` 令牌。
- 主文字使用 `--qherp-text`，辅助文字使用 `--qherp-steel` 或 `--qherp-stone`。
- 正常状态使用 `--qherp-brand-green-deep`。
- 警告和严重状态使用 `--qherp-brand-warn`、`--qherp-danger`。
- 不新增第二套品牌色、渐变、字体或阴影系统。

### 密度与层级

- 页面沿用 `.module-page` 的 16px 垂直节奏。
- 标题区沿用 `.page-heading`，标题 28px，说明 14px。
- 外层业务分区使用白色表面、1px 边框、12px 圆角；原则上无阴影。
- 区块标题 16px，正文 14px，辅助信息 12–13px。
- 数字使用等宽数字表现。
- 行高保证按钮完整可点，不出现文字、标签或按钮堆叠。
- 行悬浮只使用轻微表面色变化。

### 桌面布局

- 常用入口位于标题区下方，优先单行展示，空间不足时整按钮换行。
- 主体使用左右双栏：
  - 左栏约占 60%–66%，承载我的工作流。
  - 右栏约占 34%–40%，纵向承载经营概况和关键状态。
- 业务关注位于双栏下方并占整行。
- `1280×720` 下：
  - 标题、刷新、常用入口、工作流核心分组和经营概况不重叠。
  - 所有可见操作按钮文字完整、可点击。
  - 页面滚动发生在主内容区。
  - 业务关注允许位于首屏下方，通过主内容区纵向滚动访问。
- 更窄布局可以转为单栏，但本阶段自动化测试与验收仅覆盖桌面端。

## 空态和降级

- 没有工作流记录：`当前没有需要处理的事项`。
- 没有经营数据：`当前期间暂无经营概况`。
- 没有关键状态：`暂无经营与关账状态`。
- 没有经营异常：`当前没有需要关注的经营异常`。
- 当前账号只有少量权限时，已有分区自然补位，不保留无权限空卡片。
- 当前账号没有任何工作台数据权限时，展示简洁说明：`当前账号暂无可展示的工作台数据，请从左侧菜单进入已授权功能。`

## 测试与验收

### 自动化测试

- 根路由使用真实工作台组件并要求登录。
- 有权限时才调用对应 API；无权限时不调用。
- 审批、处理中任务和失败任务的总数、排序、截取正确。
- 所有工作流状态和任务类型使用中文文案。
- 快捷入口和管理按钮按权限显示。
- 单个接口失败不影响其他分区。
- 空态、失败态、部分权限态和刷新态正确。
- 行操作生成正确路由并保留 `returnTo=/`。
- 数字、金额、时间和进度格式稳定。

### 真实桌面浏览器检查

检查视口：

- `1280×720`：主要验收视口。
- `1440×900`：完整结构与信息密度复核。

检查状态：

- 超级管理员全量数据。
- 部分权限账号。
- 空数据。
- 单分区请求失败。
- 工作流记录较长文本。
- 快捷入口不足五个。

检查项：

- 页面骨架、标题和说明符合 `docs/ui/page-standards.md`。
- 无营销式大留白、装饰性首屏和卡片嵌套。
- 扫描顺序清楚，工作流是主视觉焦点。
- 文案、状态、日期、金额和数量均可读。
- 按钮不堆叠、不裁切、完整可点击。
- `1280×720` 下无关键内容重叠，无页面级横向滚动。
- 加载、空、失败、无权限状态可区分。
- 真实页面与选定的第三套视觉方案保持结构一致，并完成设计差异对比。

## 完成条件

- `/` 不再显示历史占位文案。
- 工作台使用真实权限与真实只读业务数据。
- 五个页面分区按本规格完成并具备独立状态。
- 不新增后端接口和业务写操作。
- 定向前端测试、类型检查、生产构建通过。
- 唯一交付前全量验证通过。
- 主代理完成全部代码和自验后，固定测试角色完成独立代码评审；阻断和严重问题修复并完成差异复审。
- 工作台从最新主分支服务启动，用户可在浏览器验收真实页面。

## 规格自检

- 本规格只覆盖根工作台及其直接依赖，不扩展其他业务页面。
- 本规格不新增业务模块、状态机、权限点或写接口。
- 工作台不是 BI 大屏，不复制现有业务页面的完整功能。
- 所有数据都来自现有只读接口和现有权限口径。
- 选定方案的布局层级已保留，概念图中的装饰性多彩样式已按项目规范收敛。
- `docs/ui/page-standards.md` 的页面骨架、标题、按钮、权限、状态、桌面端布局和真实浏览器检查要求已纳入验收。
