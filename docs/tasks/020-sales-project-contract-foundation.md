# 任务记录：销售项目与合同管理基础

## 任务目标

建立 QH ERP 第一版后续阶段的销售项目主维度，让后续销售履约、项目专采、库存计价、生产工单、费用、发票、收付款、项目成本和利润分析都能引用统一 `projectId`。

020 阶段只交付销售项目与合同管理基础，形成项目主档、主合同、补充合同、销售订单可选关联、权限、审计、页面和验收矩阵。当前 020 尚未实现，本文件是阶段范围、业务规则和验收的唯一权威来源。

本阶段不是项目成本核算、库存计价、项目专采、发票、收入确认、总账、月结或复杂审批阶段。

## 背景与输入

- 000 至 019 已完成并交付，系统已有客户、销售订单、销售出库、采购、生产、库存、往来、报表、期间保护、质量状态、库存可用量、批次序列追溯和权限基础。
- 最新产品路线确认：销售项目是一级成本、收入和利润对象；一个项目允许一个主合同、多个补充合同和多次追加订单。
- 后续 023 至 033 的库存计价、项目专采、项目生产、发票、费用、项目成本、月结、总账和经营财务分析都依赖稳定项目标识。
- 020 启动时，采购订单、生产工单、库存余额、费用、发票和正式成本尚无项目归属字段，不得在 020 通过无业务入口字段提前塞入。
- 当前工作区可能存在主代理治理文档改动，阶段实施不得撤销、覆盖或整理非本阶段文件改动。

## 范围

### 本次包含

- 建立销售项目主档，字段至少包含项目编号、项目名称、客户、项目负责人、计划开始日期、计划结束日期、状态、目标收入、目标成本和备注。
- 项目负责人引用现有 `sys_user`，不新建员工主数据。
- 建立项目内合同管理，包含主合同和补充合同。
- 一个项目最多存在一个非取消主合同；补充合同必须引用该项目的主合同，且仅允许在项目为 `ACTIVE`、主合同为 `EFFECTIVE` 时创建，生效时再次校验。
- 项目编号和内部合同编号由后端生成且不可编辑；外部纸质合同号可空、可重复，不建唯一约束，可按关键词查询。
- 主合同金额必须大于 0；补充合同调整金额允许正数或负数，但不得为 0。
- 合同金额在 020 只是商业金额，不拆税、不作为收入确认、发票、应收或总账依据。
- 目标收入和目标成本为非负管理目标，不等同实际收入或实际成本。
- 项目状态：`DRAFT`、`ACTIVE`、`CLOSED`、`CANCELLED`。
- 合同状态：`DRAFT`、`EFFECTIVE`、`CLOSED`、`TERMINATED`、`CANCELLED`。
- 销售订单增加项目和合同可选关联：`projectId` 与 `contractId` 必须同时为空或同时有值。
- 仅销售订单草稿创建或更新允许关联或解除项目合同；确认时必须再次校验关联有效性。
- 项目详情展示合同、关联销售订单摘要和操作记录。
- 关联销售订单摘要继续受 `sales:order:view` 权限约束，无权限不得泄露销售订单明细。
- 销售菜单下新增“销售项目”入口，固定前端路由为 `/sales/projects`、`/sales/projects/create`、`/sales/projects/:id`、`/sales/projects/:id/edit`；`/sales` 根路径进入当前用户第一个可见销售子页面。
- 项目列表、新建、编辑和详情为独立页面；合同在项目详情内通过抽屉维护，合同摘要和详情字段明确区分，列表和项目详情摘要包含状态动作所需版本。
- 项目和合同终态动作使用原因确认弹窗，包含 1 至 200 字文本域、错误区域、确认和取消按钮；390px 窄屏下按钮必须可达。
- 桌面和 390px 窄屏均需浏览器验收；窄屏表格允许横向滚动，但不得遮挡核心操作、状态和错误信息。
- 所有项目和合同写入、状态动作、销售订单关联或解除必须写系统审计；审计基于现有 `sys_audit_log`，不扩展审计表。

### 本次不包含

- 项目专属库存、公共库存分配、库存计价、成本层和库存金额。
- 请购、询价比价、价格协议、项目专采、采购订单项目关联和采购结案。
- 销售报价、信用控制、完整交付计划、项目交付结案和销售深化。
- 缺料净算、供给建议、计划订单、完整 MRP 和 APS。
- 生产工单项目字段、工单项目成本自动归集、外协成本和在制完工成本。
- 销售发票、采购发票、收入确认、费用单、预收预付、凭证草稿。
- 项目实际成本、项目毛利、月结、期间快照、总账、财务结账和正式财务报表。
- 泛化合同版本、独立合同变更业务表、完整审批流或工作流设计器。
- 工艺路线、工作中心、工序派工、复杂质量体系、MES、设备采集和扫码。
- 多公司、多组织、多账套、多租户、多币种、固定资产和工资薪酬。

## 核心业务规则

### 项目规则

- 项目编号由后端生成，系统内唯一，一旦生成不得编辑。
- 项目客户必须引用已存在客户；项目创建后客户锁定，不通过普通更新变更。
- 项目负责人引用现有启用用户；负责人候选使用专用最小用户候选能力，由 `sales:project:create` 或 `sales:project:update` 控制，不要求系统用户列表查看权限。
- 项目计划开始日期不得晚于计划结束日期。
- 项目目标收入和目标成本必须大于等于 0，允许为空时按未填写处理；不得由前端临时计算作为后端可信值。
- `ACTIVE` 项目允许更新负责人、计划日期、目标收入、目标成本和备注，必须写审计摘要；项目编号和客户锁定；`CLOSED`、`CANCELLED` 项目不可更新。
- 项目创建状态固定为 `DRAFT`。
- 项目激活要求存在已生效主合同。
- 项目关闭要求项目下合同均处于终态，关联销售订单状态固定为 `SHIPPED`、`CLOSED` 或 `CANCELLED`；其他销售订单状态均阻止关闭。
- 项目取消仅限未发生有效合同和销售订单的草稿项目。
- 项目关闭和项目取消均要求填写原因，长度 1 至 200；原因缺失使用项目原因缺失错误码。

### 合同规则

- 合同归属于销售项目，必须引用同一项目客户，不允许跨项目或跨客户复用。
- 合同类型分为主合同和补充合同。
- 一个项目最多一个非取消主合同；已取消主合同不计入该限制，但保留历史和审计。
- 补充合同必须引用同项目下的主合同；补充合同仅允许在项目为 `ACTIVE` 且主合同为 `EFFECTIVE` 时创建，生效时再次校验，否则返回稳定业务错误。
- 内部合同编号由后端生成且不可编辑；外部纸质合同号可为空、可重复，不建唯一约束，如填写必须在页面和详情中保留，并支持关键词查询。
- 合同草稿可编辑和取消。
- 合同生效后，项目、合同类型、主合同引用、客户、内部合同编号、合同金额、签订日期、履约开始日期和履约结束日期等关键商业字段不可直接覆盖。
- 合同生效后商业增减通过新增补充合同表达，不建立泛化合同版本或独立合同变更业务表。
- 主合同金额必须大于 0；补充合同调整金额允许正负但不得为 0。
- 合同“生效”动作统一使用 `activate` 命名，状态仍为 `EFFECTIVE`。
- 合同正常履约结束为 `CLOSED`；提前终止为 `TERMINATED`；两者均为终态。
- 合同关闭、合同终止和合同取消均要求填写原因，长度 1 至 200；合同原因缺失使用合同错误码。
- 合同取消只允许草稿状态。
- 合同日期允许超出项目计划周期，020 不做强制校验。

### 销售订单关联规则

- 020 只改造销售订单，不改造采购订单和生产工单。
- 销售订单的 `projectId` 与 `contractId` 必须同时为空或同时有值。
- 仅销售订单草稿创建或更新允许关联或解除项目合同。
- 关联时项目必须为 `ACTIVE`，合同必须为 `EFFECTIVE`，合同必须属于该项目，且项目客户、合同客户、销售订单客户一致。
- 销售订单确认时必须再次校验项目和合同仍满足关联规则。
- 已确认、部分出库、全部出库、已关闭或已取消销售订单不得变更项目合同关联。
- 历史销售订单允许 `projectId` 和 `contractId` 均为空，不回填伪造归属。
- 销售订单项目合同选择使用专用最小候选能力，权限由 `sales:order:create` 或 `sales:order:update` 控制，不要求完整项目或合同查看权限。
- 销售订单列表筛选中 `projectLinked` 与 `projectId` 或 `contractId` 同时出现时返回 `VALIDATION_ERROR`，禁止含糊组合。
- 项目关闭时，关联销售订单必须处于 `SHIPPED`、`CLOSED` 或 `CANCELLED`；存在 `DRAFT`、`CONFIRMED` 或 `PARTIALLY_SHIPPED` 等其他状态订单时不得关闭。

### 期间、审计和历史保留

- 项目和合同属于商业主数据与管理事实，020 不使用 `BusinessPeriodGuard`。
- 所有写入和状态动作必须审计，包括项目创建、编辑、激活、关闭、取消，合同创建、编辑草稿、生效、关闭、终止、取消，销售订单关联和解除。
- 草稿变更和状态动作写系统审计；终止、取消和关闭原因按规格要求记录。
- 项目和合同操作日志基于现有 `sys_audit_log`，只保证动作、目标、操作人、时间、原因或关键变更字段摘要写入 `targetSummary`；不要求结构化变更前后值，不扩展审计表。
- 销售订单关联或解除写目标为 `SALES_PROJECT` 的审计记录，摘要包含订单号及旧、新关联，使项目操作记录可查询。
- 更新请求与创建请求拆分，项目和合同更新请求必须显式包含版本；所有状态动作也必须携带版本。
- 已生效合同不通过覆盖关键商业字段表达变更，避免破坏历史商业事实。
- 000 至 019 的历史订单、采购、生产、库存、往来和报表不因 020 自动补项目归属。

## 状态机

### 项目状态

| 状态 | 含义 | 允许操作 |
|---|---|---|
| `DRAFT` | 草稿项目，尚未成为后续业务可引用对象 | 编辑、维护合同草稿、取消、在存在已生效主合同后激活 |
| `ACTIVE` | 生效项目，可被草稿销售订单关联 | 更新负责人、计划日期、目标收入、目标成本和备注，维护补充合同，关联销售订单，关闭 |
| `CLOSED` | 项目正常结束 | 查询、查看合同和关联订单，不允许新增业务关联 |
| `CANCELLED` | 未发生有效业务的草稿项目作废 | 查询和审计追溯 |

状态流转：

```text
DRAFT -> ACTIVE -> CLOSED
DRAFT -> CANCELLED
```

### 合同状态

| 状态 | 含义 | 允许操作 |
|---|---|---|
| `DRAFT` | 合同草稿，尚未形成生效商业事实 | 编辑、生效、取消 |
| `EFFECTIVE` | 已生效合同，可被销售订单关联 | 关闭、终止；关键商业字段不可直接覆盖 |
| `CLOSED` | 合同正常结束 | 查询和追溯 |
| `TERMINATED` | 合同提前终止 | 查询和追溯，必须保留终止原因 |
| `CANCELLED` | 草稿合同作废 | 查询和追溯 |

状态流转：

```text
DRAFT -> EFFECTIVE -> CLOSED
DRAFT -> EFFECTIVE -> TERMINATED
DRAFT -> CANCELLED
```

## 验收矩阵

### 主路径

- 管理员能创建草稿项目，填写客户、负责人、计划日期、目标收入和目标成本。
- 管理员能在项目详情中创建主合同草稿并生效。
- 项目存在已生效主合同后可以从 `DRAFT` 激活为 `ACTIVE`。
- 项目为 `ACTIVE` 且主合同为 `EFFECTIVE` 时，管理员能在项目详情中创建多个补充合同，补充合同均引用主合同。
- 管理员能在草稿销售订单中选择 `ACTIVE` 项目和 `EFFECTIVE` 合同，保存和确认时关联保持一致。
- 项目详情能展示合同列表、关联销售订单摘要和操作记录。
- 合同关闭、终止和项目关闭按状态规则执行。

### 异常路径

- 无已生效主合同时激活项目被拒绝。
- 同一项目创建第二个非取消主合同被拒绝。
- 项目非 `ACTIVE` 或主合同非 `EFFECTIVE` 时创建、激活补充合同被拒绝。
- 补充合同未引用主合同或引用非本项目主合同被拒绝。
- 主合同金额小于等于 0 被拒绝。
- 补充合同调整金额等于 0 被拒绝。
- 项目、合同、销售订单客户不一致时关联被拒绝。
- 销售订单 `projectId` 与 `contractId` 只填一个时保存或确认被拒绝。
- 销售订单关联非 `ACTIVE` 项目或非 `EFFECTIVE` 合同被拒绝。
- 非草稿销售订单变更或解除项目合同关联被拒绝。
- 已生效合同关键商业字段直接覆盖被拒绝。
- 合同终止未填写原因被拒绝。
- 项目关闭、项目取消、合同关闭、合同终止或合同取消未填写 1 至 200 字原因被拒绝。
- 存在非终态合同或关联销售订单状态不属于 `SHIPPED`、`CLOSED`、`CANCELLED` 时项目关闭被拒绝。
- 已有有效合同或销售订单关联的项目取消被拒绝。
- `projectLinked` 与 `projectId` 或 `contractId` 同时筛选时被拒绝。

### 权限路径

- 无项目查看权限的用户不能访问销售项目菜单、列表、详情或接口数据。
- 只读用户可以查看授权范围内的项目、合同、关联订单摘要和操作记录，但不能新增、编辑或执行状态动作。
- 无 `sales:contract:view` 时，项目列表和详情的合同字段返回空或 `null`，`contractSummaryRestricted=true`，`supplementContractCount=null`，不得泄露合同编号、金额、状态或数量。
- 无合同维护权限的用户不能创建、编辑、生效、关闭、终止或取消合同。
- 无 `sales:order:view` 时，`salesOrderCount` 和订单摘要返回空或 `null`，`salesOrderSummaryRestricted=true`，不得显示为 0，不得泄露订单明细。
- 销售订单项目合同选择候选能力由销售订单创建或更新权限控制，不要求完整项目或合同查看权限。
- 项目负责人候选能力由销售项目创建或更新权限控制，不要求系统用户列表查看权限。
- 直接调用接口不能绕过前端按钮权限。

### 历史兼容路径

- 000 至 019 已有销售订单允许 `projectId` 和 `contractId` 为空，列表、详情、出库、往来和报表既有路径继续可查询。
- 020 不为历史采购订单、生产工单、库存流水、应收应付和报表自动伪造项目归属。
- 未关联项目的历史销售订单不阻塞项目、合同新功能使用。
- 已有关联为空的销售订单在草稿状态下可按新规则选择项目和合同。

### 审计路径

- 项目创建、编辑、激活、关闭、取消均产生审计记录。
- 合同创建、编辑草稿、生效、关闭、终止、取消均产生审计记录。

## 风险与处理

- 项目和合同边界膨胀：020 只建立主维度和合同基础，不进入库存、成本、发票、月结和总账。
- 合同变更过度设计：020 不建立泛化版本或合同变更业务表，生效后商业增减通过补充合同表达。
- 历史数据被误归属：历史订单允许项目合同为空，不自动回填，避免伪造业务事实。
- 销售订单关联破坏既有链路：仅草稿允许关联和解除，确认时再次校验，不影响已确认后的出库、往来和报表链路。
- 权限泄露：项目详情中的合同摘要必须复用合同查看权限，销售订单摘要必须复用销售订单查看权限；无权限返回空或 `null` 与受限标记，不得泄露编号、金额、状态、数量或订单明细。
- 关闭条件不完整：项目关闭必须检查合同终态和销售订单 `SHIPPED`、`CLOSED`、`CANCELLED` 终态集合，避免后续项目成本和收入对象提前封存。
- 页面信息过密：项目详情用合同、关联订单和操作记录分区，合同维护放抽屉，移动端以查看和状态识别优先。

## 变更控制

- 变更原因：020 从 019 后续路线启动，先建立销售项目和合同主维度，为后续项目型制造业财成本闭环提供统一对象。
- 影响范围：销售项目、合同、销售订单、权限、审计、菜单、前端页面、测试和视觉验收。
- 涉及文件：以本任务记录、设计规格、后续接口契约、测试计划和实施计划为准。
- 验收影响：新增项目合同主路径、销售订单关联路径、权限路径、异常路径、历史兼容路径、审计路径和视觉分析。
- 决策依据：用户确认的 020 固定决策、最新交接路线和产品决策记录。
- 变更规则：任何扩大到采购、生产、库存计价、成本、发票、总账、月结或审批流的要求，必须记录为范围变更并提交主代理决策；不得在实现中静默加入。

## 缺陷与阻断

- 阻断缺陷：迁移失败、项目或合同核心页面不可访问、项目激活或关闭规则错误、合同唯一主合同规则错误、补充合同项目和主合同状态规则错误、销售订单关联校验错误、历史订单无法继续查询、权限绕过、受限摘要泄露、审计缺失、关键接口不可用、视觉分析缺失。
