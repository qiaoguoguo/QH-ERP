## 2026-07-15 决策记录

### 方案比较

1. 轻量补丁：只增加报价、简单计划和静态额度。改动较小，但确认后变更、退货后的履约事实和 026 输入仍不可信，不采用。
2. 分层履约闭环：复用 020 合同与既有订单/出库，新增报价、确认时商业快照、受控订单变更、行级交付计划、商业信用、项目履约聚合和只读有效需求。边界完整、来源可信、可渐进交付，采用。
3. 泛化销售平台：同时引入 CRM、合同版本引擎、复杂价格引擎、运输和正式财务。明显越过 025，不采用。

### 已冻结产品决策

- 025 固定使用人民币 `CNY`，不引入多币种和汇兑。
- 020 主合同和补充合同继续是唯一合同商业增补机制；不建立第二套泛化合同版本模型。
- 报价可只关联客户，也可提前关联同客户的非终态项目；正式转项目订单时必须同时选择同客户的 `ACTIVE` 项目与 `EFFECTIVE` 合同。
- 已批准报价只能选择一次转换目标：合同/补充合同草稿，或销售订单草稿；同键重试返回同一目标，不允许生成重复业务事实。
- 继续保留销售订单直接新建入口。手工订单的价格来源固定为 `MANUAL`、报价来源为空；只有报价转单才要求报价来源有效。
- 已确认订单的客户、项目、合同、报价来源、物料和单位不可变；数量、税价和交期只能通过受控订单变更生效。
- 已过账出库行保存自己的税价和来源快照，后续订单变更不得反向改变既有出库或基础应收金额来源。
- 超量发货容差固定为 0。提前交付允许，但必须引用开放计划并填写原因；欠量结束必须通过短交关闭审批。
- 商业信用独立于正式财务。信用计算只读复用现有订单、出库和基础应收/收款业务台账，不创建发票、应收、收款、收入确认或凭证。
- 无信用档案视为信用额度 0；新订单确认或增加信用暴露的变更必须先维护信用档案或走信用例外审批。历史订单继续可查，不补造信用事实。
- 销售退货继续复用既有 reversal 入口。退货减少净履约并调整既有基础应收；不自动重开订单或重新制造有效需求，替换需求必须通过新订单或受控订单变更表达。
- 项目销售关闭是独立的履约维度，不替代 020 项目 `CLOSED`；项目正式关闭前必须先完成销售履约关闭。
- 销售履约已关闭的项目不得再被新报价转换、手工订单或订单确认引用；如需新增业务必须建立新的有效项目和合同关系。
- 025 只输出稳定的有效销售需求/履约只读结果；026 才负责缺料、齐套、净算和建议。

## 阶段目标

在 020 至 024 既有能力上建立以下完整闭环：

1. 销售报价草稿经固定审批成为可接受报价，并可幂等转换为合同/补充合同草稿或销售订单草稿。
2. 销售订单确认时形成客户、报价、合同、项目、税价、结算和信用的不可变商业快照。
3. 确认后的数量、价格、税率和交期变化通过可审批、可审计、不可覆盖历史的订单变更表达。
4. 每个订单行形成可拆分交付计划，销售出库引用计划并在统一库存过账事务内推进履约。
5. 客户信用档案、信用占用、逾期风险和例外审批形成可并发保护的确认门禁。
6. 销售订单和项目履约可按真实出库、退货、短交和关闭事实结案。
7. 026 可只读取得稳定、可筛选、带来源和不计入原因的有效销售需求。

## 业务对象、状态机与规则

### 销售报价

业务状态固定为：

| 状态 | 含义 | 允许操作 |
|---|---|---|
| `DRAFT` | 草稿商业意向 | 编辑、提交审批、取消 |
| `APPROVED` | 固定审批已通过且在有效期内 | 查看、打印、导出、选择一次转换目标、取消 |
| `CONVERTED` | 已转换为合同/补充合同或销售订单 | 查看和追溯，不允许再次转换 |
| `EXPIRED` | 已超过有效期 | 查看和追溯，不允许转换 |
| `CANCELLED` | 已取消 | 查看和追溯 |

- 审批实例状态与业务状态分离。审批中报价仍是 `DRAFT`，但后端 `allowedActions` 必须禁止编辑、重复提交和取消。
- 报价字段至少包含客户、可选项目、报价日期、有效期、交付承诺、币种 `CNY`、价格模式、默认税率、结算方式、账期、收款条件、备注和版本。
- 报价行包含行号、物料、单位、数量、未税单价、含税单价、税率、未税金额、税额、含税金额、承诺日期和备注。
- 所有数量、单价、税率和金额以后端十进制计算为准；接口使用十进制字符串，前端不得用浮点数形成可信结果。
- 普通报价不关联项目和合同。项目报价可只关联同客户的 `DRAFT` 或 `ACTIVE` 项目；合同不得脱离项目单独关联。
- 报价提交审批时锁定版本并保存审批摘要；审批最终通过时重新校验客户、项目、行、税价、有效期和版本。
- 超过有效期后即使后台状态尚未归档为 `EXPIRED`，转换动作也必须拒绝；后台归档只负责状态可见性，不是安全边界。
- `APPROVED` 报价仅允许一次转换：
  - 转合同：必须关联项目；用户选择 `MAIN` 或 `SUPPLEMENT`，补充合同仍按 020 校验主合同，生成 `DRAFT` 合同并保存报价来源。
  - 转订单：普通报价生成普通订单；项目报价必须在转换时选择同客户的 `ACTIVE` 项目和 `EFFECTIVE` 合同，生成 `DRAFT` 订单。
- 报价转换锁定报价、行、客户、项目和目标合同；转换目标写入报价，不能物理删除。

### 合同增补与合同来源快照

- 主合同和补充合同状态继续完全沿用 020，不新增平行合同版本状态机。
- 生效合同的项目、类型、主合同引用、客户、金额、签订日期和履约日期不可直接覆盖。
- 商业增减继续通过同项目补充合同表达；补充合同生效不自动改写既有销售订单或出库。
- 报价转合同后，合同保存 `sourceQuoteId`、`sourceQuoteNo`、报价版本、`CNY` 和含税商业金额快照。
- 订单确认时保存所选合同、主合同、当时已生效补充合同汇总、有效合同金额和合同版本快照；后续补充合同不反向改变该快照。
- 合同正常关闭前不得存在关联的开放订单、开放交付计划或待处理订单变更；提前终止继续按 020 原因和审计规则执行。

### 销售订单商业快照

- 订单状态继续沿用 `DRAFT`、`CONFIRMED`、`PARTIALLY_SHIPPED`、`SHIPPED`、`CLOSED`、`CANCELLED`。
- 普通订单的 `projectId`、`contractId` 同为空；项目订单二者同时存在并满足 020 同项目、同客户、`ACTIVE`/`EFFECTIVE` 规则。
- 继续允许直接新建销售订单。直接新建行的 `priceSourceType=MANUAL` 且报价来源为空；报价转单行保存 `priceSourceType=QUOTE` 和报价行来源。两类订单都必须完成税价、计划、信用、版本和项目合同校验。
- 草稿订单允许维护行与交付计划。确认后客户、项目、合同、报价来源、物料和单位不可直接修改。
- 订单确认前必须重新校验：客户启用、项目合同有效、行可销售、税价完整、交付计划覆盖、信用通过和版本一致；仅当订单或合同声明报价来源时才校验报价仍存在且来源链一致，手工订单不得被强制补造报价。
- 确认成功时创建不可变订单头/行商业快照，至少记录：
  - 客户、项目、合同、主合同和报价来源；
  - 客户开票名称、默认税率、结算方式、账期、收款条件，不保存完整税号、银行账号等非必要敏感值；
  - `CNY`、价格模式、行级未税/含税单价、税率、未税金额、税额、含税金额；
  - 合同有效金额、订单总未税金额、总税额和总含税金额；
  - 信用检查日志和审批例外引用。
- 同一订单允许同一物料因不同报价行、合同来源、税价或交期形成多行；唯一业务约束使用订单行号和可信来源组合，不再使用 `(order_id, material_id)` 阻断。
- 既有订单确认、取消、关闭统一要求 `version` 和 `idempotencyKey`；需要原因的动作同时要求 1 至 200 字原因。

### 受控订单变更

订单变更业务状态固定为：`DRAFT`、`APPLIED`、`CANCELLED`。审批中、审批通过、审批驳回和审批撤回全部由 022 审批实例状态表达，不写入订单变更业务状态。

- 变更只适用于 `CONFIRMED`、`PARTIALLY_SHIPPED` 或 `SHIPPED` 订单。
- 变更可调整既有行的目标数量、未交付部分税价和交期；不得改变客户、项目、合同、报价/合同来源、物料或单位，不在本阶段新增或删除订单行。
- 目标数量必须大于 0 且不得小于已过账出库数量；超量发货不能以变更后补录方式掩盖。
- 已过账出库行保存当时价格、税率、金额和来源快照；价格变更只影响后续出库与后续基础应收来源。
- 每个变更保存完整变更前/后快照、原因、提交版本、审批实例、申请人和应用时间。
- 审批中后端 `allowedActions` 禁止编辑、重复提交和取消；审批驳回或撤回后，变更仍为 `DRAFT`，可按最新版本修订后重新提交或取消。
- 信用未超限的变更走 `SALES_ORDER_CHANGE_APPROVAL`；增加信用暴露且检查不通过时走 `SALES_ORDER_CHANGE_CREDIT_OVERRIDE`，最终审批权限必须是信用例外审批权限。
- 审批最终通过后锁定订单、变更、计划和信用档案，重新校验并在同一事务内应用；任一条件变化则不应用并返回稳定冲突。
- 增加数量可使 `SHIPPED` 订单回到 `PARTIALLY_SHIPPED`；不得使 `CLOSED` 或 `CANCELLED` 订单重开。

### 交付计划、出库和退货

交付计划行状态固定为：

| 状态 | 含义 |
|---|---|
| `PLANNED` | 尚未出库 |
| `PARTIALLY_SHIPPED` | 已部分出库 |
| `SHIPPED` | 计划数量已全部出库 |
| `CLOSED` | 经正常或短交关闭后终止 |
| `CANCELLED` | 所属草稿订单取消前取消 |

- 一个订单行可按多个承诺日期拆分交付计划；新建 025 订单确认时各计划数量之和必须等于订单行数量。
- 未显式拆分时，后端以订单行预计发货日期生成一条草稿计划，仍需在确认时校验。
- 确认后的计划调整要求订单版本、计划版本、幂等键和原因；计划总量必须等于当前有效订单数量，不能小于已出库量。
- 新建 025 出库行必须引用开放交付计划；历史订单和历史出库可返回 `legacyDeliveryPlanCompatible=true`，不得伪造计划来源。
- 出库业务日期早于计划日期时必须填写提前交付原因；超计划或超订单数量一律拒绝。
- 出库过账锁定订单、订单行、计划、库存资源和幂等记录，在同一事务内完成统一库存过账、出库快照、计划进度、订单状态和审计。
- 已过账出库不可编辑或物理删除；错误修正继续走既有销售退货/反向链路。
- 已过账销售退货减少项目和订单的净交付数量，并通过既有 reversal 能力调整基础应收；不减少订单的历史已发数量，不自动增加有效销售需求。
- 如客户要求补发，必须创建新订单或受控增加原订单数量；不得由退货静默重开需求。

### 商业信用控制

信用档案至少包含：客户、信用额度、是否冻结、是否阻断逾期、复核日期、备注、版本和审计字段。

- 无信用档案时信用额度按 0 处理，订单确认被阻断；可先维护档案或提交信用例外。
- `usedCredit` 使用以下三部分的 CNY 含税净额相加，均由后端只读聚合且必须互斥：
  1. `orderCommitmentAmount`：已确认、部分发货订单未交付数量对应的含税金额；
  2. `unsettledShipmentAmount`：已过账出库但不存在非取消基础应收来源的出库行快照金额；仅关联 `CANCELLED` 应收的来源行按尚未形成有效基础应收继续计入本段；
  3. `receivableOutstandingAmount`：已存在非取消基础应收来源的未收净额，状态至少覆盖 `DRAFT`、`CONFIRMED`、`PARTIALLY_RECEIVED`、`CLOSED`；`RECEIVED` 未收额为 0，`CANCELLED` 不计入本段。
- 出库推进时订单承诺减少、待建应收出库增加；生成基础应收时待建出库减少、未收业务台账增加，任何转换不得重复占用或产生空窗。
- 生成 `DRAFT` 基础应收及 `fin_receivable_source` 的同一事务完成后，该出库行立即从第二段切换到第三段；不得等待应收确认才计入信用。
- 已过账收款和销售退货通过现有业务台账余额自然减少信用占用；025 不另建正式收款或贷项事实。
- `overdueRisk` 包含到期日早于当前业务日期且未收金额大于 0 的基础应收，以及超过客户账期仍未生成基础应收的已过账出库。
- 订单确认或增加未交付含税金额的订单变更必须锁定客户信用档案并重新计算；多个并发订单不得同时穿透额度。
- 客户停用、信用冻结、项目或合同无效不可通过信用例外覆盖。
- 超限、逾期或无额度可提交信用例外，快照至少记录额度、三类占用、候选增量、逾期明细摘要、原因、有效期、订单版本和审批人。
- 信用审批通过时必须重新计算；若暴露继续扩大，审批结论不得盲目应用，应返回稳定冲突或要求重新提交。
- 信用额度、占用、逾期和例外原因使用独立权限；无权限接口返回 `creditRestricted=true` 且敏感字段为 `null`，不得返回 0 伪装。

### 销售关闭与项目履约

- 已全部履约订单可直接正常关闭，但必须填写原因并满足：无草稿出库、无开放计划、无待处理订单变更、无待处理信用例外。
- 仍有有效未交付数量的订单只能提交固定审批场景 `SALES_ORDER_SHORT_CLOSE`；最终通过后在同一事务关闭余量、计划和库存预留，移出有效需求。
- 正常关闭和短交关闭都不得改写历史订单快照、已发数量、出库或退货事实。
- 项目增加销售履约维度 `OPEN`、`CLOSED`；销售履约关闭为终态，不替代项目自身状态。
- 项目销售履约关闭要求：
  - V27 后形成 025 履约事实的关联订单全部为 `CLOSED` 或 `CANCELLED`；
  - 所有交付计划终态；
  - 无待处理报价转换、订单变更、信用例外或短交审批；
  - 已过账退货已进入履约聚合，净交付差异有关闭原因；
  - 合同关闭/终止条件没有开放销售履约阻断。
- 020 项目正式关闭前必须先完成销售履约关闭。历史无 025 快照、状态为 `SHIPPED`、已全量出库且无开放退货/变更/信用例外的订单，可按兼容规则视作履约完成，但不补造 025 订单关闭记录。
- 项目履约汇总稳定展示合同有效金额、订单含税金额、计划数量、历史已发数量、退货数量、净交付、开放需求、逾期计划、信用风险和关闭阻断原因。

### 有效销售需求/履约视图

- 固定接口：`GET /api/admin/sales/effective-demands`，只读、可分页和筛选。
- 查询参数 `countedOnly` 默认 `true`，这是 026 的权威消费模式，只返回真实有效开放需求；`countedOnly=false` 返回计入与未计入候选，未计入行必须带稳定 `excludedReasonCode`，用于桌面诊断和验收排除规则。
- 行粒度固定为订单行与交付计划行；不得只返回订单头汇总供 026 猜测拆分。
- 至少返回：
  - `sourceType`、`sourceId`、`sourceNo`、`sourceVersion`、`sourceUpdatedAt`；
  - 订单、订单行、交付计划标识与单号；
  - 项目、客户、合同和报价来源；
  - 物料、单位、预留仓库；
  - 订单需求数量、计划数量、历史已发数量、已退数量、净交付数量、开放需求数量；
  - 承诺日期、计划状态、订单状态；
  - `countedAsEffectiveDemand` 和稳定 `excludedReasonCode`。
- 默认 `countedOnly=true` 只返回已确认、未关闭、未取消且仍有未交付计划数量的订单/计划。
- 报价、草稿、审批中、已关闭、已取消、已全量履约或短交关闭余量不计入。
- 退货本身不自动增加开放需求；补发必须有新订单或订单变更来源。
- 025 不返回库存短缺、齐套率、采购建议、生产建议或计划订单。

## 权限、审批、附件、文档任务与审计

### 权限

至少新增或冻结以下权限族：

- `sales:quote:view/create/update/submit/approve/cancel/convert`
- `sales:delivery-plan:view/manage`
- `sales:order-change:view/create/submit/cancel/approve`
- `sales:credit:view/manage/override-submit/override-approve`
- `sales:order:short-close-submit`
- `sales:order:short-close-approve`
- `sales:fulfillment:view/close`
- `sales:effective-demand:view`

继续复用：`sales:order:*`、`sales:shipment:*`、`sales:return:*`、`sales:project:*`、`sales:contract:*`、平台审批和文档任务权限。

- 后端权限是最终边界；前端隐藏或禁用不构成授权。
- 销售金额沿用对应销售对象查看权限；合同金额沿用合同权限；信用额度、占用、逾期和例外原因必须额外要求 `sales:credit:view`。
- 成本层、库存价值、内部价值流水继续要求 `inventory:valuation:view`，025 页面不得扩大。
- 无合同权限的项目履约响应不得泄露合同编号、金额或数量；无信用权限不得泄露信用字段。
- 文档任务创建、查询、下载必须同时校验平台权限和对应销售对象权限，响应不得暴露对象存储 key。

### 固定审批场景

- `SALES_QUOTE_APPROVAL`：销售报价审批；审批通过把业务状态从 `DRAFT` 置为 `APPROVED`。
- `SALES_ORDER_CHANGE_APPROVAL`：信用通过的订单变更审批；通过后应用变更。
- `SALES_ORDER_CHANGE_CREDIT_OVERRIDE`：增加暴露且信用不通过的订单变更；通过后重算信用并应用。
- `SALES_ORDER_CREDIT_OVERRIDE`：草稿订单信用例外；通过后重算并确认订单。
- `SALES_ORDER_SHORT_CLOSE`：欠量短交关闭；通过后关闭余量、计划和预留。
- 继续复用 `SALES_PROJECT_CONTRACT_ACTIVATION`，不得建立重复合同审批场景。
- 提交人不得审批本人申请；审批状态与业务状态分离；同对象同动作只允许一个活动实例。
- V27 固定审批步骤候选权限映射必须为：
  - `SALES_QUOTE_APPROVAL -> sales:quote:approve`
  - `SALES_ORDER_CHANGE_APPROVAL -> sales:order-change:approve`
  - `SALES_ORDER_CHANGE_CREDIT_OVERRIDE -> sales:credit:override-approve`
  - `SALES_ORDER_CREDIT_OVERRIDE -> sales:credit:override-approve`
  - `SALES_ORDER_SHORT_CLOSE -> sales:order:short-close-approve`
- 审批人除候选审批权限外还必须具备对应业务对象查看权限；无候选权限拒绝，提交人不可自批。
- 审批最终回调必须重新锁定对象并校验版本、状态、来源、信用和权限，失败不得部分应用。

### 附件、打印和导出

- 报价、订单变更和销售项目继续复用统一附件平台；类型、大小、病毒扫描、权限和审计沿用 022。
- 报价提供固定打印模板和导出任务；交付计划与有效需求提供导出任务；不新建同步大文件下载接口。
- 文档任务类型和对象形状固定为：
  - `SALES_QUOTE_PRINT`：通过 `/api/admin/print-tasks` 创建，`objectType=SALES_QUOTE`、`objectId=quoteId`；
  - `SALES_QUOTE_EXPORT`：通过 `/api/admin/export-tasks` 创建，保存报价列表筛选快照；
  - `SALES_DELIVERY_PLAN_EXPORT`：通过 `/api/admin/export-tasks` 创建，保存交付计划全局列表筛选快照；
  - `SALES_EFFECTIVE_DEMAND_EXPORT`：通过 `/api/admin/export-tasks` 创建，保存 `countedOnly` 等有效需求筛选快照。
- 任务详情、轮询、失败明细、下载和过期继续统一走 `/api/admin/document-tasks`。
- 文档任务返回业务可读错误、失败明细和过期状态，不返回内部异常堆栈或存储 key。
- 真实桌面验收不截图、不保存视觉证据目录。

### 审计

- 覆盖报价创建/编辑/提交/批准/取消/失效/转换，合同来源写入，订单确认/取消/变更/关闭，计划调整，信用档案维护/检查/例外，出库过账，项目销售履约关闭，附件、打印、导出和权限拒绝。
- 审计摘要保存业务单号、版本、动作、原因、关键金额/数量变化和来源摘要；不得明文写完整敏感结算资料。
- 历史快照保存在业务表，不得只依赖非结构化审计日志恢复。

### 写动作通用契约

- 所有更新和副作用动作请求必须带当前 `version`。
- 所有副作用动作必须带非空 `idempotencyKey`；同键同请求返回原业务结果，同键不同请求返回 `409`。
- 幂等命中判断优先于版本冲突；已成功同键重试不能因当前版本变化先报陈旧版本。
- 原因字段长度统一 1 至 200；备注最长 500。
- `409` 后前端不得自动重放不同请求，必须重新获取详情并让用户确认。
- 复杂对象响应必须由后端返回 `allowedActions` 和受限标志；前端不得复制状态机决定可操作性。
- 所有渲染行级副作用动作的列表摘要必须返回 `allowedActions`、`actionDisabledReason` 和对应受限标志；未返回行级动作的列表只能查看或进入详情，不得在列表展示确认、转换、提交审批、关闭等副作用动作。

## 数据模型、V27 迁移与历史兼容

### 主要模型

必须新增：

- `sal_sales_quote`、`sal_sales_quote_line`
- `sal_sales_order_snapshot`、`sal_sales_order_line_snapshot`
- `sal_sales_order_change`、`sal_sales_order_change_line`
- `sal_sales_delivery_plan`
- `sal_customer_credit_profile`
- `sal_credit_check_log`
- `sal_action_idempotency`

必须扩展：

- `sal_project_contract`：报价来源、`CNY` 和商业金额口径快照。
- `sal_sales_order`：来源报价、汇总税价、信用检查、关闭原因、销售履约兼容标志。
- `sal_sales_order_line`：报价行/合同来源、价格来源、`CNY`、税率、未税/含税单价与金额。
- `sal_sales_shipment_line`：交付计划、出库时税价/来源快照。
- `sal_project`：销售履约状态、关闭人、时间、原因和版本影响。
- 文档平台对象类型、审批场景、权限菜单、打印模板和导出类型种子。

### 数据库约束

- 状态、类型、币种和价格模式使用 check 约束。
- 数量大于 0；税率 0 至 100；金额非负；含税、未税和税额恒等式由后端高精度计算并做数据库可表达约束。
- 报价号、变更号和审批活动业务键唯一。
- 删除 `sal_sales_order_line(order_id, material_id)` 唯一约束，保留 `(order_id, line_no)` 唯一；新增可信来源组合索引，允许同物料多来源、多税价和多交期。
- 信用档案每客户唯一；销售动作幂等键按操作人、动作、资源和键唯一，并保存请求指纹。
- 计划必须引用订单行；出库计划引用允许历史空值，但新 025 写路径必须强校验非空。
- 常用索引覆盖报价客户/项目/状态/有效期、订单来源、计划日期/状态、变更订单/状态、信用客户、有效需求项目/物料/日期。

### 历史兼容

- V27 必须支持空库从 V1 至 V27，以及代表性 V26 存量库升级。
- 历史订单和订单行默认 `currency=CNY`、`taxRate=0`、未税价等于含税价、价格来源为 `LEGACY_MANUAL`；标记为兼容快照，不伪称来自 021 税率或报价。
- 历史已过账出库行按当前能确认的订单行单价回填兼容税价快照并标记 `legacySnapshot=true`；不得改写库存流水或基础应收来源。
- 基础应收生成必须从销售出库行税价快照读取来源金额；`FinanceAdminService` 不得继续用可能已被订单变更更新的订单行当前价格计算已过账出库金额。历史出库使用迁移后的兼容快照。
- 不为历史报价、订单变更、信用检查或交付计划伪造业务记录。
- 历史未关闭订单可使用订单/行预计发货日期进入兼容履约读取，返回 `legacyDeliveryPlanCompatible=true`；用户后续显式初始化计划后才形成 025 计划事实。
- 历史无项目订单保持 `projectId/contractId` 同为空，继续支持列表、详情、出库、退货、财务台账和报表。

## 并发、幂等和事务边界

- 各动作可先无锁读取父级标识以确定目标，但持有行级锁后必须严格按以下顺序：幂等记录占位 -> 客户/信用档案 -> 项目/合同 -> 报价 -> 订单 -> 订单行/计划/变更 -> 出库 -> 库存资源；各服务不得反向加锁。
- 报价转换、出库过账、应收来源读取和退货履约聚合不得持有报价、出库或退货锁后再反向锁客户、合同或订单。
- 报价转换在单事务完成报价锁定、目标创建、来源写入、状态更新、幂等结果和审计。
- 订单确认在单事务完成来源校验、商业快照、信用日志、状态、库存预留、幂等结果和审计。
- 订单变更最终批准在单事务完成审批终态校验、信用重算、订单/行/计划修改、快照、状态、幂等和审计。
- 出库过账在单事务完成版本与幂等校验、计划数量、统一库存过账、出库快照、订单/计划状态和审计。
- 正常/短交关闭在单事务完成阻断校验、计划关闭、预留释放、订单状态、有效需求移除和审计。
- 任一步失败必须整体回滚；不得出现审批已通过但业务未应用、出库已过账但计划未更新、订单已确认但信用日志缺失等部分状态。

## 验收矩阵

### 主路径

- 普通报价审批后幂等转普通订单，确认形成商业快照和计划，出库、收款台账信用转换、正常关闭完整可追溯。
- 项目报价关联项目，审批后转主/补充合同草稿或在合同生效后转项目订单，项目身份和来源不丢失。
- 项目订单确认、部分出库、继续出库、退货冲减、订单关闭、项目销售履约关闭和项目正式关闭可执行。
- 同一物料不同报价来源、税价或交期可形成多行并独立履约。
- 订单变更审批后只修改未交付部分，既有出库和基础应收金额来源不变。
- 交付计划拆分后出库引用正确计划，计划与订单状态同步。
- 信用额度内直接确认；超限/逾期经例外审批后重算并确认。
- 有效需求默认 `countedOnly=true` 只返回真实开放订单/计划行，026 按该模式消费；`countedOnly=false` 可验证每类未计入候选的稳定排除原因。

### 异常路径

- 报价未审批、已过期、已取消或已转换时再次转换被拒绝。
- 报价、项目、合同、订单客户不一致或项目/合同状态无效时被拒绝。
- 项目订单缺少项目或合同、只填一个或跨项目错绑被拒绝。
- 订单变更低于已发数量、改客户/项目/合同/来源/物料/单位或用于已关闭订单被拒绝。
- 计划合计不等于订单数量、低于已发、跨订单行引用或状态无效被拒绝。
- 出库缺少计划、超计划、超订单、提前交付无原因或已关闭后继续出库被拒绝。
- 信用档案缺失、冻结、超限、逾期或客户停用按规则阻断；不可覆盖条件不能审批绕过。
- 重复幂等同键异载荷、陈旧版本、并发信用占用和审批回调状态变化返回稳定 409 且无部分写入。
- 正常关闭有开放计划，短交关闭无审批，项目销售关闭有开放对象时被拒绝。

### 权限和脱敏

- 无报价、计划、变更、信用、项目履约或有效需求权限时接口 403，路由不可进入。
- 无信用查看权限时额度、三类占用、逾期和例外原因为 `null`，显示受限，不返回 0。
- 无合同权限时项目履约不泄露合同编号、金额或数量。
- 无估值权限时不泄露成本层、价值流水或成本金额；普通出库方向、仓库、物料、数量仍可见。
- 无对象业务权限或文档平台权限时不能创建、查询或下载对应任务；不泄露存储 key。

### 数据、迁移和并发

- V1 空库应用 V27 成功；V26 存量升级成功，约束、索引、权限、审批和文档类型存在。
- 历史普通/项目订单、同物料订单、出库、退货、应收、收款、库存和报表继续可查询。
- 历史订单只返回兼容税价/计划标志，不伪造报价、信用、变更或计划事实。
- 两个并发订单确认不能穿透同一客户信用额度。
- 同一报价转换、出库过账、审批回调、短交关闭同键重试只产生一个业务事实。
- 出库过账失败不得留下计划已发但库存未扣，或库存已扣但出库/计划未完成。

## 风险与处理

- 信用被误解为正式应收：响应和页面明确标记“商业信用”，使用只读基础业务台账，不生成正式财务事实。
- 信用转换重复或空窗：三段暴露互斥聚合，并用出库行与应收来源关联防重。
- 订单变更污染历史出库：出库行保存独立税价/来源快照，变更只影响未来未交付部分。
- 退货自动重开错误需求：履约显示净交付，但有效需求只认订单/变更，退货不直接加需求。
- 同物料多来源被合并：移除旧物料唯一约束，以行号和来源组合保证合法拆分。
- 项目/合同关闭顺序冲突：销售履约维度先关闭，再按 020 完成合同和项目终态门禁。
- 现有大服务继续膨胀：报价、信用、履约使用独立服务包，只在既有销售服务保留必要兼容入口。
- 前端浮点污染：025 DTO 使用十进制字符串，后端计算并返回；前端只格式化展示。

## 缺陷与阻断

### 阻断缺陷

- 报价、订单、计划、出库、退货、关闭主闭环不可执行。
- 项目订单可跨客户/项目/合同错绑或项目身份在转换中丢失。
- 信用缺失、冻结、超限或逾期可无审批绕过，或并发确认穿透额度。
- 已确认/已过账来源、价格、数量或库存事实可直接改写。
