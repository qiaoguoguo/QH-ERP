### 方案一：独立业务月结领域、原子关闭、不可变多版本快照（采用）

- 016 继续只表达业务期间 `OPEN/LOCKED`，030 独立表达检查和月结状态。
- 每次关闭周期绑定一个业务期间和递增版本；预检查可以反复执行，关闭必须在事务内重检。
- 关闭成功时一次性写入快照、锁定期间和完成审计；任何一步失败全部回滚。
- 重开不删除、不覆盖旧快照；新业务事实落入重新开放后的期间，再次月结生成新版本。
- 优点：领域边界清楚，历史稳定，可并发验证，可为 031/032 提供可信但非财务的业务基线。

### 方案二：只增强 016 锁定并附加检查字段（不采用）

- 优点：表和页面改动较少。
- 不采用原因：无法稳定回答月末库存、在制、项目成本和报表当时值；期间保护与月结快照耦合，后续正式财务阶段难以演进。

### 方案三：查询时实时重算或只冻结报表而不锁期（不采用）

- 优点：持久化模型较少。
- 不采用原因：上游新增、重算、重开和跨期反向会改变历史结果；快照与可继续写入的期间会分叉，不满足历史稳定与交付目标。

### 方案四：直接建设财务关账中心（不采用）

- 不采用原因：会引入会计期间、正式凭证、总账、收入确认、损益结转和反关账，静默扩大到 031/032。

## 负责人冻结裁决

1. 菜单使用独立一级业务入口“业务月结”，主路由为 `/period-close/runs`；016 的 `/system/business-periods` 保留基础期间维护职责，只增加月结摘要和有权跳转。
2. 状态固定为 `PENDING_CHECK`、`BLOCKED`、`READY`、`CLOSED`、`REOPENED`。检查为同步动作，不持久化长期 `CHECKING/CLOSING` 业务状态；页面用提交态表达处理中。
3. 不提供单独“生成快照”或“取消运行”动作。快照仅在关闭事务中生成，避免存在已快照但未锁期的中间事实。
4. 关闭必须同时锁定 016 业务期间；已手工锁定但没有 030 快照的历史期间展示“已手工锁定/无月结快照”，不得自动补认。需要正式月结时先按 016 权限和审计解锁，再从 030 检查并关闭。
5. 阻断项永不允许放行；警告项允许关闭，但必须勾选确认并填写原因。关闭和重开均是高风险动作，原因必填。
6. 在制、合法暂估、未完成订单和首次无上期快照只产生警告；未估值、未定价、来源断链、项目串用、开放阻断差异、陈旧项目成本和核心对账不平必须阻断。
7. 期间内有项目成本来源活动的项目必须存在截止日精确等于期间结束日的最新有效 `CALCULATED` 或 `CONFIRMED` 运行，且来源为 `CURRENT`、无开放 `BLOCKING` 差异。`INCOMPLETE` 仅因 WIP 或合法暂估可警告通过；其他原因阻断。
8. 首期没有上一快照时允许从既有不可变业务事实生成，记录提示；不做历史批量补快照。
9. 每个期间允许多个历史版本，但同一时刻最多一个 `CLOSED` 当前版本。重开将当前版本置为 `REOPENED`，再检查创建下一版本。
10. 冻结报表基线覆盖现有八类固定经营报表：经营概览、销售汇总、采购汇总、库存收发存、生产执行、成本归集、往来结算、异常清单。基线只在 030 快照详情展示，不改造现有报表页为“实时/快照”双模式，不提供导出。
11. 金额权限不新增一个可绕开原域的总权限：库存金额继续要求 `inventory:valuation:view`，项目成本/毛利继续要求 `cost:project-cost:amount-view`，报表来源继续要求对应报表及来源权限；月结权限只控制月结动作和快照入口。

## 业务状态机

```text
无运行 --发起检查--> PENDING_CHECK --检查有阻断--> BLOCKED
                                  \--无阻断--> READY
BLOCKED/READY --重新检查--> BLOCKED 或 READY
READY --原子复检通过并关闭--> CLOSED
CLOSED --填写原因并重开--> REOPENED
REOPENED --发起新版本检查--> 新版本 PENDING_CHECK --> BLOCKED 或 READY
```

- `PENDING_CHECK` 只表示已建立本版本但尚无成功检查结果；同步检查过程中前端显示加载态。
- `BLOCKED` 保存最近一次不可变检查运行，关闭动作不在 `allowedActions` 中。
- `READY` 表示最近检查无阻断，不代表可以绕过关闭时复检。
- `CLOSED` 必须同时满足当前版本唯一、快照完整、期间 `LOCKED`、关闭审计存在。
- `REOPENED` 的旧快照仍可查看，明确标记历史版本，不再是当前关闭版本。
- 禁止删除运行、检查或快照；数据修正必须通过开放期间业务或重开后新版本处理。

## 快照口径

### 库存快照

- 截止日为期间结束日，维度至少覆盖仓库、物料、质量状态、所有权、项目、批次、序列和项目成本层。
- 保存期末数量、锁定数量、可用数量、估值状态、单位成本、期末价值，以及期间入库、出库、调整和反向数量/金额汇总。
- 另存公共计价池和项目成本层汇总、价值流水数量/金额对账摘要、未估值/暂估计数及分区指纹。
- 不重新估值、不修改价值流水、不改变公共池或项目成本层。

### 在制/生产快照

- 按项目、工单、物料和生产阶段冻结截止日的计划、已领、已报、已完工、已交付、在制数量及对应成本摘要。
- 在制存在是合法月末状态，只产生警告；来源断链、负数、项目串用或无计价依据仍阻断。

### 项目成本快照

- 引用截止日精确匹配的最新有效 029 核算运行和来源指纹，不创建、重算、确认或取消 029 运行。
- 保存项目、核算运行、当前性、完整性、成本分类、WIP/完工/交付/项目直接阶段、发货/开票/目标经营收入、经营毛利和差异摘要。
- 030 展示的收入和毛利继续是 029 经营口径，不得称为正式收入或财务利润。

### 固定经营报表基线

- `OVERVIEW`、`SALES_SUMMARY`、`PROCUREMENT_SUMMARY`、`INVENTORY_STOCK_FLOW`、`PRODUCTION_EXECUTION`、`COST_COLLECTION`、`SETTLEMENT_SUMMARY`、`EXCEPTIONS` 八类报表均保存 `report_code`、`schema_version`、规范化 JSONB 结果、来源计数和分区指纹。
- 由后端服务层复用现有报表口径生成，不通过 HTTP 自调用；期间起止日期强制使用目标业务期间。
- 只在 030 快照详情展示冻结基线；现有 `/api/admin/reports/*` 和报表页面继续表示实时查询。

### 不可变与版本

- 快照主表、明细和报表基线在数据库层和服务层均禁止更新、删除和覆盖；状态变化只发生在月结运行。
- `period_id + revision_no` 唯一；同一期间最多一个 `CLOSED` 当前版本。
- 每个快照保存 `schema_version`、总来源指纹、分区指纹、生成时间、操作者和来源检查运行。
- 重开不标记快照“失效”，统一展示为“历史版本/已重开”；旧版本仍是当时真实冻结基线。

## 并发、事务、幂等和审计

- `BusinessPeriodGuard.assertWritable` 与月结关闭必须使用同一期间级 PostgreSQL transaction advisory lock，并在同一事务内读取期间状态，消除“写入已通过检查、随后月结提交”的竞态。
- 关闭事务顺序固定为：获取 advisory lock → `SELECT ... FOR UPDATE` 期间和当前月结版本 → 校验版本/幂等 → 重跑检查和指纹 → 写全部快照 → 更新月结为 `CLOSED` → 更新期间为 `LOCKED` → 写月结与期间审计 → 提交。
- 重开事务顺序固定为：获取同一锁 → 锁定期间和当前月结 → 校验权限/版本/原因/幂等 → 更新月结为 `REOPENED` → 更新期间为 `OPEN` → 写双侧审计 → 提交。
- 同一操作者、动作和幂等键重复提交相同指纹返回原结果；相同键不同指纹返回 409；陈旧版本、来源变化、重复当前关闭和并发重开返回稳定 409。
- 检查、关闭、重开、阻断结果、警告确认、来源指纹冲突、幂等重放和金额受限查询均可审计。无金额权限的审计响应必须脱敏。
- 服务异常、快照任一分区失败或期间锁定失败时，不得留下半套快照、`CLOSED` 状态或已锁期间。

### 迁移与历史兼容

- V1→V32 空库迁移成功；正式 V31 副本→V32 成功；最新成功版本为 V32，失败迁移 0。

- 既有 `LOCKED` 期间不自动成为 030 `CLOSED`；页面和 API显示“已手工锁定/无月结快照”。

### 状态、检查与原子性

- 新期间可检查为 `BLOCKED` 或 `READY`；阻断永不可关闭；警告只有明确确认和原因后可关闭。
- 关闭时来源变化、版本冲突、并发业务写入或任一快照失败，不产生半快照、不锁期、不标记 `CLOSED`。
- 成功关闭后月结为 `CLOSED`、期间为 `LOCKED`、所有分区快照完整、审计存在；同期间业务写入继续由 016 拒绝，查询与追溯可用。
- 重开原因必填；成功后月结为 `REOPENED`、期间为 `OPEN`、旧快照不变；再次关闭产生新版本。
- 相同幂等键相同请求返回原结果；相同键不同请求、陈旧版本、重复当前关闭稳定返回 409。

### 数据一致性与不回写

- 库存快照与截止日库存余额、数量流水、不可变价值流水、公共池、项目层和分配合计一致。
- 在制/生产快照与工单领料、报工、完工、交付和项目成本阶段一致。

- 八类报表冻结结果与相同期间的服务层来源合计和来源计数一致，且明确不是正式财务报表。
- 快照生成后修改开放期来源、029 新建/重算、报表实时结果变化均不改变旧快照；030 不回写 023/028/029 事实。
