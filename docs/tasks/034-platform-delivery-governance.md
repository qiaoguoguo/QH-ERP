## 五角色目标讨论与采用结论

2026-07-21，当前交接登记的五个固定角色在权威基线完成唯一一轮并行只读讨论，均核对了工作区、分支前 detached HEAD、基准提交和唯一交接差异。五方一致认为：

1. 034 必须复用 022 平台底座，不能建设第二个平台域、第二套任务状态机或通用治理引擎。
2. 只增强文档任务中心无法清晰表达数据修复申请—审批—执行—验证和历史导入批次生命周期；需要新增有限治理页面族。
3. `PlatformDocumentTaskService` 已超过 3000 行，034 新能力必须通过固定处理器/适配器隔离，不能继续堆叠任务类型分支。
4. 历史导入首批应收敛在主数据和草稿基础资料；库存、成本、凭证、账簿、月结、税务等不可变事实不得直灌。
5. 页面逐页遵守 `page-standards.md`；桌面真实浏览器验收不截图、不测试移动端，响应式仍是开发约束。

### 方案比较

#### 方案一：只增强现有文档任务中心

- 优点：改动最少。
- 不采用原因：数据修复记录、历史导入批次、审批、执行、验证和不可变审计会混入普通任务列表，业务主语不清；巨型服务继续膨胀。

#### 方案二：现有平台域内新增固定治理适配器和有限页面族（采用）

- 复用审批、附件、打印、任务、导入暂存、错误明细、对象存储、消息和审计。
- 新增数据修复记录、历史数据导入、交付资料页面族，增强文档任务中心和业务对象固定打印入口。
- 后端以固定适配器和任务处理器注册替代新增大量 `if taskType`。
- 优点：范围完整、数据边界清晰、可维护、可真实验收，不引入第二平台。

#### 方案三：通用治理平台、脚本执行器或万能导入

- 优点：表面扩展性高。
- 不采用原因：引入任意脚本、动态流程、模板设计器或自由对象映射会绕过领域服务，扩大权限和数据污染风险，并越界进入第二平台域。

## 数据修复契约

### 首批固定修复适配器

| 代码 | 对象 | 允许修复 | 禁止修复 |
|---|---|---|---|
| `MATERIAL_PROFILE_CORRECTION_V1` | 物料 | 名称、规格、备注等非标识描述字段 | 编码、追踪方式、计价/成本关键属性、单位和已引用关系 |
| `CUSTOMER_PROFILE_CORRECTION_V1` | 客户 | 名称简称、联系人、联系方式、备注等白名单资料 | 编码、税务/结算关键标识、已确认交易事实 |
| `SUPPLIER_PROFILE_CORRECTION_V1` | 供应商 | 名称简称、联系人、联系方式、备注等白名单资料 | 编码、税务/结算关键标识、已确认交易事实 |

- 普通可编辑对象仍优先走原编辑页；数据修复只用于历史引用后需要留下审批、证据和验证链的例外更正。
- 每个请求只能使用一个固定适配器，明确目标对象、原因、证据、预期新值和当前对象版本；不得通过 JSON 提交未登记字段。
- 预检固化对象版本、修复前摘要、修复后摘要、影响检查和请求指纹。执行前重新锁定对象并复检，版本或业务状态变化即失败，不使用旧预检覆盖。

### 状态机和职责分离

状态固定为：

`DRAFT -> PENDING_APPROVAL -> READY_TO_EXECUTE -> EXECUTING -> EXECUTED -> VERIFIED`

分支终态为 `REJECTED`、`CANCELLED`、`FAILED`、`VERIFY_FAILED`。终态不得复活；需要再次处理时创建引用原请求的新请求，不覆盖原记录。

- 草稿由申请人创建和维护；提交后字段、目标、前后摘要和附件快照不可修改。
- 复用固定审批场景 `PLATFORM_DATA_REPAIR_EXECUTION`。申请人不得审批自己提交的修复。
- 执行人必须具备平台执行权限和目标对象原维护权限；审批通过不替代业务权限。
- 验证人不得是执行人。允许小公司采用两人闭环：甲申请并执行，乙审批并验证；禁止单人完成申请、审批、执行和验证。
- 执行后不得删除或回滚事件。发现结果不正确时，以新修复请求追加更正链。

### 幂等、并发和审计

- 创建、提交审批、执行、验证和取消都要求 `Idempotency-Key` 与版本。
- 指纹至少包含主体、动作、适配器、对象类型/ID、对象版本、附件哈希和请求正文摘要。
- 同键同指纹返回既有结果；同键不同指纹返回 `DOCUMENT_TASK_IDEMPOTENCY_CONFLICT` 或 034 对应稳定冲突码。
- 执行未知结果不得盲目重试；进入 `FAILED` 并要求验证数据库与领域事实后创建新请求。
- 专用不可变事件记录动作、主体、时间、前后摘要、审批实例、任务、结果和验证；同步向 `sys_audit_log.detail_json` 写脱敏摘要，不记录完整文件或敏感原值。

### 固定流程

1. 下载适配器当前版本模板。
2. 上传源文件，后端计算 SHA-256 并创建隔离的导入校验任务。
3. 全文件完成结构、字段、引用、业务、权限和重复检查；错误保存行号、列名、稳定错误码和中文建议。
4. 无错误进入 `READY_TO_COMMIT`，页面展示汇总、差异和关联任务。
5. 用户携带任务版本和独立幂等键确认；后端重新校验权限、模板版本、对象引用和文件指纹。
6. 通过既有业务服务在单一业务事务中全有或全无提交；任一行失败整体回滚并形成错误明细。
7. 结果、错误文件和源文件默认保留 7 天；过期后拒绝下载但保留元数据、摘要和审计。

- 取消只允许 `QUEUED`、`READY_TO_COMMIT` 等安全状态；运行中的提交不可强制取消。
- 校验任务可按租约恢复和最多重试三次；业务提交只有确认事务整体回滚时才允许重试。
- 同一用户、适配器、幂等键和同文件哈希返回既有任务；同键不同文件或模板版本冲突。

- 状态批量变更在确认前重新校验对象版本、业务状态、领域权限和开放引用；混入一项无权限、陈旧或不可执行对象时整个批次不提交。
- 批量打印可以为每个对象形成独立子结果，但父操作必须明确成功、部分生成失败或整体失败；业务事实不受打印子项失败影响。
- 不新增独立“批量工具中心”；固定批量入口位于对应主数据列表和交付资料目录，执行结果统一进入文档任务中心。

## 阻断和严重问题

以下任一项存在时不得进入交付：

- 默认正式 V34 数据库或 `qherp-private` 被 V35/V36 隐式前迁、写入、清理或测试污染。
- 修改既有迁移或 checksum，V1/V34/V35→V36 失败，或数据库存在失败迁移。
- 修复支持任意 SQL/字段、可静默修改、可单人闭环、自批、自验、陈旧覆盖、终态复活或删除审计。
- 历史导入绕过预检/确认/业务服务、出现部分业务提交、重复事实或错误明细泄露。
- 批量状态操作混入无权限/陈旧/不可执行项仍部分提交。
- 平台权限替代领域权限，文件、错误、预览、PDF 或下载泄露未授权业务或敏感字段。
- worker 重复执行产生重复事实，取消/过期后仍写业务，未知结果被盲目重放。
- MinIO 与数据库可用文件不一致，失败却标记任务成功，模板没有版本或新模板污染旧任务。
- 核心页面不可访问、关键状态不可识别、操作不可达、页面级溢出、严重控件重叠或控制台错误。
- 操作手册依赖不存在截图，或把 035/036、排除项写成已交付。
