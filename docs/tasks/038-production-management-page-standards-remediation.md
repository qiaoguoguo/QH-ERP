## 阶段目标

在不新增业务能力、不改变现有业务契约的前提下，闭合固定 UI 设计师判定的 12 个生产管理页面规范缺口：

4. 修正新建退料、新建补料候选池、明细表和底部操作区的密度、列宽、滚动反馈和可达性。
5. 由固定 UI 设计师使用与修复前一致的 1280×720 页面状态逐项复验，生成修复后截图并做前后对照；由固定测试角色完成代码评审和真实桌面路径复核。

## 页面规范冻结决策

### 页面结构

- 详情页采用生产工单详情现有的摘要区、标签化字段网格和分区表格模式；不复制生产工单业务能力。
- 新增/编辑外协单采用分组双列或紧凑栅格，长备注、警告和不可编辑原因可整行；日期仍使用原日期选择器。
- 外协发料/收货沿用现有权限和动作判断；无写权限或不可操作时，所有录入控件必须呈 disabled/read-only 视觉，原提示和返回入口保留。
- 退料/补料追溯必须由抽屉或弹层承载；查询参数、方向、来源受限、资源跳转和数据解释不变。

### 宽表与操作列

- 页面主体不得出现横向滚动；候选池、明细、库存影响、成本影响、追溯表格均在自己的 `.table-scroll` 或等价容器内横向滚动。
- 右侧关键列不能被页面边界裁切，必须有明确滚动反馈；批次、序列、质量状态、可退/可补数量、库存与成本影响不得为换取布局而隐藏。
- 外协执行列表操作列继续执行用户最新统一契约：`fixed="right" width="184"`，最多两个动作直显，超出动作进入“更多”。即使当前仅有“查看”，也不得缩小或取消统一操作列宽；通过列顺序和业务列最小宽度改善首屏扫读。

### 截图与视觉判定

- 固定 UI 设计师是 12 个 B 项视觉闭合的唯一角色判断者；主代理、前端和测试不得替代其结论。
- 修复后截图必须由 UI 角色在当前候选代码真实运行页、1280×720、与修复前相同或等价数据状态下采集，保存后逐张重新打开检查。
- 每个 B 项至少有一张修复后证据；详情下半屏、长表单底部、追溯打开态等需要多状态才能证明闭合时必须补图。
- 截图不得包含密码、令牌或其他敏感信息；不得使用旧截图、源码或单测替代视觉证据。

## 接口、权限与数据不变契约

- 后端参与结论：无实现工作包；现有接口字段足以支持本阶段展示整改。
- 外协订单状态码保持 `DRAFT/RELEASED/IN_PROGRESS/COMPLETED/CLOSED/CANCELLED`，执行单据保持 `DRAFT/POSTED/CANCELLED`；只调整确定性中文展示，不改技术值。
- 路由权限、`allowedActions`、`actionDisabledReason`、`costVisible` 和受限字段仍是最终展示与可操作性依据。
- 退料/补料追溯继续使用现有 `sourceType/sourceId/sourceLineId/direction` 查询和 `restricted/canViewResource/restrictedMessage` 语义。
- 创建、编辑和状态动作请求的路径、方法、查询参数、请求体、`version`、`idempotencyKey`、来源行、成本层和跟踪分配字段不变。

- 追溯仍作为详情页底部长面板，或内部右侧字段/关闭动作不可达。
