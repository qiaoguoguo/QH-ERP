# 财务往来基础测试计划

## 目标

验证财务往来基础模块满足功能、接口、权限、异常、金额一致性、来源追溯、审计、部署和浏览器视觉验收要求，确保已过账销售出库能够生成可信应收并登记收款，已过账采购入库能够生成可信应付并登记付款。

本计划不验收总账、凭证、会计科目、会计期间、结账、税务申报、完整发票生命周期、银行对账、报销、固定资产、工资、多币种、多组织、多公司、多账套、多租户、正式收入确认、正式成本结转、销售退货、采购退货、退款、退票、BI 报表或正式财务核算能力。

## 准入标准

- 财务往来任务文档、设计规格、接口契约、权限编码和验收路径已确定。
- 采购管理基础模块、销售管理基础模块、库存流水和来源追溯能力在当前分支可用。
- 本地 PostgreSQL、后端和前端启动方式可用。
- 已确认应收来源为已过账销售出库，应付来源为已过账采购入库。
- 已确认财务往来阶段不做总账、凭证、税务、完整发票、银行对账、正式核算或报表看板。

## 准出标准

- 后端 Maven/Testcontainers 全量测试通过。
- 前端 `npm test`、`npm run typecheck`、`npm run build` 通过。
- 管理员或财务角色浏览器主路径通过。
- 销售角色、采购角色、只读用户、无权限用户和未登录用户权限验收通过。
- 未登录和越权接口验证通过。
- 异常路径验证通过，失败操作不产生部分写入。
- 应收、收款、应付、付款金额余额和状态一致。
- 来源追溯从财务往来台账回到采购入库、采购订单、销售出库和销售订单。
- 视觉分析截图与结论已保存到 `docs/testing/financial-settlement-visual-audit/`。
- `git diff --check` 通过。

## 最小验收闭环

本阶段最小可准出的业务闭环为：准备一笔已过账销售出库和一笔已过账采购入库；管理员或财务角色基于销售出库生成应收草稿并确认；登记第一笔部分收款并过账，确认应收状态为部分收款、已收金额和未收金额正确；登记第二笔收款并过账，确认应收已收清；从应收详情追溯到来源销售出库和销售订单；基于采购入库生成应付草稿并确认；登记第一笔部分付款并过账，确认应付状态为部分付款、已付金额和未付金额正确；登记第二笔付款并过账，确认应付已付清；从应付详情追溯到来源采购入库和采购订单。

| 管理员 | 完成应收、收款、应付、付款、来源追溯和审计主路径 |
| 财务角色 | 验证财务往来全量业务操作 |
| 销售角色 | 验证销售来源应收和收款进度可查看，不得登记或过账收款 |
| 采购角色 | 验证采购来源应付和付款进度可查看，不得登记或过账付款 |
| 只读用户 | 验证财务往来列表、详情和追溯可查看，写按钮不可见，写接口拒绝 |
| 无权限用户 | 验证无财务往来菜单入口、路由受限和接口拒绝 |
| 未登录用户 | 验证认证拦截 |

## 自动化验证范围

### 后端接口测试

- 未登录访问财务接口返回 `AUTH_UNAUTHORIZED`。
- 无对应权限访问财务查询和写接口返回 `AUTH_FORBIDDEN`。
- 写接口缺少 CSRF token 被拒绝。
- 应收列表按关键词、客户、状态、业务日期、到期日期和来源单号筛选分页。
- 应收详情返回基础信息、来源销售订单、来源销售出库、来源明细、收款记录和审计摘要。
- 基于已过账销售出库生成应收草稿成功，应收金额由后端计算。
- 未过账销售出库生成应收失败。
- 同一销售出库明细重复生成应收失败。
- 同一销售出库任一来源明细已生成应收时，再次按来源单据头生成必须整单失败，应收主表和来源明细均不新增。
- 更新应收草稿只能修改到期日期和备注。
- 确认应收成功，状态从 `DRAFT` 到 `CONFIRMED`。
- 已发生收款的应收不可取消。
- 应收关闭后不可继续收款。
- 创建收款草稿成功，不更新应收余额。
- 更新收款草稿成功。
- 收款过账成功，核销明细写入，应收已收金额、未收金额和状态正确。
- 本阶段一笔收款只核销一笔应收，收款金额必须等于唯一核销金额。
- 多次部分收款累计不能超过应收金额。
- 超额收款、零金额、负金额和超精度金额返回受控错误，应收余额不变。
- 已收清应收继续收款失败。
- 已过账收款不可编辑、不可取消。
- 应付和付款覆盖与应收、收款对称的全部路径。
- 同一采购入库任一来源明细已生成应付时，再次按来源单据头生成必须整单失败，应付主表和来源明细均不新增。
- 本阶段一笔付款只核销一笔应付，付款金额必须等于唯一核销金额。
- 收付款过账中任一环节失败时整单回滚，不产生部分核销、错误余额或错误状态。
- 并发过账同一应收或应付时不得超收或超付。
- 并发测试使用两个线程同时过账同一应收或应付，最终只能有不超过余额的收付款成功，失败请求返回受控错误，余额恒等式保持成立。
- 回滚测试通过构造重复来源、超额核销或受控异常路径验证事务后数据库无已过账收付款、无核销明细残留、台账余额不变。
- 财务写操作均写入审计日志。
- 财务往来不会写总账、凭证、税务、发票、正式收入确认、正式成本结转或报表数据。

### 前端测试

- 财务 API 封装：请求路径、查询参数、候选来源查询、写操作 CSRF、错误信封处理。
- 路由权限：无权限访问财务页面跳转无权限页，未登录访问跳转登录。
- `/finance` 根路由按首个可用财务查看权限动态重定向；仅有应付查看权限的账号进入 `/finance/payables`，无任一财务查看权限进入无权限页。
- 财务菜单：有财务查看权限时显示财务往来入口，无权限时不显示。
- 应收列表：筛选、重置、分页、加载失败、空状态、金额右对齐和权限按钮显示。
- 应收生成表单：候选来源接口只返回已过账且未生成应收的销售出库，重复来源不会出现在候选列表。
- 应收详情：状态、金额摘要、来源追溯、收款记录、操作按钮和只读提示。
- 收款表单：来源应收摘要、收款日期、金额、方式、备注、超额提示、防重复提交。
- 收款详情：基础信息、核销应收、已过账只读状态。
- 应付列表和详情覆盖与应收对称的筛选、展示、追溯和权限。
- 应付生成表单：候选来源接口只返回已过账且未生成应付的采购入库，重复来源不会出现在候选列表。
- 付款表单和详情覆盖与收款对称的校验、过账和只读状态。
- 状态标签：应收、收款、应付、付款状态到中文文案和 Element Plus 标签类型映射。
- 只读权限：无生成、编辑、确认、关闭、创建收付款、过账或取消按钮。
- 销售角色和采购角色权限：只显示对应方向查看能力，不显示财务操作按钮。
- 窄屏视口：财务列表、详情和收付款表单可访问，关键操作不重叠。

## 浏览器验收路径

1. 管理员或财务角色登录。
2. 确认财务往来菜单、应收台账、收款记录、应付台账和付款记录入口可见。
3. 准备已过账销售出库和已过账采购入库。
4. 基于销售出库生成应收草稿。
5. 编辑应收草稿到期日期和备注。
6. 确认应收。
7. 基于应收登记第一笔部分收款草稿并过账。

12. 基于采购入库生成应付草稿。
13. 编辑并确认应付。
14. 基于应付登记第一笔部分付款草稿并过账。
15. 返回应付列表和详情，确认状态、已付金额、未付金额变化。
16. 基于同一应付登记第二笔付款并过账至已付清。
17. 从应付详情追溯来源采购入库和采购订单。
18. 尝试超额付款，确认前端提示和后端拒绝一致。
19. 销售角色登录，验证仅可查看应收和收款进度，不可写。
20. 采购角色登录，验证仅可查看应付和付款进度，不可写。
21. 只读用户登录，验证只能查看台账、详情和追溯。
22. 无权限用户访问财务页面和接口，验证拒绝。
23. 未登录访问财务页面和接口，验证认证拦截。
24. 保存桌面和窄屏视觉截图。

## 追溯验收口径

- 应收详情必须展示来源销售出库和来源销售订单，至少包含来源单号、业务日期、客户、来源金额和详情入口。
- 收款详情必须展示对应应收单号、客户、核销金额、收款日期和状态。
- 应付详情必须展示来源采购入库和来源采购订单，至少包含来源单号、业务日期、供应商、来源金额和详情入口。
- 付款详情必须展示对应应付单号、供应商、核销金额、付款日期和状态。
- 来源跳转不得误入库存单据、采购入库、销售出库以外的错误单据。
- 缺少来源业务查看权限时，应隐藏对应来源跳转或进入统一无权限页，不得泄露越权数据。

## 金额验收口径

- 前端 payload 中收款金额、付款金额按字符串提交，避免浮点精度污染。
- 后端必须按 decimal 精度校验金额；零值、负数、超精度和非法格式返回受控错误。
- 应收金额、已收金额、未收金额展示为两位小数，列表和详情一致。
- 应付金额、已付金额、未付金额展示为两位小数，列表和详情一致。
- 部分收付款后状态必须与金额关系一致。
- 全额收付款后未收或未付金额必须为 `0.00`，状态为已收清或已付清。
