# 采购管理基础缺陷记录

## PROC-T9-001 库存流水到采购入库追溯断链

- 发现日期：2026-07-04
- 发现角色：测试
- 影响级别：阻断
- 阻断判断：是
- 相关范围：采购入库详情、库存流水列表、采购来源追溯

### 前置条件

- 后端地址：`http://127.0.0.1:18080`
- 前端地址：`http://127.0.0.1:5188`
- 登录账号：`admin`
- 已完成采购主路径：
  - 采购订单：`PO-20260704050958938-000`
  - 第一张采购入库：`PR-20260704050959177-000`
  - 第二张采购入库：`PR-20260704050959341-001`
  - 采购型物料：`T9-MAT-0704130250`
  - 仓库：`T9-WH-0704130250`

### 复现步骤

1. 使用管理员登录前端。
2. 打开库存流水页面：`/inventory/movements?warehouseId=4&materialId=13`。
3. 确认页面显示两条采购入库来源库存流水，原因列显示“采购入库”，方向为“入库”，数量分别为 `4` 和 `6`。
4. 点击任意一条采购入库流水的“查看单据”。
5. 再打开采购入库详情 `/procurement/receipts/1`，查看“库存流水追溯”区域。

### 预期结果

- 对 `sourceType=PURCHASE_RECEIPT` 的库存流水，点击“查看单据”应跳转到对应采购入库详情，例如 `/procurement/receipts/1` 或 `/procurement/receipts/2`。
- 采购入库详情在后端已存在对应 `PURCHASE_RECEIPT` 库存流水时，应展示真实流水追溯数据，至少能看到流水号、方向、变动数量、变动前后数量和来源关系。
- 采购订单、采购入库、库存流水三者之间应能完成浏览器追溯闭环。

### 实际结果

- 点击库存流水“查看单据”后跳转到 `/inventory/documents/2`，打开的是无关库存单据详情，示例页面显示 `INV-OPEN-20260704032459186-005`，不是采购入库详情。
- 采购入库详情 `/procurement/receipts/1` 的“库存流水追溯”区域显示“暂无库存流水追溯”。
- 真实接口已确认库存流水存在：
  - `PROC-RCP-MOV-20260704050959251-1-000`：`movementType=PURCHASE_RECEIPT`、`sourceType=PURCHASE_RECEIPT`、`sourceId=1`、数量 `4`。
  - `PROC-RCP-MOV-20260704050959390-2-001`：`movementType=PURCHASE_RECEIPT`、`sourceType=PURCHASE_RECEIPT`、`sourceId=2`、数量 `6`。

### 影响

- 采购入库已正确增加库存，但浏览器端来源追溯断链。
- 仓库或审计人员无法从库存流水回到采购入库来源单据。
- 采购入库详情无法展示已有真实库存流水，导致验收路径“库存流水出现 `PURCHASE_RECEIPT`，且从库存流水可追溯到采购入库”不成立。
- 命中 `docs/testing/procurement-management-test-plan.md` 阻断标准“采购订单、采购入库和库存流水追溯断链”。

### 建议修复方向

- 库存流水列表的来源跳转应按 `sourceType` 分流：
  - `PURCHASE_RECEIPT` 跳转采购入库详情。
  - 现有库存单据来源继续跳转库存单据详情。
  - 生产来源按既有生产单据路由处理或显式禁用不可追溯入口。
- 采购入库详情应展示真实后端库存流水追溯数据；可以由采购入库详情接口返回对应流水，或为库存流水查询增加可按 `sourceType/sourceId` 过滤的接口能力后由前端加载。
- 修复后需重新验证：
  - 库存流水点击采购入库来源进入 `/procurement/receipts/{id}`。
  - 采购入库详情展示对应 `PURCHASE_RECEIPT` 流水。
  - 两条采购入库流水各自只追溯到自己的采购入库明细。
  - 库存单据来源原有跳转不回归。

  - 第一张采购入库：`PR-20260704053202169-000`

  - 第一张入库流水：`PROC-RCP-MOV-20260704053202246-3-000`
  - 第二张入库流水：`PROC-RCP-MOV-20260704053202387-4-001`

## PROC-T9-002 最小采购/仓库角色访问采购页面时引用数据提示无访问权限

- 发现日期：2026-07-04
- 发现角色：测试
- 影响级别：非阻断
- 阻断判断：否
- 相关范围：采购订单列表、采购入库列表、非管理员角色的供应商/仓库/物料引用数据加载

### 前置条件

- 前端地址：`http://127.0.0.1:5188`
- 后端地址：`http://127.0.0.1:18080`
- 账号按 Task 9 建议的最小角色口径创建：
  - 采购员：`t9r_buyer_0704133200`
  - 仓库角色：`t9r_warehouse_0704133200`
  - 只读：`t9r_readonly_0704133200`

### 复现步骤

1. 使用采购员或仓库角色登录前端。
2. 打开 `/procurement/orders` 或 `/procurement/receipts`。
3. 查看筛选区供应商、仓库等引用下拉加载状态。

### 预期结果

- 用户拥有采购查看或写权限时，采购页面主体、筛选和表单所需的业务引用数据应能按当前角色预期加载。
- 如果系统要求业务角色额外具备基础资料只读权限，应在角色口径或验收说明中明确。

### 实际结果

- 页面主体列表可加载，采购菜单、路由和主要操作按钮按采购权限显示。
- 页面同时出现“无访问权限”提示，来源是供应商、仓库、物料等基础资料引用接口缺少对应 `master:*:view` 权限。
- 后端采购接口权限本身未发现绕过：采购员入库创建在有效已确认订单上返回 `403 AUTH_FORBIDDEN`，仓库角色订单确认返回 `403 AUTH_FORBIDDEN`，只读写接口返回 `403 AUTH_FORBIDDEN`。

### 影响

- 不影响 `PROC-T9-001` 追溯闭环修复。
- 不影响管理员主路径和后端接口鉴权。
- 可能影响采购员按最小权限在浏览器中使用创建/编辑表单选择供应商和物料，也会让列表筛选区出现混淆性错误提示。

### 处理结论

- 处理状态：阶段内文档口径修正，非代码修复。
- 是否阻断 Task 10：否。
- 本阶段不新增采购专用引用数据接口，不扩大采购阶段接口范围。
- 采购员、仓库角色和只读业务验收账号应补齐必要基础资料只读权限，至少包括 `master:supplier:view`、`master:warehouse:view`、`master:material:view`、`master:unit:view`。
- 该授权包用于支持采购页面加载供应商、仓库、物料和单位引用数据，不改变采购订单、采购入库和库存流水的后端权限边界。
- Task 10 视觉分析应使用补齐基础资料只读权限后的采购员、仓库和只读账号采集权限状态截图。

### 需要复验点

- 采购员打开采购订单列表、采购订单表单、采购入库详情时，供应商、物料和单位等引用区域不再出现“无访问权限”。
- 仓库角色打开采购入库列表、采购入库表单和采购入库详情时，仓库、物料和来源数据相关区域不再出现“无访问权限”。
- 只读用户打开采购订单、采购入库和追溯页面时，无写按钮，引用展示不报权限错误。
- 原写权限边界仍成立：采购员不能创建或过账采购入库，仓库角色不能确认、取消或关闭采购订单，只读用户全部采购写接口仍返回 `AUTH_FORBIDDEN`。
- 无权限用户仍不能访问采购页面或采购接口。
