# 质量检验与库存质量状态基础测试计划

## 测试目标

验证 017 质量检验与库存质量状态基础模块满足库存底座、入库质量确认、非合格不可用、采购退货质量状态、权限、期间保护、事务一致性、来源追溯、浏览器操作和视觉证据要求。测试重点是库存余额从仓库和物料维度升级为仓库、物料、质量状态维度后，`availableQuantity` 始终只代表合格可用库存，且待检、不合格、冻结库存不会被普通业务消耗或计划可用量误用。

## 范围

### 本次测试

- 历史库存默认迁移为合格库存，迁移前后总库存数量一致。
- 库存余额按仓库、物料、质量状态分层，库存流水记录质量状态。
- `availableQuantity` 只等于合格可用库存。
- 采购入库、生产完工入库、销售退货、生产退料默认形成待检库存并生成质量确认记录。
- 质量确认支持待检拆分为合格、不合格、冻结，转换前后总库存不变。
- 合格库存可冻结，冻结库存可解冻回合格，原因、业务日期和审计记录完整。
- 销售出库、生产领料、生产补料只能消耗合格库存。
- 采购退货按显式质量状态扣减待检、不合格或合格库存，冻结库存需先解冻。
- 质量确认、冻结、解冻、质量状态过账和按状态扣减受权限、后端鉴权和业务期间保护约束。
- 异常路径下单据、余额、流水、质量确认记录和审计不能出现部分写入。
- 来源追溯覆盖来源单据、来源行、质量确认记录、状态转换流水和后续消耗记录。
- 前端质量确认列表、处理抽屉、库存余额质量分布、候选库存禁用原因、权限态、异常态、空态和窄屏视觉验收。

### 本次不测试

- 完整 QMS、检验标准库、抽样方案、缺陷等级、供应商质量评分。
- SPC、质量趋势分析、质量成本归集或质量经营报表。
- 批次、序列号、库位、条码、拣配。
- MRP、计划净算实现、排产、产能计划。
- 审批流、工作流设计器、消息通知。
- 正式财务、凭证、总账、税务、银行对账、成本重算。
- 复杂返工返修、不合格评审、报废处置闭环。
- 多组织、多公司、多账套、多币种、多租户。

## 跨模块回归路径

- 采购：采购入库进入待检；质量确认合格后可用；采购退货显式选择质量状态；冻结不可直接退货。
- 销售：销售出库只扣合格；销售退货进入待检；确认前不可再次销售；合格确认后可再次销售。
- 生产：完工入库进入待检；生产退料进入待检；生产领料和生产补料只扣合格。
- 库存：余额、流水、状态分布、`availableQuantity`、来源追溯一致。
- 权限：菜单、路由、按钮和接口鉴权一致，前端控制不能替代后端拒绝。
- 期间保护：锁定期间写入拒绝，开放期间操作正常，锁定期间历史查询和追溯可用。
- 报表和计划前置口径：017 不实现 MRP，但任何暴露可用量的接口和页面不得把非合格库存计入可用库存。

## 异常路径和阻断缺陷

以下问题按阻断缺陷处理：

- 数据库迁移失败，或历史库存未默认迁移为合格库存。
- 迁移后同一仓库、物料下各质量状态数量合计与迁移前现存数量不一致。
- 库存余额、库存流水、质量确认记录、来源单据之间数量或质量状态不一致。
- `availableQuantity` 包含待检、不合格或冻结库存。
- 采购入库、完工入库、销售退货、生产退料未默认形成待检库存。
- 质量确认转换前后总库存变化，或数量不平、重复提交、超过待检余额仍被接受。
- 销售出库、生产领料、生产补料成功消耗待检、不合格或冻结库存。
- 采购退货未按显式质量状态扣减，或直接扣减冻结库存。
- 无权限、只读、未登录用户能处理质量确认、冻结、解冻或绕过质量状态校验。
- 锁定期间仍允许质量确认、冻结、解冻、入库质量落账或按质量状态扣减。
- 查询锁定期间历史质量记录、库存余额、库存流水或来源追溯被错误阻断。
- 失败事务产生单据状态、库存余额、流水、质量记录或审计的部分写入。
- 并发质量确认或并发扣减导致负库存、重复流水、重复消耗或状态错乱。
- 来源追溯无法从质量记录追到来源单据、来源行、状态转换流水或后续消耗。
- 核心质量页面无法浏览器访问、关键接口不可用、浏览器主路径不可执行。
- 视觉分析缺失，或视觉问题影响核心操作、关键数据识别、权限状态判断或质量状态业务识别。

## 阶段工作包测试建议

1. 库存质量状态底座测试：覆盖迁移、余额分层、流水质量状态、`availableQuantity` 合格口径和查询筛选。
2. 质量确认与状态转换测试：覆盖入库待检、质量确认拆分、冻结解冻、权限、期间保护、原因审计和事务回滚。
3. 业务入口消耗控制测试：覆盖采购、销售、生产、反向业务接入，重点验证非合格不可消耗和采购退货显式质量状态。
4. 前端浏览器与视觉验收测试：覆盖质量确认页面、库存余额质量分布、候选禁用原因、权限差异、异常提示、桌面和 390px 窄屏截图证据。

## 全量验证窗口关注点

- 后端全量测试、数据库迁移验证、前端全量测试、类型检查和生产构建统一在阶段交付前执行。
- 全量验证前必须完成定向自动化、浏览器主路径、权限路径、异常路径和视觉截图证据。
- 全量验证发现问题时先用定向测试定位和修复，再复验受影响路径，最终回到全量确认。
- 交付前抽查同一物料四种质量状态并存场景，核对余额、流水、质量确认、来源追溯和 `availableQuantity`。
- 交付前确认视觉分析目录、截图清单、视口尺寸、账号权限、问题处理和最终结论完整。
